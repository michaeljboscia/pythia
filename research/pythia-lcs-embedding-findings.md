## Default Embedding Model: all-MiniLM-L6-v2 is DEAD

### What we assumed
all-MiniLM-L6-v2 (384d, ONNX) is "good enough" because tree-sitter AST chunking compensates for model quality.

### What the research proves
- all-MiniLM-L6-v2 scores ~42 NDCG@10 on MTEB Retrieval — severely outclassed
- `nomic-embed-text-v1.5` scores ~50, has 8,192-token context window (essential for code chunks)
- Nomic supports Matryoshka Representation Learning (MRL) — truncate to 256d and still beat MiniLM at 384d
- Runs locally via ONNX (`@xenova/transformers`) or Ollama

### Decision D-60: Default model is `nomic-embed-text-v1.5`
- ONNX via `@xenova/transformers` (no daemon dependency)
- Fallback: Ollama endpoint if user has it running
- Output truncated to 256 dimensions via MRL

## Dimension Choice: 256-512 is the Universal Sweet Spot

### What we assumed
Higher dimensions = meaningfully better retrieval. 384 was a compromise.

### What the research proves (EM-06)
- MRL has decoupled quality from dimension count
- 3072d OpenAI → 1024d retains 99.2% performance
- 1024d → 256d retains ~99% performance
- 256d → 512d adds only 0.44 NDCG points

### Decision D-61: Default dimension is 256 (float32)
- Even premium APIs (OpenAI, Voyage) should request truncated dimensions
- Reduces SQLite storage 3-6x, makes KNN exponentially faster
- Virtually zero impact on code retrieval accuracy

## Surprise A: The Migration Black Hole (EM-09)

### The problem
Switching embedding provider in `pythia.config.json` instantly blinds the system. Vectors from Model A are garbage when queried with Model B — completely different vector spaces.

### Decision D-62: Embedding provider hash + automatic re-index
- Config loader hashes `provider_name + model_version`
- Hash stored in SQLite schema (e.g., `embeddings_nomic_v1_5`)
- On provider change: detect mismatch → refuse stale queries → trigger background re-index into new table
- Old table preserved until new index is complete, then dropped

## Surprise B: Binary Quantization for Premium Path (EM-03)

### The finding
Voyage-code-3 supports native binary quantization API returns. At 1024 binary dimensions, retains 91.50% NDCG while shrinking vectors to 128 bytes.

### Decision D-63: VectorStore interface supports binary vectors
- `search()` must support both Cosine similarity (float) and Hamming distance (binary)
- Default path: float32 at 256d (nomic)
- Premium path: binary at 1024d (Voyage) — massive storage savings
