## Tree-sitter: Perfect for Chunks, Blind for Graphs

### What we assumed
Tree-sitter alone builds both the vector index (chunks) AND the AST graph (edges).

### What the research proves
- **Chunking: YES** — Tree-sitter maps every byte to a precise syntax node. Function-boundary chunking with docstring attachment works perfectly (CI-01, CI-02).
- **Graph edges: NO** — Tree-sitter is syntactic only. It knows `logger.info()` is a `call_expression` but CANNOT resolve which `logger` or which `info`. Cannot distinguish `console.log` from a custom imported `logger.log`. False-positive edges guaranteed (CI-04).

### Decision D-64: Dual-engine indexer (Tree-sitter + TypeScript Compiler API)
- **Tree-sitter** = Vector Engine (chunks). Fast, runs on every file save.
- **TypeScript Compiler API / headless tsserver** = Graph Engine (edges). Slower, runs async background pass. Uses `getDefinitionAtPosition()` for hyper-accurate `:CALLS` and `:IMPORTS` resolution.

## The Concrete Pipeline

### Step 1: Fast Path — Vector Indexing (on every file save)

```
User saves auth.ts
  → tree-sitter-typescript generates CST
  → SCM query extracts (function_declaration) @func, (class_declaration) @class
  → Format chunk: "[File: src/auth.ts | Class: AuthManager]\n/** JSDoc */\nfunction login() { ... }"
  → nomic-embed-text-v1.5 (ONNX) → 256d float vector
  → Upsert into sqlite-vec with ID "src/auth.ts::AuthManager.login"
```

### Step 2: Slow Path — Graph Building (async background)

```
Background daemon triggers after batch of saves
  → Instantiate TypeScript LanguageService (tsconfig.json)
  → Walk AST: find CallExpression inside AuthManager.login calling verifyToken()
  → LSP resolves: verifyToken → src/crypto.ts::verifyToken
  → Upsert edge into SQLite graph:
      Source: src/auth.ts::AuthManager.login
      Target: src/crypto.ts::verifyToken
      Relation: CALLS
```

### Step 3: Query Time

```
User asks: "What depends on verifyToken?"
  → lcs_graph_query: 2-hop BFS from src/crypto.ts::verifyToken (CALLED_BY edges)
  → Returns: [auth.ts::login, session.ts::refresh, middleware.ts::validateRequest]
  → lcs_code_search: retrieve chunks for those 3 functions
  → Cross-encoder re-ranks → top 3 chunks
  → Feed to Pythia CLI (< 4000 tokens)
```

## LSP is Required, Not Optional (CI-03)

### What we hoped
Tree-sitter alone = zero heavyweight dependencies.

### What CI-03 proves
- Headless `tsserver` provides Go-to-Definition, Find-All-References, type inference
- Required for accurate `:CALLS`, `:IMPORTS`, `:IMPLEMENTS` edges
- Can run as background process without IDE
- TypeScript-specific — each language needs its own LSP for graph accuracy

### Implication for pluggable architecture
The `GraphStore` interface remains language-agnostic, but the **graph builder** (the thing that populates it) must be language-specific. For v1: TypeScript only via `tsserver`. Python, Rust, etc. would need their respective LSPs.
