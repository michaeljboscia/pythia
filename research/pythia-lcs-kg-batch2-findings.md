## LightRAG Verdict: Cost Validated, Quality Invalidated

### What we assumed
LightRAG's dual-level retrieval matches GraphRAG quality at a fraction of the cost.

### What the research proves
- **Cost model confirmed** — LightRAG uses <100 tokens for keyword extraction vs GraphRAG's 610K map-reduce. The $0 CLI architecture survives.
- **Quality claims debunked** — Independent studies (Wuhan University, arXiv:2506.06331) found severe evaluation biases in the original LightRAG paper. Once corrected for length and position bias, LightRAG's advantage over NaiveRAG vanished.
- **Small model degradation** — LightRAG extraction quality degrades severely with small open-source models (our $0 default path).

### Architectural Pivot
- Keep LightRAG's dual-level routing (Entity vs Relationship paths) — fits $0 CLI
- **ONNX Cross-Encoder Re-ranker becomes mandatory** — compensates for LightRAG's noisy raw retrieval
- Pipeline: LightRAG retrieves 50 candidates → Cross-encoder filters to top 3-5 → Pythia CLI generates

## Community Detection: "Weekly Batch" → Volume-Based Threshold

### What we assumed
Leiden community detection is the computational bottleneck → run weekly as batch.

### What the research proves (KG-06)
- **Leiden is NOT the bottleneck** — runs in milliseconds to seconds on 10K-100K node graphs
- **LLM summary generation IS the bottleneck** — regenerating community summaries costs real tokens
- The bottleneck is writing, not partitioning

### Architectural Pivot: Accumulator Threshold (replaces weekly cron)

1. **Continuous AST updates** — edit `auth.ts` → AST edges update instantly in SQLite. Local Search always real-time.
2. **5% churn threshold** — system tracks mutated edges. Once >5% of total graph edges change, trigger Global Update.
3. **Fast Leiden** — run across whole graph (<1 second)
4. **Selective diffing** — compare new vs old communities. Only regenerate LLM summaries for communities whose membership actually changed.

## ADR Graph Nodes (KG-07): Free Ontology

Pythia's episodic checkpoints should use the **MADR 4.0 specification** (Decision, Context, Option, Consequence) with established edges:
- `SUPERSEDES` — new decision replaces old
- `IMPLEMENTS` — code node fulfills decision
- `AFFECTS` — decision impacts other components
