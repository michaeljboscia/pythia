# Noosphere Requirements Interrogation: Cycle 2 Answers

## Q1 — Soft-delete zombie accumulation
**Decision:** `is_deleted=1` rows are kept forever for referential integrity; there is no automated garbage collection.
**Rationale:** The primary purpose of the Noosphere is to act as an architectural memory. If a file is deleted but a Pythia MADR checkpoint references its AST nodes, those nodes must remain in the graph as historical artifacts ("Ghosts of Code Past") to explain *why* decisions were made. Because we are talking about text chunks in SQLite, the storage overhead is negligible for a solo developer codebase. ⚠️ (Codex may argue for a TTL to save space, but this breaks the structural memory).

## Q2 — FTS5 trigram vs. keyword mismatch
**Decision:** The FTS index must use standard tokenization (`tokenize="unicode61"`), not `trigram`.
**Rationale:** We need exact keyword matching for code symbols. If a user searches for `login`, they want `login`, not every file containing `log`. Trigram tokenization destroys precision for exact symbol lookup, which is the primary use case for the sparse half of the hybrid search. 

## Q3 — The `ask()` context injection ceiling
**Decision:** The `additionalContext` array passed to `ask()` has a strict ceiling of 3,000 tokens (approx. 12,000 characters) per call.
**Rationale:** The `ask()` call is meant for immediate retrieval augmentation (IRCoT pattern), not full-corpus reloading. By capping it at 3K tokens, we guarantee that the LLM only receives the absolute highest-scored chunks from the re-ranker, protecting both the CLI provider from crashing and the SDK provider from astronomical token costs.

## Q4 — Worker Thread crash isolation
**Decision:** The Main thread runs a Supervisor that listens for the Worker Thread's `error` and `exit` events; if the Worker dies, the Supervisor marks the current sync job as `failed`, logs the stack trace to SQLite, and immediately spawns a fresh Worker Thread.
**Rationale:** The MCP Server (Main Thread) must never crash due to a background indexing failure. The Supervisor pattern ensures the system remains highly available to Claude Code, even if a malformed file panics the Tree-sitter or ONNX C++ bindings in the background.

## Q5 — CNI collision in monorepos
**Decision:** Re-exports do NOT get separate CNI entries; the indexer resolves the export and links the AST node back to the canonical source definition's CNI.
**Rationale:** Creating duplicate CNIs for the same logical symbol breaks the graph structure and pollutes the vector space with identical code blocks. The LSP Slow Path must walk the export chain and assign the edge to the original `packages/auth/src/utils.ts::function::login`.

## Q6 — Graph query depth limit
**Decision:** Recursive CTE traversals for `intent: "structural"` are hard-capped at a depth limit of 3 hops, and use SQLite's native `LIMIT` or a cycle-detection `WHERE` clause tracking the path.
**Rationale:** A 3-hop limit is sufficient to answer "what does this call, and what calls this" without traversing the entire application graph. Without a limit, cyclic dependencies (A imports B imports A) will cause the CTE to infinite-loop and lock the database.

## Q7 — `pythia_transcripts` content schema
**Decision:** The `content` column stores a JSON object: `{ "text": string, "tool_calls": [{ "name": string, "args": object, "result": string }] }`.
**Rationale:** We must preserve the exact orchestration sequence. If Pythia called `lcs_investigate` and got a result, that tool usage and its output must be stored in the transcript so the offline Conversational Coherence evaluator can see exactly what context the model was acting upon.

## Q8 — MADR ID collision under concurrent writes
**Decision:** We use SQLite's implicit `rowid` combined with an `AFTER INSERT` trigger to generate the public MADR ID atomically.
**Rationale:** `SELECT COUNT(*) + 1` is unsafe. By allowing SQLite to assign the internal auto-incrementing `rowid`, and then firing a trigger to `UPDATE pythia_memories SET id = 'MADR-' || printf('%03d', rowid) WHERE rowid = new.rowid`, we guarantee sequential, human-readable IDs without race conditions. ⚠️ (Codex might prefer application-level locks, but DB triggers guarantee absolute atomicity).

## Q9 — `oracle_decommission` verification phrase contract
**Decision:** The verification phrase is a 6-character alphanumeric string generated by the MCP server when `spawn_oracle` is called, returned as a property of the session metadata, and must be echoed back exactly to decommission.
**Rationale:** Decommissioning deletes the active temporal session state. It should not be possible for an LLM to accidentally trigger it via hallucination. Forcing the LLM to store and recall a random 6-character string ensures the decommission is intentional.

## Q10 — Obsidian vault path resolution
**Decision:** The workspace path is strictly defined as the Current Working Directory (CWD) where the `pythia` process is executed. If `.obsidian/` does not exist in that CWD, the `obsidian_sync` feature is silently disabled.
**Rationale:** The MCP server is executed by Claude Code, which runs in the context of the user's project directory. If the user hasn't initialized an Obsidian vault in that specific project folder, we shouldn't create rogue `.obsidian` directories on their machine.

## Q11 — Embedding model version pinning
**Decision:** The `pythia_metadata` table (new) stores the active model name and dimensions; if the config changes the model or version, the system refuses to query the vectors and flags a `FULL_REINDEX_REQUIRED` state.
**Rationale:** Cosine similarity across different models is mathematically invalid. We must hard-pin the model version in the DB. If the model upgrades, the existing `vec_lcs_chunks` table is dropped and rebuilt from the raw code still stored in `lcs_chunks`.

## Q12 — `pythia_force_index` atomicity
**Decision:** `pythia_force_index` queues a synchronous message to the Worker Thread; if the Worker is already syncing, it debounces the request and returns an "Index already in progress" status to the caller.
**Rationale:** The Main thread must not attempt to index files or write to SQLite concurrently with the Worker thread. SQLite only allows one writer. Routing all indexing commands through the Worker's message queue guarantees sequential, atomic execution.

## Q13 — Cold start indexing progress reporting
**Decision:** The `lcs_investigate` tool will return a `system_warning` field in its output string: `[WARNING: Indexing is currently in progress (45% complete). Results may be incomplete.]`
**Rationale:** Claude Code needs to know why it can't find a file it knows exists. Injecting the warning directly into the search result ensures the LLM reads it, whereas a separate `pythia_status` tool might be ignored by the model.

## Q14 — Premium stack migration path
**Decision:** The `pythia` CLI will include a `pythia migrate` command that drops the `vec_lcs_chunks` virtual table, executes the `CREATE VIRTUAL TABLE` statement with the new dimension count from config, and re-triggers the Worker Thread to re-embed `lcs_chunks`.
**Rationale:** SQLite virtual tables cannot alter their schema after creation. The transition from 256d to 1024d requires a destructive rebuild of the derived vector index. The raw data remains safe in `lcs_chunks`.

## Q15 — `session_name` vs `session_id` duality
**Decision:** The schema must add `session_name TEXT UNIQUE NOT NULL` to `pythia_sessions`; `session_id` is the internal UUID used for primary keys, while `session_name` is the human-readable identifier used by the LLM caller.
**Rationale:** LLMs are terrible at passing raw UUIDs around. They need semantic names (e.g., "auth-refactor-session"). When `spawn_oracle` is called, it checks if `session_name` exists; if so, it returns the existing `session_id`.

## Q16 — Graph edge validation enforcement point
**Decision:** Enforcement is strictly Application-Level via the Worker Thread during the `BEGIN TRANSACTION` sync block.
**Rationale:** SQLite foreign keys on virtual tables or complex triggers introduce massive performance penalties during bulk inserts. The TypeScript Worker Thread must validate that the extracted LSP edges point to valid CNIs *before* executing the SQLite insert. ⚠️ (Codex may push for DB-level foreign keys, but performance dictates application-level).

## Q17 — LSP server lifecycle
**Decision:** The Worker Thread maintains a single, long-lived instance of the TypeScript `LanguageService` in memory, updating its internal file snapshots as files change.
**Rationale:** Spawning `tsserver` per file takes 1-3 seconds of boot time. Keeping the `LanguageService` hot in the Worker Thread reduces semantic resolution time to milliseconds. If the Worker crashes, the Supervisor restarts it, naturally rebooting the LSP.

## Q18 — Config schema and validation
**Decision:** The config file is strictly validated via `zod` at startup; if parsing fails, the MCP server logs an error and halts execution entirely, refusing to boot.
**Rationale:** An invalid config (e.g., specifying Voyage AI but providing no API key) will cause catastrophic downstream failures. The system must fail fast and loud at boot time rather than throwing obscure errors during an LLM chat session.

## Q19 — BLAKE3 hash dependency
**Decision:** We use `@noble/hashes/blake3` (pure JavaScript, audited, no native bindings).
**Rationale:** Relying on native Node.js bindings (node-gyp) is the #1 cause of failed `npm install -g` executions across different OS environments. A pure JS implementation guarantees frictionless installation, and the slight performance hit for hashing text files is irrelevant compared to the ONNX embedding time.

## Q20 — `lcs_investigate` ranking algorithm
**Decision:** The system uses Reciprocal Rank Fusion (RRF) with `k=60` to combine the SQLite-vec and FTS5 rankings, taking the top 50 results, which are then passed through the ONNX Cross-Encoder for the final Top-5 extraction.
**Rationale:** RRF is parameter-free and requires no tuning weights. It mathematically normalizes the wildly different score scales of Cosine Similarity and BM25. The cross-encoder then acts as the ultimate semantic arbiter to filter out the noise.
