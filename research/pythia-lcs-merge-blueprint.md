## The Abstraction

Pythia v2 = a localized, self-updating digital twin of your codebase, equipped with a persistent reasoning daemon.

- LCS = the deterministic SPACE (the code as it exists now)
- Pythia = the episodic TIME (the history of decisions and rationale)
- Together = a unified spacetime for the project that Claude navigates

---

## Why Merge (Not Separate)

1. **Zero-latency internal loops** — Pythia queries LCS in-process (milliseconds), not via MCP round-trips through Claude
2. **Unified persistence** — Episodic memory + factual graph in same database. Atomic rollbacks cover the whole brain.
3. **No sync nightmare** — LCS file-watcher runs in background, graph is always current. No manual corpus management.

## Why Separation Fails

- "Separation of concerns" — Refuted: LCS and Pythia are two sides of RAG. Can't design retrieval without knowing how the LLM consumes it.
- "Claude might use LCS alone" — Merged server exposes BOTH raw LCS tools AND stateful oracle tools.

## Build Order (Bottom-Up)

### Phase 1: The Indexer (LCS Data Plane)
- tree-sitter parsing + embedding generation (Voyage/OpenAI)
- Write to local LanceDB/SQLite
- Expose: `pythia_code_search`

### Phase 2: The Graph (LCS Structural Plane)
- Relationship extraction (Imports, Calls, Implements)
- Expose: `pythia_graph_query`

### Phase 3: The Engine (Pythia Cognition Plane)
- Native Gemini SDK wrapper
- Direct TypeScript access to Phase 1+2 databases
- Expose: `spawn_oracle`, `ask_oracle`

### Phase 4: The Persistence (Pythia Memory Plane)
- JSON checkpoint → SQLite nodes with edges to codebase graph
- Expose: `oracle_checkpoint`, `oracle_reconstitute`, `oracle_decommission`
