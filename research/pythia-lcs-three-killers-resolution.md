## KILLER 1: GraphRAG Global Search vs. $0 CLI Architecture

### The Problem
Microsoft GraphRAG Global Search requires up to 610,000 tokens per query via map-reduce fan-out across all community summaries. The free Gemini CLI cannot physically execute 50 parallel LLM map operations.

### Is Global Search Necessary for a Developer Tool?
Yes, but constrained. Developers ask "What is our standard pattern for error handling across the backend?" — not "summarize all themes." This requires multi-document synthesis but NOT reading every community.

### The Engineering Decision: LightRAG Dual-Level Retrieval (Drop Map-Reduce)

1. Extract entities and edges during indexing
2. Form communities via Leiden algorithm (batch, at index time)
3. Pre-generate a summary for each community
4. **Crucially:** Embed the community summary itself
5. At query time: vector search against community summary embeddings → retrieve top 3-5 → feed to CLI

Route queries via Claude: `lcs_local_search` (raw chunks + graph edges) vs `lcs_global_search` (pre-computed community summary vector search).

- **Implementation Cost:** Medium — requires community detection + summarization in indexing pipeline
- **Sacrifice:** Absolute comprehensiveness. Patterns scattered across 50 communities → Pythia sees only top 3. We sacrifice Microsoft-scale global synthesis to maintain $0 runtime cost.

## KILLER 2: The Incremental Update Deadlock (CAS vs. Community Detection)

### The Problem
Leiden community detection is a global modularity optimization. Adding a single edge can cascade and reshuffle the entire graph, invalidating every pre-computed community summary.

### How LightRAG Solves This: Graph Union + Lazy Invalidation

1. Extract entities/edges for the changed document only
2. Merge into global graph (union)
3. Do NOT immediately run global community detection
4. Graph topology updates in real-time for Local Search (just traverses edges)
5. Community Summaries for Global Search are allowed to go temporarily stale

### The Engineering Decision: Split the Graph

1. **Deterministic Graph (AST):** Built via Tree-sitter. Updates instantly. Never requires "communities." Used for all code-tracing queries. 100x cheaper, faster, and more accurate than LLM community detection for code.
2. **Semantic Graph (Documentation & Pythia Checkpoints):** Built via LLM extraction. Community detection runs as a **weekly batch job**, not continuously.

- **Implementation Cost:** Low/Medium — Tree-sitter-to-SQLite edge extractor is straightforward
- **Sacrifice:** Real-time global thematic updates for documentation. New ADR on Monday → Pythia finds it via Vector Search instantly, but won't recognize its impact on "global architectural themes" until the weekly batch runs.

## KILLER 3: The Evaluation Paradox (RAGAS vs. Event-Sourced Multi-Turn)

### The Problem
RAGAS assumes stateless, single-turn query-response. Pythia is stateful and event-sourced. If Pythia references a fact from 20 turns ago (stored in its event log, not the graph), RAGAS flags it as "hallucination" because the fact isn't in the immediate retrieved chunk.

### The Engineering Decision: Strict Separation of Evaluation

**Layer 1 — Evaluate LCS (Data Plane) via RAGAS:**
- Generate synthetic testset of queries based on codebase
- Run queries directly against `lcs_local_search` and `lcs_global_search` tools
- Measure with RAGAS (Context Precision, Answer Relevance)
- Pythia is NOT involved in this evaluation

**Layer 2 — Evaluate Pythia (Cognition Plane) via Stateful Tracing:**
- Abandon RAGAS for Pythia entirely
- Quality measured by **Event Sourcing Log** (`vN-interactions.jsonl`)
- Offline script replays session log to LLM Judge (Claude) with rubric:
  "Did the Oracle maintain conversational coherence? Did it accurately incorporate facts stated by the user in turn 3 when answering in turn 10?"

- **Implementation Cost:** Low — RAGAS is built for stateless eval; Pythia log judge is a ~50-line Python script wrapping the Anthropic API
- **Sacrifice:** No single unified "System Quality Score." Two dashboards: "Is my index good?" (LCS/RAGAS) and "Is my daemon paying attention?" (Pythia/Coherence).

## Summary: The Merged Path

By applying:
1. **LightRAG dual-level** (kill the map-reduce cost)
2. **Split AST graph from Semantic graph** (kill the incremental update deadlock)
3. **Decoupled evaluation frameworks** (kill the statefulness paradox)

**The unified Pythia Noosphere architecture is structurally sound, mathematically viable, and remains $0/month for the end user.**
