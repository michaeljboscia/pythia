## GRAPH DB VERDICT: Kuzu is Dead, SQLite Wins

### The shock
Kuzu was acquired by Apple and archived in October 2025. No security patches, no updates. Cannot build a production open-source tool on an abandoned engine.

### Decision D-71: Drop Kuzu from architecture
- Was the Premium graph store choice — now dead
- Replacement: FalkorDBLite (embedded via Python, matrix multiplication traversals, New BSD license)

### Decision D-72: SQLite is the $0 graph store (validated)
- GD-02 proves: 100K nodes, 2-3 hop queries execute in milliseconds via recursive CTEs
- Limitations (no community detection, 2x overhead vs joins) are acceptable at our scale
- Combined with sqlite-vec + FTS5 = everything in ONE file

### Revised Premium Graph: FalkorDBLite
- GD-05: Embedded, uses GraphBLAS matrix multiplication for traversals
- New BSD license (safe for open source)
- Neo4j (GD-03) and ArangoDB (GD-04) both require separate servers — disqualified

## MCP TOOL DESIGN: Three Mandatory Changes

### Change 1: Plain Text Output (D-73)
MC-03 proves JSON-wrapping distracts models and reduces reasoning ability. When `code_search` returns chunks, it must return formatted plain text/markdown:
```
--- File: src/auth.ts | Class: AuthManager ---
function login(credentials: LoginPayload) {
  ...
}
```
NOT a JSON array of hit objects.

### Change 2: Consolidate Similar Tools (D-74)
Collapse `code_search` and `graph_query` into a single `pythia_investigate` tool. LLMs struggle to choose between similar tools. One tool with an intent parameter:
```
pythia_investigate({ intent: "semantic" | "structural", query: "..." })
```
MCP server routes to the correct SQLite/Vector logic internally.

### Change 3: Verbose Descriptions + Examples (D-75)
Tool descriptions must be 3-4 sentences, explicitly stating when NOT to use the tool. Add `input_examples` to MCP schema so the LLM knows exactly how to format queries.

## EVALUATION: Minimum Viable Setup

### The 50-Question Golden Set (D-76)
- Use LLM to synthetically generate 100 questions from the codebase
- Manually review and fix 50 of them
- Commit as `eval_golden_set.json` in git
- 50-100 examples is scientifically sufficient for development (EQ-04)

### The Single Metric: Faithfulness (D-77)
- "Are the claims in the answer supported by the retrieved context?"
- 95% agreement with human judgment (EQ-01, EQ-06)
- Grade with cheap model (Haiku / GPT-4o-mini) on binary Pass/Fail
- Don't track 10 metrics — one is enough for a solo developer

### The CI Script (D-78)
- Single 50-line Python script (DeepEval or Promptfoo)
- Runs Golden Set against LCS retrieval tools on major commits
- Reports Faithfulness pass rate
- That's the entire eval infrastructure
