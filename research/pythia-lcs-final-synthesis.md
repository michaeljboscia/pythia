## What Prior Art Taught Us: The Missing Fast-Path

### The Cursor Pattern (PA-01)
Cursor maintains an in-memory LRU cache of the user's recent files (last 10 touched). Bypasses embedding search entirely — jams raw text into the prompt. Only triggers the slow semantic index when the fast-path fails.

### The Copilot Workspace Pattern (PA-02)
Separates Planning from Generation. Generates a plan, grounds it against the graph, THEN writes code. We should adopt this for `ask_oracle`.

### Decision D-79: LRU Recent-File Cache in MCP server
- In-memory `LRU_Cache` of active working directory (watched via Node `fs.watch`)
- On any Pythia question: check LRU cache first → answer instantly if found
- Only invoke SQLite/ONNX index for deep structural questions
- Cache size: last 10-20 files the user touched

## Data Management: Chokidar is a Trap

### What we assumed
Real-time file watcher triggers re-indexing on save.

### What DM-03 + DM-04 prove
- `chokidar` / `fs.watch` on every keystroke strangles the Node.js event loop
- ONNX embedding on every save turns the laptop into a space heater

### Decision D-80: Git-Driven Incremental Indexing
1. Background thread polls `git diff` on a 5-minute interval (or listens for Git events)
2. Only runs Tree-sitter + ONNX on committed/staged files, or files saved >60 seconds ago
3. CAS table in SQLite: `(file_path, last_modified_hash)` — if hash matches, skip
4. Most-recently-modified files indexed first

## Production Engineering: Worker Threads are Mandatory

### The fatal flaw
ONNX embeddings in the main Node.js thread = 2-second blocking → MCP server freezes → entire IDE hangs. SQLite single-writer lock: indexer writing while Claude reading = database lock.

### Decision D-81: Two-thread MCP server architecture
- **Thread 0 (Main):** Speaks JSON-RPC to Claude. Reads from SQLite only.
- **Thread 1 (Worker):** Runs ONNX models. Writes to SQLite.
- Communication via `MessageChannel`
- SQLite must use `PRAGMA journal_mode=WAL` for concurrent read/write

## The One Open Question: Cold Start on Large Repos

A 500K-line enterprise repo takes 2-3 hours to build the initial SQLite/Vector index via ONNX on CPU.

**Decision D-82: Non-blocking async cold start**
- Daemon boots instantly
- Reads Git tree, starts indexing most-recently-modified files first
- Answers questions using whatever it has indexed so far + LRU cache
- Full index completes in background — no blocking
