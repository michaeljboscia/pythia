# Text chunking algorithms: a deep technical analysis

**The choice of chunking algorithm can swing RAG retrieval recall by 10+ percentage points**, yet most teams default to fixed-size splitting without understanding the mechanics or measured tradeoffs. This analysis dissects the exact implementation details of major chunking algorithms — from LangChain's recursive splitter to AST-aware code chunking — grounds quality claims in benchmark data from six empirical studies, and maps strategies to specific artifact types. The core finding across all evidence: content-aware chunking consistently outperforms naive fixed-size approaches, but the optimal strategy depends heavily on document type, query complexity, and embedding model architecture.

## How RecursiveCharacterTextSplitter actually works

LangChain's `RecursiveCharacterTextSplitter` is the most widely used chunking implementation in production RAG systems. Its source lives in `langchain_text_splitters/character.py`, with the merge logic inherited from the base `TextSplitter` class in `base.py`.

The algorithm operates on a **four-level separator hierarchy**, defined as the default in `__init__`:

```python
self._separators = separators or ["\n\n", "\n", " ", ""]
```

These separators progress from coarsest to finest: paragraph boundaries (`\n\n`), line breaks (`\n`), word boundaries (` `), and individual characters (`""`). The recursive algorithm's core method, `_split_text(text, separators)`, follows a precise sequence. First, it iterates through separators and selects the **first one that exists** in the text via `re.search`. It then splits the text on that separator using `_split_text_with_regex`. Each resulting piece is classified: pieces smaller than `chunk_size` are accumulated as "good splits," while oversized pieces trigger recursion with the **remaining finer separators**. Good splits are merged together via `_merge_splits` to produce chunks as close to `chunk_size` as possible.

The `_merge_splits` method implements a greedy packing algorithm with overlap. It accumulates pieces into a `current_doc` list while tracking total length (including separator characters between pieces). When adding the next piece would exceed `chunk_size`, the current accumulation is emitted as a chunk. The overlap mechanism then pops pieces from the **front** of `current_doc` until total length drops to or below `chunk_overlap` (default **200 characters**). These retained pieces form the bridge into the next chunk. The default `chunk_size` is **4000 characters**, and the default `length_function` is Python's `len()` — meaning character count, not tokens. Token-based measurement is available via the `from_tiktoken_encoder()` class method.

A critical edge case: when no separator produces pieces small enough and the algorithm reaches the empty-string separator, it splits into individual characters. If even individual characters somehow exceed `chunk_size` (practically impossible), or if a single atomic unit like a long URL exceeds the limit, the chunk is **emitted anyway** with a warning logged. The algorithm never discards content. The `keep_separator` parameter defaults to `True` in `RecursiveCharacterTextSplitter` (unlike the base class default of `False`), attaching separators to the start of subsequent pieces via lookahead regex.

## LlamaIndex's SentenceSplitter and its three-level hierarchy

LlamaIndex's `SentenceSplitter` takes a fundamentally different approach: it prioritizes **sentence integrity** over arbitrary size boundaries. The implementation in `llama_index/core/node_parser/text/sentence.py` uses a three-level hierarchical splitting strategy with an internal `_Split` dataclass that tracks each piece's text, whether it constitutes a complete sentence (`is_sentence: bool`), and its token count.

**Level 1** splits on paragraph boundaries using `paragraph_separator` (default: `\n\n\n`, triple newline). **Level 2** applies sentence boundary detection via NLTK's `PunktSentenceTokenizer` — an unsupervised pre-trained model that handles abbreviations, ellipses, and other edge cases far more robustly than regex. **Level 3** handles oversized sentences using a `secondary_chunking_regex` pattern: `[^,.;。？！]+[,.;。？！]?|[,.;。？！]`, which breaks on commas, semicolons, periods, and CJK punctuation. If pieces still exceed the budget, a **Level 4** fallback splits on spaces (word-level).

Token budget management uses a greedy packing loop. The default `chunk_size` is **1024 tokens** (not characters) with **200-token overlap**. The tokenizer defaults to tiktoken's `cl100k_base`. A distinctive feature is **metadata-aware splitting**: the `split_text_metadata_aware` method computes an effective chunk size by subtracting the token count of metadata that will be prepended to each chunk. If remaining space drops below 50 tokens, it issues a warning. This prevents the common production bug where metadata pushes chunks past embedding model context windows.

The overlap mechanism operates at the sentence level when possible, carrying complete sentences from the end of one chunk to the beginning of the next, avoiding mid-word or mid-sentence breaks that plague character-based approaches.

## Semantic chunking: embedding-driven boundary detection

Semantic chunking detects topic boundaries by measuring **embedding similarity between consecutive sentences**. LangChain's `SemanticChunker` (in `langchain_experimental/text_splitter.py`, based on Greg Kamradt's "5 Levels of Text Splitting") implements this in six steps.

First, text is split into sentences using the regex `(?<=[.?!])\s+`. Each sentence is then combined with its neighbors in a **context window** controlled by `buffer_size` (default 1) — so each sentence's embedding actually represents a 3-sentence window (previous + current + next). All combined sentences are embedded in a single batch API call. The algorithm then computes **cosine distances between each consecutive pair** of sentence embeddings, producing N-1 distance values where spikes indicate semantic shifts.

Four breakpoint detection methods determine where to split:

- **Percentile** (default, threshold 95): splits where distance exceeds the 95th percentile of all distances
- **Standard deviation** (threshold 3): splits where distance exceeds mean + 3σ
- **Interquartile range** (threshold 1.5): splits where distance exceeds mean + 1.5 × IQR
- **Gradient** (threshold 95): computes `np.gradient(distances)`, splits where the rate of change exceeds the 95th percentile

The computational cost tradeoff is stark. A BitPeak comparison study on a 75,042-character text found embedding-based semantic chunking cost **under $0.01** — negligible. But LLM-based proposition chunking with GPT-4 on the same text cost **$29.33**. Latency is the real cost: semantic chunking takes **5–7 seconds** (embedding bottleneck) versus near-instant for rule-based methods. A Superlinked evaluation across HotpotQA, SQuAD, and QuAC found that `SentenceSplitter` **consistently outperformed** `SemanticSplitterNodeParser` across all datasets and embedding models — sentences are natural meaning boundaries, and semantic grouping can introduce noise, particularly with code, formulas, and non-prose content.

## Structure-aware methods: markdown headers and code ASTs

LangChain's `MarkdownHeaderTextSplitter` is a standalone class (not derived from `TextSplitter`) that splits at header boundaries while preserving the full hierarchy path as metadata. It takes a `headers_to_split_on` parameter — e.g., `[("#", "Header 1"), ("##", "Header 2")]` — and produces `Document` objects where each chunk carries its complete ancestor chain: `{'Header 1': 'API Reference', 'Header 2': 'Authentication'}`. The `strip_headers` parameter (default `True`) removes header lines from chunk content. For large sections, the recommended pattern chains `MarkdownHeaderTextSplitter` with `RecursiveCharacterTextSplitter` for secondary size-based splitting while preserving header metadata.

For source code, **AST-based splitting** represents the state of the art. The cAST paper (CMU, 2025, arXiv:2506.15655) formalizes a recursive split-then-merge algorithm: iterate through AST nodes, greedily pack complete syntactic units into each chunk, recurse into children when a node exceeds the size budget, and merge adjacent small chunks. It uses **non-whitespace character count** rather than line count, because two code segments with identical line counts can have vastly different amounts of actual code. Benchmark results showed a **5.5-point average gain on RepoEval** for StarCoder2-7B, **4.3-point gain on Recall@5** for CrossCodeEval, and **2.67-point gain on Pass@1** for SWE-bench compared to naive splitting.

The `code-chunk` library (Supermemory) implements these principles in production using tree-sitter for multi-language AST parsing. It extracts semantic entities (functions, methods, classes) with full signatures, builds a hierarchical scope tree, and prepends **contextualized text** to raw code — including file path, scope chain, entity signatures, and imports used. This metadata dramatically improves embedding quality since embedding models are trained primarily on natural language.

## What the benchmarks actually show about chunk size

The Dense X Retrieval paper (Chen et al., 2023, arXiv:2312.06648, published at EMNLP 2024) provides the most rigorous comparison of chunking granularity. The authors built "FactoidWiki" — an English Wikipedia corpus decomposed into three granularities: **41.4M passages** (100-word chunks, avg 58.5 words), **114.2M sentences** (avg 21.0 words), and **256.9M propositions** (avg 11.2 words). Propositions are atomic, self-contained factual statements generated by a finetuned Flan-T5-large model ("Propositionizer") that resolves coreference and adds necessary context.

Across five QA datasets and six retrievers, **proposition-level retrieval improved Recall@20 by an average of +10.1 points for unsupervised retrievers** (SimCSE: 49.1→61.7; Contriever: 62.8→70.5) and **+2.2 points for supervised retrievers**. Downstream QA accuracy (EM@100) showed even more dramatic gains: GTR improved from 26.1 to **33.0** (+6.9 points). The biggest improvements appeared on **out-of-distribution datasets** — EntityQuestions showed 17–25% relative Recall@5 improvement — and on long-tail entities, suggesting finer granularity helps most where retrieval is hardest.

Anthropic's Contextual Retrieval blog post (September 2024) attacked chunking quality from a different angle: enriching chunks with LLM-generated context. Using Claude 3 Haiku, they prepend **50–100 tokens** of document-level context to each chunk before embedding. Their evaluation used ~800-token chunks across four knowledge domains (codebases, fiction, ArXiv papers, science papers) and measured retrieval failure rate (1 − Recall@20). **Contextual embeddings alone reduced failures by 35%** (5.7%→3.7%). Adding contextual BM25 yielded **49% reduction** (→2.9%). The full stack — contextual embeddings + contextual BM25 + Cohere reranking — achieved **67% reduction** (→1.9%). The one-time cost: **$1.02 per million document tokens** with prompt caching. Anthropic did not publish chunk-size ablation results; their primary variable was contextual enrichment, not size.

NVIDIA's 2025 benchmark tested token-based chunks at 128, 256, 512, 1024, and 2048 tokens (with 15% overlap) plus page-level and section-level strategies across five diverse datasets. **Page-level chunking achieved the highest average accuracy at 0.648 with the lowest variance** (σ=0.107), making it the most consistent strategy. Among fixed-size approaches, **1024 tokens performed best overall**, but results were highly dataset-dependent: FinanceBench peaked at 1024 tokens, Earnings at 512, and KG-RAG at page-level. Extreme sizes (128 and 2048) consistently underperformed. The takeaway: **256–512 tokens for factoid queries, 1024+ for complex analytical queries**.

A Fraunhofer IAIS study (arXiv:2505.21700) tested fixed-size chunks at 64, 128, 256, 512, and 1024 tokens across six QA datasets with two embedding models. The results revealed a critical interaction: **decoder-based embedding models (Stella) benefit from larger chunks, while encoder-based models (Snowflake) perform better with smaller chunks**. SQuAD (fact-based) peaked at 64 tokens (Recall@1: 0.641), while NarrativeQA (context-requiring) peaked at 1024 tokens (0.107). ChunkBench (arXiv:2603.06976), the first large-scale systematic evaluation across 36 segmentation methods, concluded that **content-aware chunking significantly outperforms fixed-size baselines** with their top strategy — Paragraph Group Chunking — achieving mean nDCG@5 of ~59% versus <0.244 for simple fixed-size character chunking.

Chroma Research's 2024 evaluation quantified the gap between best and worst strategies at **up to 9 percentage points of recall**. Their LLM-based semantic chunker achieved 0.919 recall, while `RecursiveCharacterTextSplitter` ranged from 85.4% to 89.5%, peaking at **400 tokens** (88.1–89.5%).

## Matching chunking strategy to artifact type

**Source code** demands structure-aware chunking. The cAST paper's results — 5.5 points on RepoEval, 4.3 on CrossCodeEval — demonstrate that respecting function and class boundaries materially improves both retrieval and generation. The practical approach: use tree-sitter for multi-language AST parsing, never split inside a function or class unless it exceeds the maximum budget, and for oversized functions, recurse into child AST nodes. Each chunk should include contextual metadata — file path, scope chain, entity signatures, and relevant imports. LangChain's `RecursiveCharacterTextSplitter` offers language-specific modes (`Language.PYTHON`, `Language.JAVASCRIPT`) with code-aware separators, but these are regex-based approximations of true AST parsing and will produce broken code fragments in complex cases. For serious code RAG, AST-based tools like `code-chunk` or `ASTSnowballSplitter` are strongly preferred.

**Markdown documentation** benefits from a two-stage pipeline: first split at header boundaries using `MarkdownHeaderTextSplitter` (preserving the full hierarchy path as metadata on each chunk), then apply `RecursiveCharacterTextSplitter` to any sections that exceed the target size. The header hierarchy metadata enables filtered retrieval — a query about "OAuth2 configuration" can be matched against chunks carrying `{'Header 2': 'Authentication', 'Header 3': 'OAuth2'}`. Code blocks within markdown should be kept intact as atomic units; LangChain's `ExperimentalMarkdownSyntaxTextSplitter` handles this by extracting code blocks as separate chunks with language metadata. Target **256–512 tokens** per chunk to capture complete API descriptions with parameters and examples.

**ADR (Architecture Decision Record) documents** have a well-known structure — Title, Status, Context, Decision, Consequences — that makes custom parsing straightforward. Most ADRs are short enough (**200–1,000 words**) to embed as a single chunk, which preserves the critical Decision-Context-Consequences relationship. For longer ADRs, split at section boundaries using regex or markdown header parsing, but always include the ADR title, number, and status in every chunk's metadata. The Context and Decision sections should ideally remain together or at minimum cross-reference each other, since a Decision chunk without its Context loses essential retrieval signal. Microsoft's Azure Architecture Center recommends custom regex-based parsing for structured documents with known formats — ADRs are a textbook case where this medium-engineering-effort approach yields high returns at low processing cost.

## Conclusion: what the evidence demands

The evidence converges on several non-obvious conclusions. First, **the interaction between embedding model architecture and chunk size matters as much as the chunking algorithm itself** — decoder-based models prefer larger chunks, encoder-based models prefer smaller ones, and ignoring this interaction leaves performance on the table. Second, **proposition-level decomposition delivers the largest single improvement** measured in any study (+10.1 Recall@20 for unsupervised retrievers), but at significant LLM cost ($29+ per large document); contextual enrichment of standard chunks achieves much of the benefit at $1.02 per million tokens. Third, the Superlinked finding that simple `SentenceSplitter` outperformed semantic chunking across all tested datasets challenges the intuition that more sophisticated algorithms always win — sentences are natural semantic units, and embedding-based boundary detection can introduce noise. Fourth, **no single chunk size is optimal**: 256–512 tokens for factoid retrieval, 1024+ for analytical queries, and AST-boundary-based sizing for code. The practical starting point remains recursive character splitting at 400–512 tokens with 10–20% overlap, but teams should treat this as a baseline to beat with content-aware strategies matched to their specific document types and query patterns.
