# Schema design for heterogeneous software knowledge graphs

**The central tension in modeling a knowledge graph over source code, docs, ADRs, configs, and tests is polymorphism versus performance.** Neo4j's multi-label nodes let any node carry multiple type classifications simultaneously — a function is both a `Function` and a `DocumentedEntity` with zero DDL — but property indexes remain scoped to a single label, and the schema-free model accumulates silent inconsistencies over time. Kuzu's structured node tables enforce strong typing and columnar storage that delivers [15×–188× faster analytical queries](https://thedataquarry.com/blog/embedded-db-2/) than Neo4j Community Edition, but it deliberately rejects multi-label nodes, forcing workarounds for polymorphic types. Existing software ontologies like CodeOntology (65 classes, 86 object properties) and OMG KDM (12 packages spanning code, platform, and conceptual layers) provide well-tested edge taxonomies to build on, while RAG-oriented schemas from GraphRAG and LightRAG introduce weight-based, keyword-tagged relationships designed for retrieval. The practical schema design challenge is choosing the right combination of these patterns and planning for continuous evolution as new artifact types emerge.

## Neo4j multi-labels give modeling flexibility at an indexing cost

Neo4j's property graph model allows [zero or more labels on any node](https://neo4j.com/docs/getting-started/appendix/graphdb-concepts/). Creating a node that is simultaneously a function and a documented entity is syntactically trivial: `CREATE (:Function:DocumentedEntity {name: "parse", docstring: "..."})`. Labels are stored as UTF-8 string membership markers, and in Neo4j's [block storage format](https://neo4j.com/docs/operations-manual/current/database-internals/store-formats/), each node occupies a fixed **128-byte block** that can fit approximately 10 labels, 6–7 properties, and 5 relationships inline before spilling to dynamic overflow stores. The legacy aligned format had roughly 4 label slots, making heavy multi-labeling [more expensive on disk](https://medium.com/neo4j/graph-modeling-labels-71775ff7d121).

The performance story pivots on the **token lookup index**, which succeeded the label scan store in [Neo4j 4.3](https://neo4j.com/docs/upgrade-migration-guide/current/version-4/migration/neo4j-indexes/). This index maps labels to node IDs and powers the `NodeByLabelScan` operator. When a query matches on multiple labels (e.g., `MATCH (n:Function:DocumentedEntity)`), the planner runs a `NodeByLabelScan` on the most selective label, then filters results against the second label. This intersection strategy is efficient when label cardinalities differ significantly. For OR-semantics across labels, the recommended pattern is `UNION` across separate scans, since `WHERE n:A OR n:B` [can trigger a full `AllNodesScan`](https://neo4j.com/developer/kb/how-do-i-report-on-nodes-with-multiple-labels/).

The critical constraint: **property indexes are per-label**. A [range index on `Function(name)`](https://neo4j.com/docs/cypher-manual/current/indexes/search-performance-indexes/create-indexes/) does not cover the same property when queried through the `DocumentedEntity` label. Neo4j does not support multi-label property indexes — a limitation [confirmed for vector indexes](https://community.neo4j.com/t/vector-indexing-across-multiple-node-labels/69864/2) as well — meaning that each label-property pair needs its own index. For a knowledge graph with 8–10 artifact types and 5–6 indexed properties each, this means managing 40–60 indexes, which is operationally tractable but requires deliberate planning.

## Kuzu trades polymorphism for columnar query speed

Kuzu takes a fundamentally different position. Every node belongs to [exactly one node table](https://docs.kuzudb.com/cypher/data-definition/create-table/), and all properties must be declared with explicit types before data insertion. Kuzu uses the term "table" deliberately — as its documentation states, "Kuzu is ultimately a relational system in the sense that it stores and processes sets of tuples." Node tables require a primary key, and Kuzu auto-generates a hash index on it.

Multi-label support was a conscious omission. Kuzu's creator Semih Salihoğlu [stated explicitly](https://github.com/kuzudb/kuzu/discussions/3114): "I have always found the multi-label decision of the original property graphs not well-thought. If we were to support multiple labels, I would want to think about how to support a general type hierarchy. However, any system that supports general type hierarchies is losing in performance." His concern traces back to the object-oriented database systems of the 1990s, where type hierarchies consistently undermined storage optimization.

The payoff is architectural. Kuzu stores data in **columnar node groups** with independent compression per column chunk (constant, bit-packing, or dictionary encoding) and zone maps for predicate pushdown. Combined with [vectorized execution, morsel-driven parallelism, and worst-case optimal joins](https://www.cidrdb.org/cidr2023/papers/p48-jin.pdf), this produces dramatic speedups: an independent benchmark on a 100K-node social graph showed Kuzu **18× faster on ingestion** and **15×–188× faster across 9 OLAP queries** versus Neo4j Community Edition. Multi-hop traversals showed the largest gains.

For polymorphic modeling in Kuzu, the practical workarounds are: (1) a single broad table like `CodeEntity` with a `LIST<STRING>` property for type tags, filtered at query time; (2) separate tables (`Function`, `DocumentedEntity`) connected via [relationship table groups](https://docs.kuzudb.com/cypher/data-definition/create-table/) that union multiple FROM/TO pairs under one name; or (3) denormalized duplication across tables, which sacrifices referential identity. None is as elegant as Neo4j's native multi-labels, but the performance difference makes the tradeoff real for analytical workloads like impact analysis across a large codebase.

## Software ontologies converge on a core edge taxonomy

The most comprehensive code-level ontology is **CodeOntology** by [Atzeni and Atzori (ISWC 2017)](https://link.springer.com/chapter/10.1007/978-3-319-68204-4_2), which defines **65 OWL classes and 86 object properties** for Java source code, available under CC BY 4.0 at [doi:10.5281/zenodo.577939](http://doi.org/10.5281/zenodo.577939). Its relationship taxonomy spans structural containment (`hasMethod`, `hasField`, `hasConstructor`), type system relations (`hasType`, `hasReturnType`, `extends`, `overrides`), behavioral references (`references`), and access control (`hasModifier`). It reuses the [XKOS vocabulary](http://codeontology.org/examples) for transitive part-whole relations (`xkos:hasPart`/`xkos:isPartOf`), enabling queries like "all code elements contained within this package" without manual recursive traversal.

The **OMG Knowledge Discovery Metamodel** ([KDM 1.4](https://www.omg.org/spec/KDM/1.4/About-KDM)) operates at a higher abstraction level, organizing 12 packages across four layers. Its Code and Action packages define structural relationships (containment, inheritance, interface implementation) and behavioral relationships (control flow, data flow between statements). The Platform layer adds runtime relationships between components. KDM is the canonical ["software mining" ontology](https://en.wikipedia.org/wiki/Knowledge_Discovery_Metamodel) — it standardizes knowledge discovery across enterprise software artifacts, explicitly covering build systems, UI, events, and persistent data, which extends naturally to configs and deployment artifacts.

IBM's **Graph4Code** ([Abdelaziz et al., K-CAP 2021](https://wala.github.io/graph4code/)) adds a semantic dimension absent from the purely structural ontologies: it builds **inter-procedural data flow and control flow edges** using WALA program analysis, then links functions to documentation and StackOverflow discussions. The [authors note](https://ar5iv.labs.arxiv.org/html/2002.09440) that their approach differs from CodeOntology primarily in using data flow analysis rather than AST-based structural extraction.

The **SEON framework** from the [University of Zurich](http://se-on.org/) bridges code, issues, and version history through a 5-layer ontology pyramid. Its cross-cutting relationships — `hasAuthor`, `dependsOn`, `isSimilarTo`, and change-coupling relations — are directly relevant for a knowledge graph that must connect ADRs to the code they govern and tests to the functions they verify.

Synthesizing across these ontologies, five core edge categories emerge for a software knowledge graph: **structural containment** (file contains class, class declares method), **type system** (extends, implements, parameterizes), **behavioral dependency** (calls, references, data flow), **documentation linkage** (documents, describes, governs), and **evolution coupling** (co-changes-with, supersedes, deprecates).

## GraphRAG and LightRAG edge schemas prioritize retrieval over structure

Microsoft's [GraphRAG](https://microsoft.github.io/graphrag/index/outputs/) stores relationships as flat parquet records with fields `source`, `target`, `description`, `weight`, `combined_degree`, and `text_unit_ids`. The `weight` is an **aggregate** — when the same source-target pair appears across multiple text chunks, the LLM-assigned `relationship_strength` scores are [summed](https://tech.bertelsmann.com/en/blog/articles/how-microsoft-graphrag-works-step-by-step-part-12), making weight a frequency-weighted measure of relationship salience. Descriptions from multiple extractions are LLM-summarized into a single consolidated string. Notably, GraphRAG assigns no keywords or semantic tags to relationships; retrieval relies on entity embedding similarity and Leiden community summaries.

[LightRAG](https://github.com/HKUDS/LightRAG) diverges structurally. Its edge schema uses `src_id`, `tgt_id`, `weight`, `description`, `keywords`, `source_id`, `file_path`, and `created_at`. The **`keywords` field is the distinguishing design choice** — it stores high-level semantic tags (e.g., "collaboration, research" or "conflict, competition") extracted by the LLM alongside each relationship. These keywords enable LightRAG's [dual-level retrieval](https://neo4j.com/blog/developer/under-the-covers-with-lightrag-extraction/): high-level keyword queries match against relationship embeddings (constructed as `"{src_id}\t{tgt_id}\n{keywords}\n{description}"`), while low-level queries match entity embeddings. Unlike GraphRAG's aggregated weight, LightRAG stores per-extraction strength scores and maintains `file_path` and `created_at` timestamps on each edge, providing better provenance tracking.

For a software knowledge graph, the LightRAG schema pattern is more directly applicable. The `keywords` field maps naturally to edge type classification ("calls", "documents", "tests", "configures"), and the `file_path` and `created_at` fields support the provenance requirements inherent in tracking which source file produced each relationship and when. GraphRAG's `combined_degree` and `text_unit_ids` are useful for importance-ranking but less relevant when edges have well-defined semantic types from a controlled vocabulary.

## Schema evolution demands a hybrid of up-front structure and additive flexibility

When a software knowledge graph needs to accommodate a new artifact type — say, adding Terraform configs alongside existing source code and ADR nodes — the practical approach depends heavily on the database engine.

Neo4j's [schema-optional model](https://neo4j.com/docs/getting-started/appendix/graphdb-concepts/) makes this frictionless: creating `(:TerraformConfig {path: "main.tf", ...})` requires no prior DDL. The label appears implicitly. Indexes and constraints can be [added at any time](https://neo4j.com/docs/cypher-manual/current/indexes/search-performance-indexes/create-indexes/) using `IF NOT EXISTS` for idempotency. The [Neo4j-Migrations](https://neo4j.com/labs/neo4j-migrations/2.0/concepts/) framework provides versioned, tracked schema changes (Cypher scripts, XML catalogs, or Java classes), similar to Flyway for relational databases. The risk is "schema debt" — without explicit constraints, different `:ADR` nodes may have incompatible property shapes, and the database offers no compile-time feedback.

Kuzu requires explicit DDL: `CREATE NODE TABLE TerraformConfig(path STRING, ...)` must precede any data insertion. Adding properties to existing tables uses [`ALTER TABLE ... ADD column_name TYPE DEFAULT value`](https://docs.kuzudb.com/cypher/data-definition/alter/), with existing rows receiving NULL or the specified default. New relationship connections require either new `CREATE REL TABLE` statements or `ALTER TABLE rel ADD FROM NewType TO ExistingType`. Kuzu supports [database export/import](https://docs.kuzudb.com/migrate/) for version migration, generating `schema.cypher`, `copy.cypher`, and data files. This is more ceremony than Neo4j, but the schema itself serves as living documentation — you can read the DDL to understand the full graph structure.

Stanford's [CS520 Knowledge Graphs course](https://web.stanford.edu/class/cs520/2020/notes/How_To_Evolve_A_Knowledge_Graph.html) identifies five categories of schema change: changing world (new entities), changing requirements (reclassifying existing types), changing sources (new data feeds), changes affecting inferences (updated rules), and changes requiring redesign (e.g., converting a property to a relationship). For a software knowledge graph, the first three dominate: new artifact types appear as tooling evolves, requirements shift as analysis capabilities expand, and new source integrations (CI/CD metadata, observability data) arrive continuously.

The safest evolution strategy is **additive-only schema change**: add new labels/tables, properties, and relationship types, but never remove or rename existing elements. This guarantees backward compatibility for existing queries. In Neo4j, this is the natural default. In Kuzu, it requires disciplined use of `CREATE ... IF NOT EXISTS` and `ALTER TABLE ADD ... IF NOT EXISTS` in startup reconciliation scripts.

## Conclusion

The schema design decision ultimately depends on workload profile. For an exploration-heavy knowledge graph where the artifact taxonomy is still stabilizing, Neo4j's multi-label flexibility — where a node can be `:Function:DocumentedEntity:TestTarget` without any migration — reduces friction at the cost of index proliferation and schema ambiguity. For a performance-critical analytical system where the artifact taxonomy is well-understood, Kuzu's structured tables deliver order-of-magnitude query speedups but require every type change to flow through DDL.

The edge taxonomy should draw from CodeOntology's structural relationships and SEON's cross-cutting evolution links, augmented with LightRAG-style `keywords` and `weight` fields for retrieval relevance. A practical schema would define five edge super-categories — containment, type-system, behavioral, documentation, and evolution — with specific relationship types nested under each. Schema evolution should follow an additive-only discipline backed by versioned migration scripts, with `IF NOT EXISTS` guards on all DDL operations to ensure idempotency across deployments. The most resilient approach is a schema-as-code pattern: declare the expected graph structure in application configuration, reconcile it against the live database at startup, and create any missing elements automatically.
