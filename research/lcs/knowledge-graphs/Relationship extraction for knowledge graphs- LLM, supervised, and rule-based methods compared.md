# Relationship extraction for knowledge graphs: LLM, supervised, and rule-based methods compared

**Version metadata:** Created 2026-03-10T14:00:00Z · Research synthesis from 25+ primary sources · Covers literature through early 2026

Fine-tuned supervised models still outperform zero-shot LLMs on relation extraction benchmarks by **10–30 F1 points** on complex datasets, but the gap narrows sharply when LLM outputs are used to augment supervised training. [Wadhwa et al. (ACL 2023)](https://aclanthology.org/2023.acl-long.868/) demonstrated that Flan-T5 fine-tuned with chain-of-thought explanations distilled from GPT-3 achieved new state-of-the-art results across all tested RE benchmarks — **80.76 F1 on CoNLL04, 92.17 on ADE, and 95.23 on NYT** — surpassing both pure supervised models and few-shot LLMs. This finding reshapes the conventional framing of "LLM vs. supervised" into a more productive question: how should these approaches be composed? For teams building knowledge graphs from technical corpora, the answer increasingly involves hybrid pipelines where LLMs handle ambiguous extraction, supervised models process at scale, and rule-based methods provide structural validation.

## Benchmark reality: where each method excels and where it breaks

The clearest picture of relative performance comes from head-to-head evaluations on standard RE benchmarks. On **TACRED** — a sentence-level dataset with 42 relation types across 106,264 examples — [SpanBERT](https://aclanthology.org/2020.tacl-1.5/) achieved **70.8 F1** by introducing contiguous span masking and boundary objectives over BERT. Subsequent models pushed this further: [LUKE reached 72.7](https://arxiv.org/pdf/2107.09332), Typed Marker hit **74.6**, and [RAG4RE (Efeoglu & Paschke, 2024)](https://arxiv.org/pdf/2404.13397) recently achieved **86.8 F1** using Flan T5-XL with retrieval-augmented prompting. These numbers must be interpreted carefully — [TACRED Revisited (Alt et al., ACL 2020)](https://aclanthology.org/2020.acl-main.142.pdf) found that **~8% of test labels were incorrect**, and the [Re-TACRED reannotation](https://gstoica27.github.io/assets/pdf/AAAI_Re_TACRED_CR.pdf) changed 22.1% of labels, inflating apparent model improvements by roughly 13 F1 points on average.

For document-level RE, [DocRED](https://aclanthology.org/P19-1074/) presents a harder challenge: 96 relation types across 5,053 documents where **61.1% of relations span multiple sentences**. The current supervised state of the art, [DREEAM (Ma et al., EACL 2023)](https://ar5iv.labs.arxiv.org/html/2302.08675), reaches approximately **80+ F1** with RoBERTa-large via evidence-guided attention. Zero-shot LLMs on DocRED perform dramatically worse — [AutoRE (2024)](https://arxiv.org/html/2403.14888v1) reports that ChatGPT, Mistral-7B, and similar models deliver "far from satisfactory" results, partly because embedding 96+ relation types directly into a prompt is impractical.

The [REBEL model (Babelscape)](https://huggingface.co/Babelscape/rebel-large) takes a distinctive approach, framing RE as sequence-to-sequence generation over BART-large (460M parameters). It achieves **75.44 F1 on CoNLL04 and 92.0 on NYT** by linearizing triplets with special tokens. Meanwhile, few-shot GPT-3 matched REBEL on CoNLL04 (76.53 F1) but collapsed to **61.79 on NYT** — a dataset with 24 relation types that overwhelmed the prompt's capacity. This pattern recurs throughout the literature: LLMs perform competitively when relation schemas are small but degrade rapidly as type inventories grow.

## How GraphRAG and LightRAG actually extract relationships

Both [Microsoft GraphRAG](https://github.com/microsoft/graphrag) and [LightRAG (HKUDS)](https://github.com/HKUDS/LightRAG) use structurally similar LLM prompts for extraction, but diverge sharply in post-processing. GraphRAG's [core extraction prompt](https://github.com/microsoft/graphrag/blob/main/graphrag/prompts/index/entity_extraction.py) instructs the LLM to identify entities as four-field tuples (name, type, description) and relationships as five-field tuples (source, target, description, **numeric strength score**). Default entity types are `organization, person, geo, event`. The prompt includes three few-shot examples and uses a multi-pass "gleaning" mechanism — a continuation prompt stating "MANY entities and relationships were missed" — to improve recall across subsequent LLM calls.

LightRAG's [extraction prompt](https://github.com/HKUDS/LightRAG/blob/main/lightrag/prompt.py) mirrors this structure almost exactly but replaces the numeric strength score with **relationship keywords** — "high-level keywords summarizing the overarching nature, concepts, or themes of the relationship." This seemingly small change enables LightRAG's distinctive dual-level retrieval: low-level queries match specific entities while high-level queries match thematic keyword indices across the full graph.

The critical divergence lies downstream. GraphRAG runs Leiden community detection to build a hierarchy of entity clusters, then generates LLM summary reports for each community. This powers its global search capability but means **graph construction consumes roughly 75% of total indexing cost** according to [Microsoft's documentation](https://microsoft.github.io/graphrag/index/methods/). LightRAG skips community detection entirely, achieving [~6,000x lower token consumption per query](https://arxiv.org/html/2410.05779v1) by relying on vector + graph retrieval instead. Neither system reports precision, recall, or F1 for extraction quality — both evaluate end-to-end answer quality using LLM-as-judge, an approach that [recent analysis (arXiv 2506.06331)](https://arxiv.org/html/2506.06331v1) found suffers from significant position and length biases.

Microsoft also offers [FastGraphRAG](https://microsoft.github.io/graphrag/index/methods/), a hybrid alternative that uses NLTK and spaCy for entity extraction (noun phrases via dependency parsing) and defines relationships through text-chunk co-occurrence — no LLM calls for extraction at all. This dramatically reduces cost but produces noisier graphs. A [2025 SAP study](https://arxiv.org/html/2507.03226v2) validated this general approach, finding that dependency-parsing-based KG construction achieved **94% of LLM-based performance** (61.87% vs 65.83%) on enterprise datasets at a fraction of the cost.

## Prompt engineering that actually moves the needle

The most impactful prompt engineering strategies for relation extraction fall into five categories, catalogued comprehensively by [Yu (2025)](https://medium.com/@yu-joshua/from-prompt-engineering-to-knowledge-graphs-part-1-8b6fa01fd06e): schema-constrained generation, multi-stage pipeline decomposition, chain-of-thought reasoning, few-shot exemplars, and structured output formatting.

**Schema-constrained generation** provides the largest single improvement. [SPIRES (Caufield et al., VLDB 2024)](https://vldb.org/workshops/2024/proceedings/LLM+KG/LLM+KG-12.pdf) uses LinkML schema specifications to define allowed entity types, relation classes, and cardinality constraints, then validates LLM output against existing knowledge graphs. This approach "outperforms OpenAI GPT-3.5t or Llama 2 alone both in terms of precision and recall" on biomedical RE. Similarly, [LangChain's LLM Graph Transformer](https://medium.com/data-science/building-knowledge-graphs-with-llm-graph-transformer-a91045c49b59) found that "defining a clear graph schema, including allowed node and relationship types, improves extraction consistency and performance."

**Multi-stage decomposition** — extracting entities first, then candidate relations, then normalizing — makes each LLM call simpler and failures debuggable. A [zero-shot biomedical RE framework (ScienceDirect 2025)](https://www.sciencedirect.com/science/article/pii/S1476927126001039) using this two-stage approach (NER → RE with predefined schemas) achieved competitive performance with supervised methods on ChemDisGene and CDR benchmarks. For technical relationships like `calls`, `implements`, `depends-on`, and `supersedes`, [GraphAware's analysis](https://graphaware.com/blog/episode-2-gpt-prompt-engineering/) confirms that a generic prompt is "not sufficient for the complex task of Relation Extraction" — each technical relation type requires explicit definition and domain-specific exemplars.

**CoT distillation** represents the frontier. Wadhwa et al.'s key finding was that using GPT-3 to generate chain-of-thought explanations for training data, then fine-tuning Flan-T5 on those explanations, produced models that exceeded both the teacher LLM and all previous supervised baselines. The reasoning traces teach smaller models to disambiguate relation types that surface patterns alone cannot distinguish.

## Failure modes: hallucination, confusion, and attention dilution

LLM-based extraction exhibits three characteristic failure patterns that intensify with corpus complexity.

**Hallucinated relationships** occur when the model generates plausible but unsupported triples. A [Frontiers study on KG construction for medical text (2024)](https://pmc.ncbi.nlm.nih.gov/articles/PMC11250484/) found that even with cyclic verification, precision reached only **95.33%** — implying a ~5% hallucination floor. The [GDELT Project's analysis](https://blog.gdeltproject.org/llm-infinite-loops-failure-modes-the-current-state-of-llm-entity-extraction/) documented more severe failures: entity normalization drift ("President Biden" becoming "President Joe Biden"), missing prominent entities (Pentagon, White House omitted), and infinite output loops consuming maximum token budgets. Their conclusion: "Current generation LLMs are not yet at the point where they can be reliably and robustly used for highly structured codified extraction tasks."

**Subject-object confusion** is particularly damaging for directional technical relationships. [Han et al. (2023)](https://arxiv.org/abs/2305.14450) found that ChatGPT "cannot understand well the subject-object relationships in RE task" — swapping entity order in inputs often failed to change predictions. For relationships like `depends-on` or `supersedes` where directionality is semantic load-bearing, this represents a critical vulnerability.

**Attention dilution in complex documents** causes the most systematic quality degradation. [A 2026 study on structural failures in LLM extraction (arXiv 2602.10881)](https://arxiv.org/abs/2602.10881) found that "performance remains moderate for single-property queries but degrades sharply once tasks require stable binding between variables, roles, statistical methods, and effect sizes." Full relational tuples were extracted with **"near-zero reliability."** The root cause is not entity recognition failure but rather the inability to maintain role consistency and instance completeness as relational complexity increases. Similarly, a [Towards Data Science analysis of GraphRAG](https://towardsdatascience.com/graphrag-in-practice-how-to-build-cost-efficient-high-recall-retrieval-systems/) found that a 4,000-word document produced only 22 nodes in a single pass — chunking and merging dramatically improved recall.

## The overlapping-relations problem quantified

When entity pairs participate in multiple relationship types simultaneously (Entity Pair Overlap, or EPO), all extraction paradigms suffer. [A comprehensive study (arXiv 2409.04934)](https://arxiv.org/html/2409.04934v1) found that recurrent architectures are most susceptible, but PLM and prompt-based methods show "only slight degradation in complex categories." LLM-based models in few-shot settings also "took a hit when multiple relations and overlapping entities" were present. However, LLMs showed a compensating advantage: they "performed significantly better than PLM and Prompt-PLM algorithms at predicting relationships from long input samples" due to larger context windows. The [ACM Computing Surveys RE review (2024)](https://dl.acm.org/doi/full/10.1145/3674501) formalizes three overlap categories — No Entity Overlap (NEO), Single Entity Overlap (SEO), and EPO — noting that seq2seq models face inherent linearization challenges when multiple triplets share entities.

Cross-sentence implicit relations remain the hardest case. [Efeoglu & Paschke (2024)](https://arxiv.org/html/2406.14745v2) found that fine-tuned Llama2-7B with RAG achieved "substantial gains on SemEVAL, where implicit relations are common," but concluded that zero-shot "LLMs lack the capability for logical inference" needed for implicit relations. DocRED's 61.1% inter-sentence relation rate explains why LLMs struggle so severely on this benchmark.

## Cost-quality tradeoffs favor hybrid architectures

The economics strongly favor composite pipelines. [GraphRAG's documentation](https://techcommunity.microsoft.com/blog/azure-ai-foundry-blog/graphrag-costs-explained-what-you-need-to-know/4207978) warns that "standard GraphRAG spends 75% of its token budget before a single question is asked." Processing 10,000 documents daily through GPT-4/Claude costs approximately **$50,000 annually**, while fine-tuned models achieve equivalent accuracy for roughly **$5,000** according to [Nanonets' analysis (2025)](https://nanonets.com/blog/fine-tuned-models-vs-frontier-cost/). The [ExtractGPT study](https://arxiv.org/html/2310.12537v3) quantified this precisely: fine-tuned GPT-3.5 matched GPT-4's **86% F1** at **70x lower cost** for attribute-value extraction.

The most pragmatic production architecture combines methods by complexity tier. [gBuilder (2024)](https://arxiv.org/html/2208.09705v3) exemplifies this approach with "a wide selection of built-in IE operators, including heuristic-based, deep learning-based and even large language model-based extraction methods" in pipelines that "often involve over 30 nodes." The [SAP Enterprise GraphRAG study](https://arxiv.org/html/2507.03226v2) provides the strongest evidence for this strategy: classical NLP techniques handled the bulk of extraction at 94% of LLM quality, with LLMs reserved for ambiguous cases. For technical corpora specifically, [GraphGen4Code (IBM)](https://ar5iv.labs.arxiv.org/html/2002.09440) demonstrated that AST parsing and static analysis — purely rule-based methods — extracted code relationships from 1.3M Python files into **2.09 billion triples**, a scale no LLM-based approach can match economically.

## Conclusion

Three insights emerge from this analysis that challenge common assumptions. First, the "LLM vs. supervised" framing is misleading — **the highest-performing systems use LLMs to generate training signal for smaller models**, not as direct extractors. Wadhwa et al.'s CoT distillation results represent a paradigm shift where LLMs serve as teachers rather than workers. Second, **extraction quality is not the binding constraint** for most knowledge graph applications — entity resolution, schema alignment, and incremental update costs matter more in production. Neither GraphRAG nor LightRAG even measures extraction F1, evaluating only downstream answer quality. Third, **dependency parsing deserves rehabilitation**: the SAP study's finding that classical NLP achieves 94% of LLM extraction quality at negligible cost suggests that many teams are over-investing in LLM-based extraction. The optimal pipeline for technical corpora likely uses structural parsing (AST, dependency trees) for explicit relationships, fine-tuned compact models (REBEL-class) for textual relations, and LLMs only for disambiguation and implicit relation inference — each method deployed where its cost-quality ratio is strongest.
