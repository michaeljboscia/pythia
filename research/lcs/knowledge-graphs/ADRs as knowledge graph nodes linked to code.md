# ADRs as knowledge graph nodes linked to code

**Architecture Decision Records map naturally to knowledge graph structures, with typed relationships connecting decisions to both sibling ADRs and concrete code artifacts.** The standard ADR templates—from Michael Nygard's original four-section format to the richer MADR 4.0 specification—define entity types (DECISION, CONTEXT, OPTION, CONSEQUENCE) and relationship types (SUPERSEDES, AMENDS, AFFECTS) that translate directly into graph nodes and edges. A growing ecosystem of tools already parses ADR markdown programmatically, and recent academic work on architecture knowledge graphs demonstrates how NLP techniques can automatically trace decisions to the source files they govern. What follows is a technical analysis of how these pieces fit together.

## Nygard and MADR define the node schema

Michael Nygard's [2011 blog post](https://www.cognitect.com/blog/2011/11/15/documenting-architecture-decisions) established the canonical ADR with five sections: **Title, Status, Context, Decision, and Consequences**. Each ADR lives as a numbered markdown file (e.g., `doc/arch/adr-NNN.md`), is immutable once accepted, and follows a strict lifecycle—`proposed → accepted → deprecated → superseded`—where supersession creates an explicit link to the replacement record. This minimal structure already encodes two node types (the Decision itself and its Status) and one relationship type (SUPERSEDES/SUPERSEDED_BY).

The [MADR 4.0.0 specification](https://adr.github.io/madr/) substantially enriches this schema. Developed by Oliver Kopp, Anita Armbruster, and Olaf Zimmermann and [published at ZEUS 2018](https://ceur-ws.org/Vol-2072/paper9.pdf), MADR adds **YAML frontmatter** for machine-readable metadata—`status`, `date`, `decision-makers`, `consulted`, and `informed` fields—plus structured body sections including Decision Drivers, Considered Options, Decision Outcome with Good/Bad/Neutral Consequences, a Confirmation section describing how compliance will be verified, and per-option Pros and Cons. The [MADR GitHub repository](https://github.com/adr/madr) provides four template variants (full, minimal, bare, bare-minimal) installable via npm.

For knowledge graph modeling, these templates produce a clear entity taxonomy. A single ADR file yields a **DECISION node** with properties (id, title, date, status) linked to **CONTEXT nodes** (problem statement, decision drivers), **OPTION nodes** (each considered alternative with pros/cons), **CONSEQUENCE nodes** (typed as good, bad, or neutral), and **STAKEHOLDER nodes** (decision-makers, consulted, informed parties drawn from the RACI model). The [Tyree-Akerman template](https://github.com/joelparkerhenderson/architecture-decision-record/tree/main/locales/en/templates/decision-record-template-by-jeff-tyree-and-art-akerman), published in IEEE Software, extends this further with 13 sections including explicit **Related Decisions**, **Related Requirements**, **Related Artifacts**, and **Related Principles** fields—each representing a named edge type in the graph.

## Relationship types form a rich edge vocabulary

The [adr-tools](https://github.com/npryce/adr-tools) project by Nat Pryce implements relationship types as first-class operations. Its `adr new -s 12` command creates a SUPERSEDES edge, automatically marking the target ADR as superseded. The generic `-l` flag supports arbitrary typed links: `adr new -l "5:Amends:Amended by" Use Riak CRDTs` creates a bidirectional AMENDS/AMENDED_BY edge. The `adr link` command post-hoc connects any two ADRs with custom relationship labels, and `adr generate graph` exports the full decision network in **Graphviz DOT format**—already a graph serialization.

Philippe Kruchten's foundational [2004 ontology paper](https://www.researchgate.net/publication/245352502_An_Ontology_of_Architectural_Design_Decisions_in_Software-Intensive_Systems) formalized a much broader edge vocabulary: **constrains, forbids, enables, subsumes, conflicts-with, overrides, is-bound-to, is-a-consequence-of, and is-an-alternative-for**. This ontology classifies decisions into structural, behavioral, and ban (non-existence) subtypes, each with attributes for cost, risk, scope, and rationale. The [Arteon/Quark ontology](https://www.researchgate.net/publication/224101625_Using_ontology_to_support_development_of_software_architectures) adds NFR-specific edges: **improves** and **damages** connect decisions to quality attributes, while **includes** and **excludes** link constraints to architectural components.

Synthesizing across templates and ontologies, a practical knowledge graph schema emerges with these core edge types:

- **SUPERSEDES / SUPERSEDED_BY** — decision replacement chain (Nygard, adr-tools)
- **AMENDS / AMENDED_BY** — partial modification (adr-tools)
- **ADDRESSES** — decision → context/problem (MADR)
- **CONSIDERS** — decision → option (MADR)
- **SELECTS** — decision → chosen option (MADR Decision Outcome)
- **CAUSES** — decision → consequence (all templates)
- **AFFECTS** — decision → quality attribute (Kruchten, ADeX)
- **IMPLEMENTS / IMPLEMENTED_BY** — decision → code artifact (Tyree-Akerman, archgate)
- **CONFLICTS_WITH** — bidirectional tension (Kruchten)
- **DEPENDS_ON** — prerequisite ordering (Zimmermann SOAD framework)

## Machine-readable parsing already works at scale

No general-purpose ADR parser library exists, but multiple tools demonstrate viable parsing strategies. The [adr-tools](https://github.com/npryce/adr-tools) shell scripts use `sed`, `grep`, and `awk` to manipulate known section headers—treating the first non-blank line after `## Status` as the current status value. [Log4brains](https://github.com/thomvaill/log4brains), a TypeScript/Next.js tool, parses markdown through a remark-based AST, extracting title from H1, status from the Status section, and date from filename prefixes (using `YYYYMMDD-slug.md` convention to avoid Git merge conflicts). The [adr-log](https://github.com/adr/adr-log) Node.js package reads heading-level text and optional YAML frontmatter for index generation.

The most graph-aware tool is [archgate](https://github.com/archgate/cli), which extends YAML frontmatter with an explicit **`files` field using glob patterns** to link ADRs to code paths. An archgate ADR header looks like: `id: ARCH-001`, `domain: backend`, `rules: true`, `files: ["src/api/**/*.ts"]`. Its engine matches changed files to relevant ADRs and executes companion `.rules.ts` files that define programmatic conformance checks. The [archgate MCP integration](https://cli.archgate.dev/) exposes this mapping to AI coding agents, telling them which architectural decisions apply to files being modified.

On the embedded side, the [e-ADR project](https://medium.com/olzzio/y-statements-10eb07b5a177) provides Java annotations like `@YStatementJustification` that capture decisions directly in source code, while the [architectural-decision](https://github.com/cspray/architectural-decision) PHP library uses PHP8 Attributes for the same purpose. These approaches create code-to-decision links at write time rather than extraction time. The [Backstage ADR plugin](https://github.com/backstage/community-plugins/tree/main/workspaces/adr/plugins/adr) demonstrates organizational-scale ADR search across repositories, parsing MADR 2.1.2 and 3.x formats. A [comprehensive tooling index](https://adr.github.io/adr-tooling/) maintained by the ADR GitHub organization catalogs dozens of implementations across Bash, Python, Go, Rust, C#, Java, and .NET.

## Linking decisions to code through NLP and heuristics

The hardest edge to create automatically is IMPLEMENTS—connecting a textual decision like "we chose PostgreSQL" to actual database configuration files. Research from the [ADeX/AMELIE system](https://ieeexplore.ieee.org/document/8712348/) at Siemens and TU Munich demonstrates a working NLP pipeline: a UIMA-based processor runs sentence annotation, then **DBpedia Spotlight** for entity linking (recognizing "PostgreSQL" as a database technology), then type annotation classifying entities as Genre, Software, or Concept. ADeX achieves **91% accuracy** extracting decisions from Jira and GitHub issues and **84% accuracy** annotating architectural elements.

The [Maestro tool](https://link.springer.com/chapter/10.1007/978-3-031-66326-0_24) from the University of Groningen applies deep learning—**BERT, DistilBERT, and Sentence-BERT**—to find and classify architectural design decisions in issue tracking systems, then uses similarity algorithms to trace related decisions across communication channels. A [2024 systematic mapping study](https://www.researchgate.net/publication/376738187_Traceability_of_Architectural_Design_Decisions_and_Software_Artifacts_A_Systematic_Mapping_Study) on ADR-to-artifact traceability categorizes approaches into AI-based, documentation models, system tools, and design complement tools, finding that NLP and ML play an increasingly central role.

For practical implementation, a multi-strategy pipeline yields the strongest results. **Named entity recognition** extracts technology mentions from ADR text. **Dependency file parsing** scans `pom.xml`, `package.json`, `requirements.txt`, and `go.mod` for matching packages—"PostgreSQL" maps to `org.postgresql:postgresql` in Maven or `psycopg2` in pip. **Configuration file scanning** searches `application.yml`, `docker-compose.yml`, Terraform files, and Kubernetes manifests for technology-specific keys (`spring.datasource.url=jdbc:postgresql://`). **Import analysis** traverses source code ASTs for relevant package references. **Git cross-referencing** links ADRs to commits via explicit ADR number mentions in commit messages or temporal proximity. Finally, **embedding-based similarity** using Sentence-BERT computes semantic distance between ADR text and code comments or docstrings. Each strategy produces candidate IMPLEMENTS edges with confidence scores; the graph stores all candidates above threshold.

## Architecture knowledge graphs have academic and industrial precedent

Remco de Boer's [2024 position paper](https://dl.acm.org/doi/10.1145/3643657.3643914) at SATrends explicitly argues for "Architecture Knowledge Graphs" using **linked data principles (RDF, W3C standards)** to connect architecture model elements with decision rationale. His ArchiMedes tool, built on the WikiXL platform, publishes ArchiMate model contents as browsable linked data enriched with historical notes and trade-off documentation. The [STAR system at Siemens](https://www.researchgate.net/publication/325529365_Exploring_Enterprise_Knowledge_Graphs_A_Use_Case_in_Software_Engineering), used by **200–300 software architects**, transforms a central architecture knowledge repository into an enterprise knowledge graph with faceted navigation, recommendation engines, and ontology-derived relatedness metrics across design patterns, decisions, and quality attributes.

[ArchUnit](https://www.archunit.org/) provides the compliance verification layer. Its fluent Java DSL encodes architectural rules as executable tests—`classes().that().resideInPackage("..service..").should().onlyBeAccessedByClassesThat().resideInPackage("..controller..")`—creating what amounts to machine-executable decision consequences. MADR 4.0's [Confirmation section](https://adr.github.io/madr/) explicitly references ArchUnit as the mechanism for verifying decision compliance, establishing a formal connection point between the decision graph and the test suite.

A [2016 retrospective by Capilla, Jansen, Tang, Avgeriou, and Babar](https://www.sciencedirect.com/science/article/abs/pii/S0164121215002034) covering ten years of architecture knowledge management identifies three generations of tools and reports that the ADDSS meta-model already encoded decisions composed of styles, patterns, and variation points linked to functional and non-functional requirements. The paper's conclusion—that no single unified AK metamodel will emerge—suggests that knowledge graph approaches, with their schema flexibility, may be better suited than rigid relational models.

## Conclusion

The path from ADR markdown files to a queryable knowledge graph is shorter than it appears. MADR 4.0's YAML frontmatter and structured sections provide machine-parseable input. Kruchten's ontology and adr-tools' link commands define the edge vocabulary. ADeX and Maestro demonstrate that NLP can extract architectural entities from natural language at high accuracy. Archgate proves that glob-pattern file matching creates viable decision-to-code edges. The missing piece is integration: a system that parses ADR repositories into typed graph nodes, runs entity extraction against the codebase, and materializes IMPLEMENTS edges with confidence scores. The entity schema (DECISION, CONTEXT, OPTION, CONSEQUENCE, STAKEHOLDER, CODE_ARTIFACT) and edge types (SUPERSEDES, ADDRESSES, SELECTS, CAUSES, AFFECTS, IMPLEMENTS) are well-established. What remains is engineering the pipeline—and the academic and industrial precedents from Siemens' STAR system to de Boer's Architecture Knowledge Graphs confirm the approach is sound.
