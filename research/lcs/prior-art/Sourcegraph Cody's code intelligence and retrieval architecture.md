# Sourcegraph Cody's code intelligence and retrieval architecture

**Cody assembles LLM context through a two-stage retrieval-then-ranking pipeline that combines Zoekt trigram search, SCIP-based code graph analysis, and local IDE signals — having deliberately abandoned embedding-based semantic search at enterprise scale in favor of keyword retrieval with BM25F scoring.** This architectural decision, driven by scalability and security constraints at codebases exceeding 100,000 repositories, makes Cody's context engine fundamentally different from most AI coding assistants. The system has evolved rapidly since its 2023 beta, transitioning through at least five distinct architectural phases, and now incorporates an agentic context-fetching layer that autonomously gathers and refines context before querying the LLM. Understanding these design choices — and their documented failure modes — offers a practical blueprint for anyone building a similar code-aware AI system.

## The retrieval stack: three signals, one global ranking

Cody's context engine operates as what Sourcegraph's engineers describe as a "[candidate generation system analogous to a traditional recommender system](https://arxiv.org/html/2408.05344v1)": given the user's workspace state and query, find the N most relevant sections of code. The engine distinguishes between two stages. Stage one retrieves candidates, optimizing for recall — "we care more about retrieving all relevant items rather than retrieving only relevant items." Stage two ranks those candidates, optimizing for precision by selecting only items above a relevance threshold or within the token budget.

Three primary retrieval methods feed stage one, as documented in [Cody's core context documentation](https://sourcegraph.com/docs/cody/core-concepts/context): **keyword search** (traditional text-based matching with automatic query rewriting), **Sourcegraph Search** (the full Sourcegraph Search API, powered by Zoekt, running against managed or self-hosted instances), and **code graph analysis** (examining structural relationships between code elements via SCIP). The RecSys '24 paper from Sourcegraph explicitly notes that "[context retrieval sources should be complementary — that is, we want them to retrieve distinct sets of relevant items](https://arxiv.org/html/2408.05344v1)," a design principle that explains why keyword and structural approaches coexist rather than one replacing the other.

The concrete pipeline for chat and commands works as follows, per Sourcegraph's [technical blog post](https://sourcegraph.com/blog/how-cody-understands-your-codebase): the user invokes Cody and selects up to **10 repositories** for context. Cody pre-processes the query by tokenizing the text and applying cleansing steps to produce a standardized representation. The Sourcegraph search engine then scans the selected repositories and ranks file snippets — sub-file chunks, not whole files — using "an adapted form of the **BM25 ranking function** alongside other signals learned and tuned to the specific task at hand." Simultaneously, Cody pulls snippets from open files in the user's IDE to add local context. All snippets from both remote and local sources enter a global ranking, and the top-N snippets (where N depends on snippet length relative to the token budget) are packaged into the prompt alongside a prefix and the user's query.

Autocomplete follows a fundamentally different, [latency-optimized path](https://sourcegraph.com/blog/the-lifecycle-of-a-code-ai-completion). Because completions must appear within sub-second latency, Cody "uses algorithms that prioritize searching local context for autocomplete rather than running remote searches." The autocomplete pipeline uses **Tree-Sitter** parsing (via WASM bindings) to classify the type of completion needed — function body, docstring, method call — and then retrieves context from the active file, open tabs, and recently closed tabs using a **sliding-window Jaccard similarity search**. A few lines above the cursor serve as the reference, and a sliding window over relevant files identifies the best matches. Typically **4–6 context snippets** are packed into the autocomplete prompt, which then goes to a fast completion-tuned model. This local-only retrieval is a deliberate architectural tradeoff: [Sourcegraph's RecSys '24 paper](https://arxiv.org/html/2408.05344v1) notes that latency and token budget are the binding constraints, and "approximate approaches are preferred for performance reasons."

## Inside Zoekt: trigram indexing and BM25F ranking

The backbone of Cody's remote retrieval is [Zoekt](https://github.com/sourcegraph/zoekt), a fast text search engine originally created by Han-Wen Nienhuys (the name is Dutch for "seek"). Zoekt uses **positional trigram indexing**: for each file, it extracts all 3-character sequences and stores them alongside byte offsets in posting lists. The [design document](https://github.com/sourcegraph/zoekt/blob/main/doc/design.md) explains the query execution strategy — for a query like "The quick brown fox," Zoekt looks up just two trigrams (beginning and end, e.g., "The" and "fox") from the posting lists and verifies they appear at the correct distance apart. For regex queries, literal substrings are extracted first, then candidate files are verified against the full pattern.

This architecture yields impressive efficiency. The [Zoekt FAQ](https://github.com/sourcegraph/zoekt/blob/main/doc/faq.md) reports index sizes of approximately **3.5x corpus size on disk** and **~1.2x corpus size in RAM**, since most posting lists can reside on SSD rather than memory. Rare-string queries complete in **7–10ms on warm caches**. For the Linux kernel (55K files, 545MB), indexing takes roughly 160 seconds single-threaded on a laptop. Zoekt supports indexing up to 64 branches per repository, with files identical across branches stored only once.

Ranking in Zoekt underwent a significant upgrade with the introduction of **BM25F scoring** in [Sourcegraph 6.2](https://sourcegraph.com/blog/keeping-it-boring-and-relevant-with-bm25f), which achieved approximately a **20% improvement across all key metrics** in internal evaluations. The central challenge was that standard BM25 treats all text equally, while in code search, matches on file names and symbol definitions are far more meaningful than matches in comments or string literals. Sourcegraph's solution applies BM25F — the multi-field extension of BM25 — by treating each file as having three "fields": content, filename, and symbol definitions. Term frequencies for **symbol matches and filename matches receive a 5x boost** before the BM25 score is computed. The Sourcegraph team chose this boost factor by comparing a grid of options against internal evaluation datasets, finding that "while applying a boost clearly improves search quality, the final ranking is not too sensitive to the exact choice of boost." Zoekt extracts symbol information using [tree-sitter queries for common languages and universal-ctags for the long tail](https://sourcegraph.com/blog/keeping-it-boring-and-relevant-with-bm25f), tracking exact byte offsets of each symbol definition span. This enables BM25F at line level too: each line gets its own BM25 score with symbol-match boosting.

Beyond BM25F, Zoekt applies several [static file-level ranking signals](https://docs.sourcegraph.com/dev/background-information/architecture/indexed-ranking): generated code is down-ranked, vendored and test code receive penalties, files with many symbols or short names (closer to project root) get boosted, and repository priority follows star count (manually adjustable by admins). When SCIP data is available, Zoekt incorporates a **PageRank-inspired file signal** based on inbound reference counts — how many other files reference a given file — to surface widely-used code higher in results.

## SCIP and the code graph layer

The "code graph" retrieval method relies on [SCIP](https://sourcegraph.com/blog/announcing-scip) (Sourcegraph Code Intelligence Protocol), Sourcegraph's successor to LSIF for precise code navigation. SCIP uses a [protobuf schema](https://github.com/sourcegraph/scip/blob/main/scip.proto) with human-readable string symbol IDs (structured as `<scheme> <manager> <package> <version> <descriptor>`), replacing LSIF's opaque numeric IDs. A SCIP index contains documents, each with a list of occurrences that attach semantic information to source ranges — enabling go-to-definition, find-references, and find-implementations across repositories. Available indexers include scip-typescript, scip-java (covering Scala and Kotlin), scip-clang for C/C++, scip-ruby, scip-python, and scip-php.

SCIP delivers **10x faster indexing** compared to the LSIF predecessor (measured when scip-typescript replaced lsif-node), with **4x smaller** gzip-compressed indexes. For Cody specifically, SCIP provides the structural understanding that keyword search cannot: it can follow type hierarchies, trace function calls across files, and identify which symbols a piece of code depends on. The [Cody GA announcement](https://sourcegraph.com/blog/cody-is-generally-available) noted that "graph context takes advantage of Sourcegraph's expertise in parsing code and producing a graph that captures semantic understanding of your code. Pulling in this semantic code graph into context reduces the rate of common LLM hallucinations like type errors and imaginary function names." This is particularly important for autocomplete, where graph context helps prevent the LLM from inventing nonexistent API surfaces.

Sourcegraph provides three tiers of code navigation: search-based (imprecise, available out-of-the-box using ctags/tree-sitter), syntactic fallback, and precise (SCIP/LSIF, requiring indexer setup in the build pipeline). When precise indexes are unavailable, the system degrades gracefully to search-based navigation, which [can produce false positives and negatives](https://sourcegraph.com/blog/announcing-scip).

## Context filtering, packing, and token budget management

The question of what code to include versus exclude is governed by multiple mechanisms operating at different levels. At the administrative level, [context filters](https://sourcegraph.com/docs/cody/capabilities/ignore-context) allow Enterprise admins to control which repositories Cody can access. These use RE2 regex patterns on repository names, with `include` and `exclude` rules. When include patterns are defined, Cody is restricted to matching repositories only; exclude patterns then filter those results further. A notable limitation: when exclude rules are defined, some features like Prompts are disabled entirely.

At the token budget level, Cody enforces [per-model limits](https://sourcegraph.com/docs/cody/core-concepts/token-limits) that determine how much context can be packed into each request. The system separates two distinct budgets for Claude models: **@-mention context** (explicit, user-provided) gets up to 30,000–45,000 tokens (~4,000–6,000 lines of code), while **conversation context** (auto-retrieved plus chat history) gets 15,000 tokens. For GPT-series models, these share a combined 7,000-token budget. Enterprise deployments since Sourcegraph v6.5 can enable an `enhanced-context-window` feature flag that expands input to **150,000 tokens** for Anthropic and Google models, and up to **102,000 tokens** for OpenAI.

The documentation explicitly addresses why Cody doesn't simply fill the largest available context window. "[Research shows that an LLM's ability to retrieve a fact from a context window can degrade dramatically as the size of the context window increases](https://sourcegraph.com/docs/cody/core-concepts/token-limits)," meaning cramming more code is counterproductive. The system's philosophy is to "put the relevant information into as few tokens as possible to avoid confusing the underlying LLM." There are also practical constraints: processing cost scales linearly with window size, and latency increases with larger contexts.

The packing algorithm itself works by computing a global ranking across all context sources and greedily selecting the top-N snippets that fit within the token budget. A [proprietary re-ranker](https://sourcegraph.com/changelog/improved-context-fetching) "sorts relevant context items higher and earlier to the LLM, resulting in a higher quality response." This re-ranker was trained as a **pointwise model** that predicts whether a given context item is relevant to the user's query, per the [RecSys '24 paper](https://arxiv.org/html/2408.05344v1). A significant recent improvement allows **multiple snippets from a single file**, overcoming an earlier limitation of one snippet per file that could miss critical context when a file contained multiple relevant sections. Total data transmitted per request is capped at approximately [28 KB](https://sourcegraph.com/docs/cody/faq).

For monorepo handling, the architecture leverages Sourcegraph's existing scalability. The [remote repository context blog post](https://sourcegraph.com/blog/how-cody-provides-remote-repository-context) claims Sourcegraph has "worked with customers with over **300,000 repositories and monorepos exceeding 90GB**, and Cody has been able to integrate with them for context retrieval without issue." The key is that Cody's retrieval is query-driven — it searches for relevant snippets within selected repositories rather than attempting to embed or index everything into a single vector space. [Monorepo-specific documentation](https://sourcegraph.com/docs/admin/monorepo) notes that careful resource tuning of sourcegraph-frontend, gitserver, and zoekt-webserver services is required, and that the enhanced language detection feature can create excessive load (disableable via `USE_ENHANCED_LANGUAGE_DETECTION=false`).

## Five phases of architectural evolution

Cody's context engine has undergone dramatic changes since its 2023 launch, and understanding this evolution reveals the engineering tradeoffs that shaped the current design.

**Phase 1 (mid-2023): Embeddings-based retrieval.** Cody originally used OpenAI's `text-embedding-ada-002` to create vector embeddings of codebases. User queries were converted to vectors and compared against the codebase vectors via nearest-neighbor search. The context window was limited to **7,000 tokens**. This approach [suffered from three problems](https://sourcegraph.com/blog/how-cody-understands-your-codebase): code had to be sent to OpenAI for processing (a security concern), maintaining embeddings added operational complexity for admins, and vector database search "for codebases with >100,000 repositories is complex and resource-intensive."

**Phase 2 (December 2023): GA with hybrid retrieval.** At Cody 1.0's [general availability launch](https://sourcegraph.com/blog/cody-is-generally-available), the system used "a hybrid dense-sparse vector retrieval system tailored for code and documentation," combining embedding-based and keyword approaches.

**Phase 3 (early 2024): Search-first architecture.** The [pivotal shift](https://sourcegraph.com/blog/how-cody-understands-your-codebase) came when Sourcegraph replaced embeddings with its native search platform as the primary Enterprise retrieval mechanism. The adapted BM25 ranking, combined with over a decade of search quality optimization, proved competitive with or superior to embedding-based retrieval while eliminating the operational burden. The FAQ [explicitly states](https://sourcegraph.com/docs/cody/faq): "Cody does not support embeddings on Cody Enterprise because we have replaced them with Sourcegraph Search." However, Sourcegraph noted that "there are likely opportunities to leverage embeddings in some form in the future."

**Phase 4 (mid-to-late 2024): Expanded windows and re-ranking.** Context windows expanded from 7,000 to [30,000 tokens for @-mentions in March 2024](https://sourcegraph.com/blog/cody-vscode-1-14-0-release). Output limits quadrupled from 1,000 to 4,000 tokens. The proprietary re-ranker and multi-snippet-per-file support rolled out. OpenCtx and MCP integrations enabled pulling context from [external tools like Jira, Linear, Notion, and Google Docs](https://sourcegraph.com/blog/cody-supports-anthropic-model-context-protocol). A novel [layered context architecture](https://sourcegraph.com/blog/towards-infinite-context-for-code) was experimentally tested: a "permalayer" (~90% of the window) containing the logical codebase surrounding the active code, an append-only action history layer, and a global retrieval layer for long-range search results. With Gemini 1.5 Flash's 1M-token window, this architecture achieved improved recall, concision, and helpfulness while reducing hallucination rate and cutting time-to-first-token from 30–40 seconds to approximately 5 seconds through context prefetching and caching.

**Phase 5 (2025): Agentic context fetching.** The [current architecture](https://sourcegraph.com/docs/cody/capabilities/agentic-context-fetching) adds a "mini-agent" that uses LLM reflection and tool use to autonomously gather and refine context before sending the final query. This agent has access to Code Search, codebase file retrieval, terminal commands (with user consent), web browsing, MCP servers, and OpenCtx providers. The reflection step defaults to Gemini 2.5 Flash (falling back to Claude Haiku or GPT 4.1 mini) and performs iterative review loops to ensure comprehensive context. This represents a notable philosophical shift from Cody's earlier [explicit rejection of agent-based context fetching](https://sourcegraph.com/blog/cody-is-generally-available), where the team found that agent-based approaches "compound the latency and randomness of multiple serial inference requests, leading to poor and unreliable context quality." The current design appears to mitigate this by using a separate, fast model for the reflection step rather than the primary chat model.

## Documented failure modes and engineering lessons

Cody's limitations are well-documented across official channels, GitHub issues, and community feedback. These failure modes cluster into several categories that carry implications for anyone designing a similar system.

**Hallucination remains the most severe problem.** Sourcegraph's own measurements show an **18.97% hallucination rate** with standard context windows, reduced to **10.48%** with Gemini 1.5 Flash's [1M-token context window](https://sourcegraph.com/blog/towards-infinite-context-for-code). Community reports corroborate this: [GitHub issue #3948](https://github.com/sourcegraph/cody/issues/3948) describes Cody fabricating files, processes, and entire namespace hierarchies that don't exist in the codebase, then continuing to fabricate after being corrected. A [HackerNews commenter](https://news.ycombinator.com/item?id=37277722) noted that "Claude hallucinated badly, making up functions and their descriptions for most files — in a very believable way" when asked about unfamiliar SDK headers. The Cody README itself [acknowledges](https://github.com/sourcegraph/cody-public-snapshot) that "Cody is often magical and sometimes frustratingly wrong."

**Irrelevant context retrieval degrades response quality.** The [autocomplete lifecycle blog post](https://sourcegraph.com/blog/the-lifecycle-of-a-code-ai-completion) directly states: "We have learned from internal experimentation that adding irrelevant context can make the response quality worse." Community reports on the [Sourcegraph forum](https://community.sourcegraph.com/t/wildy-inappropriate-context-retrieval/2589) describe Cody pulling in auto-generated dependencies written in different languages when an entire repository is included as context. The system lacks built-in filtering for generated versus authored code, though Zoekt's ranking does down-rank files detected as generated.

**Context truncation in long conversations causes "forgetting."** [GitHub issue #2769](https://github.com/sourcegraph/cody/issues/2769) documents how long chat sessions cause earlier context to be dropped, with internal logs showing messages like "Ignored 8 additional context items due to limit reached." This is an inherent constraint of the token budget system: as conversation history grows, less room remains for retrieved context.

**Output truncation limits practical usefulness for large tasks.** Cody's default output limit of 4,000 tokens (approximately 530 lines) means that [large refactors and file translations get cut off](https://github.com/sourcegraph/jetbrains/issues/153) mid-response. The enhanced enterprise windows partially address this with up to 64,000 output tokens for some models.

**Stale index results are a known concern.** The [remote context blog post](https://sourcegraph.com/blog/how-cody-provides-remote-repository-context) acknowledges that "if context is stored remotely and queried as needed, that context index has to be kept fresh so that stale context isn't used to inform the AI." Enterprise Sourcegraph instances re-index periodically, but freshness depends on configuration. For non-Enterprise users, local indexing freshness depends on when the local search tool last ran.

**Cross-file relationship handling degrades with complexity.** While SCIP-based code graph analysis handles standard patterns well — "Where is parseUserToken used?" returns references across an entire monorepo — it [struggles with complex multi-step logic, long dependency chains, and architectural-level questions](https://community.sourcegraph.com/t/cody-comments-on-the-wrong-context/1452). The 10-repository limit on @-mentions creates hard constraints for cross-repository dependency tracing. A [Sourcegraph internal PRD](https://github.com/sourcegraph/cody/issues/1656) acknowledged that "confusion about context is a serious pain point among users."

**Context confusion between general and codebase knowledge** creates a subtle but frustrating failure mode. When users ask general programming questions mid-conversation, Cody sometimes over-constrains to codebase context and [refuses to answer using general knowledge](https://community.sourcegraph.com/t/cody-doesnt-know-when-to-ignore-the-context/591), responding with statements like "After carefully reviewing all the provided codebase context, I cannot find any information."

## Design implications for similar systems

Several engineering lessons emerge from Cody's architecture. First, the shift from embeddings to keyword search with BM25F scoring demonstrates that traditional information retrieval techniques, properly adapted for code's structural properties, can outperform semantic search at scale — particularly when operational simplicity and security matter. The 5x field boost for symbol and filename matches is a straightforward adaptation that yields disproportionate quality gains.

Second, the separation of token budgets between user-explicit context (@-mentions) and auto-retrieved context is a pragmatic acknowledgment that user intent signals are more reliable than algorithmic retrieval. Giving explicit context a **2–3x larger budget** than automatic context ensures that when a developer knows what's relevant, the system respects that judgment.

Third, the evolution from rejecting agent-based context fetching in December 2023 to embracing it in 2025 illustrates how improvements in model speed and quality change architectural feasibility. The current design mitigates the earlier concerns about compounded latency by using a fast, cheap model (Gemini 2.5 Flash) for the reflection step rather than the expensive primary model.

Fourth, the measured ~19% hallucination rate with standard context and ~10% with expanded context provides a concrete quantification of the retrieval precision gap. Any system building on similar architecture should plan for hallucination rates in this range and implement the kind of online guardrails Sourcegraph describes — syntactic checks (does generated code parse?), semantic checks (do types match?), and unit test validation.

## Conclusion

Cody's architecture reveals a system that has deliberately traded semantic search elegance for operational robustness. The decision to replace embeddings with BM25F-powered Zoekt search was driven not by retrieval quality concerns but by the practical realities of securing, maintaining, and scaling vector infrastructure across 100,000+ repositories. The resulting three-signal retrieval stack — keyword search, code graph analysis, and local IDE context — assembled through a recommender-system-style two-stage pipeline, produces context that is demonstrably useful but far from perfect. The **~19% hallucination rate** remains the most consequential limitation, though the trajectory from embeddings to search-first to agentic context fetching shows a team iterating rapidly toward better solutions. For practitioners designing similar systems, Cody's evolution offers a clear lesson: start with the retrieval infrastructure you can operate reliably at your target scale, invest heavily in ranking quality (BM25F's 20% improvement from a conceptually simple field-weighting change), and build the token budget system to strongly preference user-explicit context over algorithmic retrieval.
