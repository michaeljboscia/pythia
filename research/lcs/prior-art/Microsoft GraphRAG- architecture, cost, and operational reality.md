# Microsoft GraphRAG: architecture, cost, and operational reality

GraphRAG transforms conventional retrieval-augmented generation by constructing an LLM-derived knowledge graph from source documents, partitioning it into hierarchical communities, and pre-generating summaries that enable both entity-specific and corpus-wide question answering. The open-source implementation (microsoft/graphrag on GitHub, now at v3.0.1 under MIT license) ships as a monorepo of eight Python packages organized in layered dependency, with a DAG-based indexing pipeline, four search strategies, and a Pydantic-validated configuration system. In practice, **indexing costs range from $0.06 to $120+ depending on corpus size and model choice**, global search queries consume ~150K tokens per invocation at roughly $0.80 each with GPT-4o, and the single most impactful optimization is pairing GPT-4o-mini with a chunk size of 600 tokens—achieving near-GPT-4-Turbo quality at roughly 2% of the cost.

## How the six-phase indexing pipeline builds a knowledge graph

The indexing pipeline executes as a directed acyclic graph of named workflows, orchestrated by `PipelineFactory` in `graphrag/index/workflows/factory.py`. Configuration lives in `settings.yaml`, validated through `GraphRagConfig` (a root Pydantic model in `graphrag/config/models/graph_rag_config.py`), with defaults defined in `graphrag/config/defaults.py`.

**Phase 1: Text chunking.** Raw documents (.txt, .csv, .json) are split into TextUnits—provenance-tracked chunks written to `text_units.parquet`. The `chunks` config section controls `size` (default **1,200 tokens**), `overlap` (default **100 tokens**), and `strategy` (token-based or sentence-based). Phase 2 links documents to their constituent TextUnits, outputting `documents.parquet`.

**Phase 3: Graph extraction.** This is the most LLM-intensive stage. Standard mode (`graphrag index --method standard`) prompts an LLM to extract entities as `(title, type, description)` tuples and relationships as `(source, target, description, strength)` from each TextUnit. The extraction prompt in `graphrag/prompts/index/extract_graph.py` includes few-shot examples and a gleaning mechanism: the `max_gleanings` parameter (default **1**) triggers follow-up prompts asking the LLM to find missed entities. Entity types default to `[organization, person, geo, event]` but are configurable via `extract_graph.entity_types`. After extraction, a summarization step (`summarize_descriptions` config section) merges duplicate entity and relationship descriptions across TextUnits using a separate LLM call, controlled by `max_length` (default **500 tokens**) and `max_input_length`. The fast alternative (`graphrag index --method fast`) bypasses the LLM entirely, using NLTK-based noun phrase extraction and co-occurrence for relationship inference—dramatically cheaper but less semantically rich.

**Phase 4: Community detection.** Implemented in `graphrag/index/workflows/cluster_graph.py`, this phase applies the **hierarchical Leiden algorithm** via the `graspologic-native` library. The algorithm recursively partitions the entity-relationship graph into communities at increasing granularity: Level 0 holds the broadest communities, and higher levels hold progressively more specific sub-communities. Each level provides a mutually exclusive, collectively exhaustive partition of all nodes. The key parameter is `max_cluster_size` in the `cluster_graph` config section; `use_lcc` optionally restricts analysis to the largest connected component, and `seed` ensures reproducible clustering. An optional `prune_graph` section offers fine-grained control over node/edge filtering via `min_node_freq`, `min_node_degree`, `min_edge_weight_pct`, and `remove_ego_nodes`.

**Phase 5: Community summarization.** For each community, `graphrag/index/workflows/community_reports.py` collects member entity descriptions, relationship descriptions, and optional claims, prioritized by combined node degree. This context is sent to an LLM that generates a structured report containing an executive summary and 5–10 key findings, each with an importance rating (1–10). The `community_reports` config section sets `max_length` (default **2,000 tokens**) and `max_input_length` (default **8,000 tokens**). For higher-level communities whose element summaries exceed context limits, the system recursively substitutes child-community summaries (shorter) for their constituent element summaries (longer)—an elegant compression strategy that preserves information hierarchy.

**Phase 6: Embedding.** `graphrag/index/workflows/embed_text.py` generates vector embeddings for text units, entity descriptions, and community report content, writing them to a configurable vector store (LanceDB by default, with Azure AI Search and CosmosDB also supported). An optional `embed_graph` section supports node2vec structural embeddings for visualization. All pipeline outputs are persisted as Parquet files: `entities.parquet`, `relationships.parquet`, `communities.parquet`, `community_reports.parquet`, and optionally `covariates.parquet`.

## Two-tier retrieval bridges specific and holistic questions

GraphRAG's query engine, rooted in `graphrag/query/structured_search/`, implements four search strategies. The two primary modes—local and global search—address fundamentally different information needs.

**Local search** (`graphrag/query/structured_search/local_search/search.py`) targets entity-specific queries like "What are the healing properties of chamomile?" It begins by embedding the user query and computing cosine similarity against all entity description embeddings, retrieving the top-k most similar entities (default **10**, with 2× oversampling to handle invalid IDs). From these seed entities, the system fans out to collect candidate communities (any containing a matched entity), candidate relationships (any edge incident to a matched entity), and candidate text units (any chunk mentioning a matched entity). The `LocalSearchMixedContext` class in `mixed_context.py` then allocates the context window budget proportionally: **50%** for text units (`text_unit_prop`), **10%** for community reports (`community_prop`), and the remaining **40%** for entity and relationship descriptions. Within these budgets, items are prioritized by semantic similarity to the query (entities), combined node degree (in-network relationships), and number of connections to matched entities (out-network relationships). The assembled context feeds a **single LLM call** that generates the response. Key tuning parameters include `top_k_mapped_entities`, `top_k_relationships`, `max_tokens` (default **12,000**), and `conversation_history_max_turns` for multi-turn interactions.

**Global search** (`graphrag/query/structured_search/global_search/search.py`) handles corpus-wide questions like "What are the top themes in this dataset?"—the query-focused summarization task that motivated GraphRAG's creation. Community summaries serve as the **sole data source**. The `GlobalCommunityContext` class loads all community reports at a user-specified hierarchy level, computes an `occurrence_weight` for each (normalized count of text units containing the community's entities), randomly shuffles reports to prevent positional bias, and partitions them into token-bounded batches. In the **map phase**, each batch is sent to the LLM in parallel (controlled by `concurrent_coroutines`), which produces 5+ intermediate key points with importance scores (0–100). Responses scoring zero are filtered out. In the **reduce phase**, surviving responses are sorted by importance, packed into a final context window (bounded by `max_data_tokens`), and sent to the LLM for synthesis into the final answer. This map-reduce pattern means global search requires **many parallel LLM calls**—practitioner Geert Baeke measured 12 parallel calls consuming roughly **150,000 tokens** for a single global query on George Orwell's *1984*, costing approximately **$0.80 with GPT-4o**.

The community hierarchy level parameter is the most consequential choice in global search. Lower levels yield more numerous, detailed reports—higher quality but more expensive. Higher levels produce fewer, broader summaries—cheaper but coarser. A post-paper enhancement, **Dynamic Global Search**, starts from the hierarchy root and uses LLM relevance ratings to prune irrelevant branches before descending, substantially reducing token consumption.

Two additional search modes round out the system. **DRIFT Search** (Dynamic Reasoning and Inference with Flexible Traversal) in `graphrag/query/drift_search/` combines global-style community priming with iterative local search refinement—generating follow-up questions and traversing the graph dynamically. **Basic Search** provides a standard vector RAG baseline for comparison.

## Operational reality: costs, performance, and incremental indexing

The defining operational characteristic of GraphRAG is its indexing cost. Microsoft's own benchmark on the Wizard of Oz novel (~39,000 words) shows the range starkly: **$3.29 with GPT-4-Turbo, $1.64 with GPT-4o, and $0.06 with GPT-4o-mini**. These figures translate to roughly $0.0000088 per token with GPT-4o-mini. For context, vectorizing the same text for traditional RAG costs $0.0056—making even the cheapest GraphRAG indexing **~11× more expensive** than pure embedding. At the extreme end, a practitioner reported spending **$120 to index a 1,000-page PDF with GPT-4-Turbo-preview**, and the TERAG paper estimates that 5GB of legal documents could cost **up to $33,000** to index.

For a **100K-token corpus** (roughly 75,000 words or ~200 pages), extrapolating from Microsoft's per-token benchmarks yields estimated indexing costs of approximately **$0.88 with GPT-4o-mini, $27 with GPT-4o, and $55 with GPT-4-Turbo**. Actual costs depend heavily on entity density and the number of gleaning passes. A practitioner indexed 1 million words (~1.2M tokens) with DeepSeek for just **$8.20**, demonstrating that open/cheaper models can dramatically compress costs.

Query-time costs split sharply between modes. Local queries require a single LLM call consuming roughly **11,500 tokens** (tunable down to ~6,000), completing in about **10 seconds** with GPT-4o—comparable to conventional RAG. Global queries, by contrast, consume **20–30× more tokens** due to the map-reduce pattern and take **20–24 seconds** including retrieval latency.

**Memory and disk consumption** improved significantly in GraphRAG 1.0. Moving vector embeddings from Parquet files to dedicated vector stores achieved an **80% reduction in output Parquet size** and **43% total disk reduction**. Community detection via Leiden operates on an in-memory NetworkX graph, which can be substantial for large corpora. Neo4j analyst Tomaz Bratanic noted that even a modest graph of 13,000 nodes and 16,000 relationships requires approximately **29,000 LLM calls** for element summarization—highlighting that the bottleneck is LLM throughput and rate limits, not RAM.

**Incremental indexing** arrived via the `graphrag update` CLI command in v1.0. It computes deltas between the existing index and new content, reusing cached LLM results for unchanged documents—making re-runs over existing content "significantly faster and cheaper" according to Microsoft. However, practitioners on GitHub report persistent friction: new content can alter community structure, forcing summary recomputation; timestamped output folders cause confusion about which index to query; and handling deletions remains unsupported. The system currently offers no real-time or near-real-time indexing capability.

**Quality trade-offs** are worth noting. Academic benchmarks show GraphRAG achieves a **4.5% improvement in reasoning depth on multi-hop questions** (HotpotQA) compared to vanilla RAG, but suffers **13.4% lower accuracy on simple factual questions** (Natural Questions) and **2.3× higher latency** on average. Entity deduplication remains imperfect—"Jon" and "Jon Márquez" may be treated as separate entities—and no disambiguation mechanism is built in.

For cost-sensitive deployments, Microsoft's own LazyGraphRAG defers all LLM processing to query time, reducing indexing costs to **0.1% of full GraphRAG** (essentially matching vector RAG embedding costs) while achieving comparable quality on both local and global queries at 4% of the global search token budget. The community-developed Fast-GraphRAG by Circlemind offers **6× cost savings** over standard GraphRAG with native incremental update support.

## Conclusion

GraphRAG's architecture elegantly solves the corpus-wide summarization problem that defeats conventional RAG, but at significant cost. The hierarchical community structure—Leiden-detected, LLM-summarized, and map-reduce-queried—is the core innovation, enabling answers that synthesize information scattered across thousands of documents. For practitioners, three insights stand out. First, **model selection dominates cost**: GPT-4o-mini with 600-token chunks delivers near-frontier quality at roughly 2% of GPT-4-Turbo pricing. Second, **local and global search have fundamentally different cost profiles**: local search is competitive with traditional RAG, while global search's map-reduce pattern makes per-query costs 15–30× higher. Third, **incremental indexing exists but remains immature**: community recomputation on graph changes is unavoidable without the architectural shift that LazyGraphRAG represents. The v3.0 monorepo's factory-pattern extensibility, LiteLLM integration supporting 100+ model providers, and configurable vector store backends make GraphRAG highly adaptable—but the decision to adopt it should be driven by a genuine need for multi-hop reasoning or corpus-wide synthesis, not as a default replacement for simpler retrieval strategies.
