# Cognee's knowledge graph architecture and how it stacks up

**Cognee is a modular, open-source framework that builds a triple-store knowledge representation — graph, vector, and relational — from unstructured documents, then retrieves across all three at query time.** Its architecture differs meaningfully from Microsoft's GraphRAG and Hong Kong University's LightRAG: where GraphRAG invests heavily in pre-computed community summaries and LightRAG optimizes for lightweight dual-keyword retrieval, Cognee bets on a pluggable pipeline of composable tasks and 14 distinct search modes. The result is a more flexible but less battle-tested system, currently at version 0.3 with [~13,000 GitHub stars](https://github.com/topoteretes/cognee) and a $7.5M seed round led by Pebblebed. This analysis examines Cognee's extraction pipeline, graph schema, and retrieval mechanisms at the code level, then compares its architectural choices against GraphRAG and LightRAG, and closes with an honest assessment of practical limitations.

## The ECL pipeline: extract, cognify, load

Cognee's processing flow follows what the team calls the **ECL pattern** — a three-operation API that maps to `cognee.add()`, `cognee.cognify()`, and `cognee.search()`. The [official documentation](https://docs.cognee.ai/core-concepts/overview) describes this as the primary interface, and the [source code](https://github.com/topoteretes/cognee/blob/main/cognee/api/v1/cognify/cognify.py) confirms a six-stage pipeline within `cognify()` that does the heavy lifting.

The ingestion step (`cognee.add()`) accepts over 38 data types — PDF, DOCX, CSV, JSON, audio, images, code files — normalizes them to plain text, hashes content for deduplication, and stores metadata in the relational backend. A [dlt (Data Load Tool)](https://www.cognee.ai/blog/deep-dives/relational-database-to-knowledge-graph-cognee-dlt) loader operates behind the scenes, enabling ingestion from APIs and databases beyond flat files. Documents are organized into datasets with ownership and permission metadata, supporting multi-tenant isolation when `ENABLE_BACKEND_ACCESS_CONTROL` is set.

The `cognify()` pipeline then executes six stages in sequence. **Stage 1** classifies documents by type using the `classify_documents` task. **Stage 2** checks user and dataset permissions. **Stage 3** chunks text via `TextChunker` (in `cognee/modules/chunking/`), producing paragraph-sized fragments whose maximum token length is determined by `get_max_chunk_tokens()` from the configured LLM provider. **Stage 4** is the core extraction step: `extract_graph_from_data` (in `cognee/tasks/graph/`) invokes an LLM to identify entities and relationships from each chunk, guided by the `KnowledgeGraph` data model defined in [`cognee/shared/data_models.py`](https://github.com/topoteretes/cognee). **Stage 5** generates concise summaries via `summarize_text`. **Stage 6** persists everything simultaneously — embeddings to the vector store, nodes and edges to the graph store, metadata to the relational store — through the `add_data_points` task in `cognee/tasks/storage/`.

Each stage is wrapped in a `Task` class (from `cognee/modules/pipelines/tasks/task.py`) that enforces typed Pydantic inputs and outputs. Tasks compose into pipelines via `run_pipeline()`, with configurable batch sizes and support for both foreground and background execution. Crucially, only new or updated files are reprocessed on subsequent runs, providing a form of incremental processing at the [document level](https://www.cognee.ai/blog/fundamentals/how-cognee-builds-ai-memory).

## DataPoints, graph schema, and the triple-store design

The fundamental abstraction in Cognee's knowledge graph is the **DataPoint** — a [Pydantic model](https://www.cognee.ai/blog/deep-dives/from-data-points-to-knowledge-graphs) defined in `cognee/infrastructure/engine/`. Every node type in the graph is a DataPoint subclass. Fields that reference other DataPoint instances automatically become edges. A `metadata` dictionary with an `index_fields` key controls which fields get embedded in the vector store. This design means the graph schema, the vector index, and the relational table are all declared in one place.

The built-in schema includes several node types. **Document** nodes carry ingestion metadata. **DocumentChunk** nodes hold the chunked text and are vector-indexed on their `text` field (stored as `DocumentChunk_text` in the vector store). **Entity** nodes represent named entities extracted by the LLM — people, places, organizations, and concepts. **TextSummary** nodes contain pre-computed summaries, also vector-indexed. **Triplet** nodes optionally store subject-predicate-object triples with their own embeddings when the `TRIPLET_EMBEDDING` environment variable is set. For code repositories, **CodeFile** nodes track source files with `depends_on` edges derived by [Jedi static analysis](https://www.cognee.ai/blog/deep-dives/repo-to-knowledge-graph).

Edge types include `contains` (Document → DocumentChunk), `mentions` (Chunk → Entity), `is_related_to` (Entity → Entity), `has_summary`, and `depends_on`. The LLM also generates **dynamic relationship types** based on content, so the schema is not fixed. Users can enforce domain-specific schemas by defining custom DataPoint subclasses or loading [RDF/XML ontologies](https://docs.cognee.ai/core-concepts/graph-generation) through the ontology resolver system in `cognee/modules/ontology/`. The open-source version supports manually defined ontologies; auto-generation of ontologies is restricted to the commercial offering.

What makes this triple-store approach distinctive is the **simultaneous write** in Stage 6. The `add_data_points` function creates a graph node, generates its vector embedding, and writes relational metadata in a single operation. This ensures that every entity in the graph has a corresponding vector representation and vice versa — a tight coupling that enables the hybrid retrieval modes described below. The [Kuzu database integration](https://blog.kuzudb.com/post/cognee-kuzu-relational-data-to-knowledge-graph/) demonstrates how this works with an embedded graph engine, while production deployments can swap to Neo4j, FalkorDB, or Memgraph via environment configuration.

## Fourteen retrieval modes and how hybrid search works

Cognee's retrieval layer is its most architecturally distinctive feature. The [search documentation](https://docs.cognee.ai/core-concepts/main-operations/search) and the [retriever deep-dive blog post](https://www.cognee.ai/blog/deep-dives/the-art-of-intelligent-retrieval-unlocking-the-power-of-search) describe **14 search modes**, each implemented as a class extending `BaseRetriever` (in `cognee/modules/retrieval/base_retriever.py`). Every retriever follows a three-step protocol: `get_retrieved_objects` fetches raw data, `get_context_from_objects` converts it to context, and `get_completion_from_context` generates an LLM answer (skippable with `only_context=True`).

The default mode, **GRAPH_COMPLETION**, is the signature hybrid retrieval. It operates in three phases. First, it performs vector similarity search across chunk, summary, and entity embedding indices to find semantically relevant content. Second, it resolves graph triplets from the top vector results, building a local subgraph of connected entities and relationships. Third, it serializes this subgraph — node descriptions and relationship descriptions — into plain text and passes it alongside the query to an LLM for grounded answer generation.

Other notable modes include **RAG_COMPLETION** (standard top-k chunk retrieval), **GRAPH_COMPLETION_COT** (iterative chain-of-thought graph reasoning), **GRAPH_COMPLETION_CONTEXT_EXTENSION** (LLM-suggested follow-ups that expand graph context across rounds), **TRIPLET_COMPLETION** (vector search directly on triplet embeddings), **NATURAL_LANGUAGE** (LLM translates queries to Cypher for execution against Neo4j), and **CHUNKS_LEXICAL** (Jaccard token-level similarity without semantic embeddings). The **FEELING_LUCKY** mode lets the LLM auto-select the best retrieval strategy.

This breadth of retrieval options reflects Cognee's positioning as a framework rather than a fixed pipeline. Custom retrievers can be registered with `use_retriever(SearchType, RetrieverClass)`, and a `session_id` parameter enables conversational memory that carries short-term context across queries. The [auto-optimization feature](https://www.cognee.ai/blog/cognee-news/product-announcement-auto-optimization) introduced in late 2025 normalizes user feedback into sentiment scores, attributes them to graph edges used in answer generation, and aggregates signals over time to reinforce stronger retrieval paths.

## How Cognee's graph construction compares to GraphRAG and LightRAG

The three systems share a common foundation — LLM-based entity and relationship extraction from document chunks — but diverge sharply in what they do after extraction.

**Microsoft's GraphRAG** ([arXiv:2404.16130](https://arxiv.org/html/2404.16130v2)) adds a substantial post-extraction layer. After extracting entities and relationships with a prompt that targets people, places, organizations, and events (configurable to domain-specific types), GraphRAG applies the **Leiden community detection algorithm** to cluster related entities hierarchically. Each community then receives an LLM-generated **community report** — a structured summary covering key entities, relationships, and claims. These reports are the foundation of GraphRAG's **global search** mode, which uses a map-reduce approach: community reports are shuffled into chunks, each generates an intermediate answer with a helpfulness score, and the best intermediate answers are synthesized into a final response. This makes GraphRAG uniquely capable of answering corpus-wide thematic questions ("What are the main themes in this dataset?"), but at significant cost. The [Microsoft Research blog](https://www.microsoft.com/en-us/research/blog/graphrag-new-tool-for-complex-data-discovery-now-on-github/) reports that global search consumes ~610,000 tokens per query in some configurations, versus single-digit thousands for local search.

GraphRAG's entity extraction uses a **gleaning mechanism** — after initial extraction, the LLM is asked whether all entities were found, with a logit bias of 100 forcing a binary yes/no answer. If "no," a follow-up prompt aggressively requests missed entities. The [paper](https://arxiv.org/html/2404.16130v2) found that **600-token chunks extracted almost twice as many entity references** as 2,400-token chunks, though at higher cost. Entity resolution relies on exact string matching, with community detection partially compensating for duplicates by clustering variants together. GraphRAG also supports optional **claim extraction** (time-bound factual statements about entities), though this is off by default because it "generally requires prompt tuning to be useful."

**LightRAG** ([arXiv:2410.05779](https://arxiv.org/html/2410.05779v1), EMNLP 2025) takes the opposite approach — it eliminates the community layer entirely. After extracting entities and relationships, LightRAG generates **key-value pairs** for each: entity keys are simply entity names, while relationship keys include high-level thematic keywords derived by the LLM. These keywords are the critical innovation. At query time, LightRAG extracts both **high-level keywords** (thematic concepts) and **low-level keywords** (specific entities) from the user query, then runs vector similarity against relationship embeddings for high-level retrieval and entity embeddings for low-level retrieval. Both paths expand results by gathering 1-hop graph neighbors. This dual-level approach handles both specific and thematic queries without pre-computing community summaries, making it roughly **6,000× cheaper per query** than GraphRAG's global search according to the [LightRAG paper](https://arxiv.org/html/2410.05779v1).

LightRAG's default entity types — organization, person, geo, event, category — are defined in [`lightrag/prompt.py`](https://github.com/HKUDS/LightRAG/blob/main/lightrag/prompt.py) and configurable via `addon_params`. Its incremental update mechanism is also significantly simpler: new entities and edges are merged via graph union (V̂_new = V̂ ∪ V̂') without requiring community restructuring, a meaningful advantage for evolving corpora.

**Cognee sits between these two architectures.** It does not use community detection or pre-computed community summaries, aligning more closely with LightRAG's approach. However, it adds a summarization stage that creates TextSummary nodes — lighter than GraphRAG's community reports but providing a similar abstraction layer. Its ontology resolver system offers something neither competitor provides: the ability to ground extraction in formal RDF/XML ontologies, constraining the LLM's output to domain-specific entity types and relationships rather than relying solely on prompt engineering.

The extraction quality differences follow from these architectural choices. GraphRAG's gleaning mechanism and community summarization produce the most comprehensive coverage of a corpus, but at **281 minutes of indexing time** for a podcast dataset and costs that can reach [$6–7 for 32,000 words](https://microsoft.github.io/graphrag/index/default_dataflow/) with GPT-4o. LightRAG's keyword-enhanced retrieval achieves **competitive comprehensiveness** (45–55% win rate against GraphRAG in head-to-head evaluations) while dramatically reducing cost. Cognee's [own benchmarks](https://www.cognee.ai/blog/deep-dives/competition-comparison-form-vs-function) on 24 HotPotQA questions across 45 cycles report that it "performed slightly to significantly better across measurements" than LightRAG, Graphiti, and Mem0, though the team honestly acknowledges that LLM-as-judge metrics are "signals, not final verdicts."

The schema comparison reveals important differences in philosophical approach:

| Aspect | Cognee | GraphRAG | LightRAG |
|--------|--------|----------|----------|
| **Entity schema** | Pydantic DataPoint subclasses; dynamic LLM types + optional RDF ontology | Prompt-defined (default: person, place, org, event); domain-customizable | Flat list (org, person, geo, event, category); configurable |
| **Relationship schema** | Dynamic LLM-generated + structural (contains, mentions) | Weighted edges with description + numeric strength (1–10) | Edges with thematic keywords + description |
| **Post-extraction enrichment** | TextSummary nodes, optional triplet embeddings, memify pruning | Leiden community detection → hierarchical community reports | Key-value profiling with thematic keywords |
| **Entity resolution** | LLM deduplication passes | Exact string matching + community clustering | Simple name matching |
| **Incremental updates** | Document-level (new files only), but graph uses delete-and-replace | Supported since v0.5.0, but community restructuring is expensive | Graph union (V̂ ∪ V̂') — no rebuild needed |

## Practical limitations and what breaks at scale

Cognee's most significant limitation is **unproven scalability**. The [Memgraph community call transcript](https://memgraph.com/blog/from-rag-to-graphs-cognee-ai-memory) from October 2025 reveals that processing 1 GB of data took approximately 40 minutes using over 100 containers — a rate that makes terabyte-scale processing impractical without substantial infrastructure. Cognee's founder acknowledged that "gaps remain in scaling to terabyte-sized datasets," even as marketing materials claim distributed architecture handling "gigabytes to terabytes." The HotPotQA benchmarks that form the basis of Cognee's published evaluations used just **24–50 questions** — a small corpus by any standard.

By comparison, GraphRAG's paper tested on datasets in the **1 million token range** (~750K words), and the community reports production usage up to similar scales. LightRAG's [UltraDomain benchmark](https://arxiv.org/html/2410.05779v1) included a legal dataset of 94 documents totaling ~5.1 million tokens. Neither system has published convincing results at truly massive scale (hundreds of millions of tokens), but both have been stress-tested with significantly larger corpora than Cognee's published benchmarks.

The **update mechanism** presents another practical concern. Cognee's graph updates use a delete-and-replace approach rather than true incremental graph mutation. As stated in the project's Q&A: "Incremental updates on unstructured data are difficult to implement reliably... the current approach is to clean and replace rather than update incrementally." This contrasts unfavorably with LightRAG's simple graph union and means that adding new documents to a large Cognee knowledge base may require reprocessing related graph regions.

Real-world deployment issues surface prominently in [GitHub issues](https://github.com/topoteretes/cognee/issues). Issue [#1557](https://github.com/topoteretes/cognee/issues/1557) reports that fresh installations fail to execute the tutorial notebook — `cognee.add` succeeds but `cognify` and `search` fail with "Empty graph projected from the database." Issue [#1869](https://github.com/topoteretes/cognee/issues/1869) documents broken Docker deployments where the Next.js frontend throws errors and the backend subprocess dies silently without restart supervision. Provider compatibility issues affect [Gemini](https://github.com/topoteretes/cognee/issues/1530), [LM Studio](https://github.com/topoteretes/cognee/issues/1742), and [Qdrant configurations](https://github.com/topoteretes/cognee/issues/1756). The MCP server integration [fails with IDEs other than Cursor](https://github.com/topoteretes/cognee/issues/1465).

Several architectural patterns are **genuinely reusable** outside Cognee's specific ecosystem. The DataPoint abstraction — declaring graph nodes, vector indices, and relational tables in a single Pydantic class — is an elegant pattern for any triple-store system. The `BaseRetriever` protocol with its three-step `get_objects → get_context → get_completion` flow provides a clean abstraction for building new retrieval strategies. The database adapter pattern supports swapping graph stores (Kuzu, Neo4j, FalkorDB, Memgraph, NetworkX), vector stores (LanceDB, Qdrant, pgvector, Weaviate, Milvus), and relational stores (SQLite, PostgreSQL) via environment configuration alone.

However, certain components are **tightly coupled** to Cognee's specific architecture. The `cognify()` pipeline fundamentally depends on LLM calls for entity extraction — there is no NLP-only fallback analogous to GraphRAG's FastGraphRAG mode. The [Dreamify optimizer](https://www.cognee.ai/blog/deep-dives/ai-memory-tools-evaluation), which treats the pipeline as a parameterized process for hyperparameter tuning, is proprietary and unavailable in the open-source release. The Memphis memory algorithms (`memify()`) that prune stale nodes and reweight edges are integrated into the core pipeline rather than being independently deployable. And while tasks are composable, the overall `add → cognify → search` flow is the prescribed pattern with limited flexibility to reorder or skip stages.

The [Cognee research paper](https://arxiv.org/pdf/2505.24478) (arXiv:2505.24478, May 2025) evaluated on three multi-hop QA benchmarks — HotPotQA, TwoWikiMultiHop, and MuSiQue — and positions the system's modularity as its primary contribution. The paper describes Cognee as a "parameterized process" where chunking strategy, entity extraction prompts, graph construction rules, and retrieval configuration are independently adjustable. This is architecturally sound, but the question of whether the system performs reliably at production corpus sizes remains unanswered by published evidence.

## Conclusion

Cognee's architecture makes a genuine contribution to the graph-RAG design space by unifying graph, vector, and relational representations under a single Pydantic-based abstraction layer and offering 14 retrieval modes that span simple chunk lookup to iterative chain-of-thought graph reasoning. Its triple-store simultaneous-write pattern and composable task pipeline are reusable architectural ideas that deserve adoption beyond this specific project.

The comparison with GraphRAG and LightRAG reveals distinct trade-off profiles. GraphRAG's community detection and summarization layer provides unmatched corpus-wide reasoning capability but at high computational cost and complex update semantics. LightRAG's dual-keyword retrieval achieves comparable quality at dramatically lower cost and with trivial incremental updates. Cognee occupies a middle position — more flexible than either, with its ontology support and diverse retrieval modes, but without the rigorous large-scale evaluation that would establish its reliability in production.

The honest assessment is that Cognee is a **well-designed framework at an early maturity stage**. Its v0.3 status, delete-and-replace update strategy, limited published benchmarks, and active GitHub issues around basic installation suggest it is best suited for prototyping and small-to-medium corpus applications today. Teams evaluating it should test with their specific corpus sizes and domain ontologies before committing to production deployment.
