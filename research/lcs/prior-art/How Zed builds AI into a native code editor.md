# How Zed builds AI into a native code editor

**Document ID:** PA-05 · **Version:** 1.0 · **Date:** March 11, 2026 · **Scope:** Technical analysis of Zed editor's AI-powered code intelligence architecture, based on primary sources (Zed engineering blog, official documentation, GitHub repository, developer interviews)

---

Zed's AI code intelligence rests on a distinctive architectural bet: **CRDT-based collaborative buffers, tree-sitter structural parsing, and protocol-first extensibility** combine to create an AI-integrated editor that treats language models as first-class participants in a system originally designed for human collaboration. Rather than bolting AI onto a text editor, Zed's Rust-native, GPU-accelerated architecture enables streaming diffs, transparent prompt construction, and sub-200ms edit predictions — tradeoffs that reveal hard-won lessons for anyone building code intelligence tooling.

Zed is an [open-source code editor built in Rust](https://zed.dev/docs/ai/overview) by a team led by Nathan Sobo (previously Atom at GitHub) and Max Brunsfeld (creator of tree-sitter). The editor ships AI features natively: an Agent Panel for agentic workflows with tool calling, an Inline Assistant for targeted code transformations, and Edit Prediction (Zeta) for per-keystroke completions. Two open protocols — the [Model Context Protocol (MCP)](https://zed.dev/docs/ai/mcp) for tool and context extensibility, and the [Agent Client Protocol (ACP)](https://zed.dev/acp) for external agent integration — anchor Zed's approach to an ecosystem strategy rather than monolithic AI features. This analysis examines the engineering decisions behind these systems.

## Context assembly works through explicit mentions, agent tools, and structured XML

The most critical question for any AI-integrated editor is how it decides what code to show the language model. Zed's answer has evolved significantly: from fully manual context assembly to a hybrid of explicit user mentions and autonomous agent-driven search, deliberately abandoning an earlier embedding-based approach.

**The original assistant panel**, [introduced in June 2023](https://zed.dev/blog/assistant), treated the LLM conversation as a special-purpose text editor. Users manually selected code with `cmd->` to insert it as fenced code blocks. Each inference sent the full conversation history — a stateless model where context compression was user-controlled, not automatic. Nathan Sobo's design philosophy was radical transparency: [every token of context visible and editable](https://zed.dev/blog/zed-ai), with no hidden system prompts. As Mikayla (Zed engineer) described it: the assistant was *"a very, very thin wrapper around the LLM APIs"* where users controlled [everything including the system prompt](https://filtra.io/rust/interviews/zed-aug-25).

**Slash commands** became the structured context injection mechanism. The [August 2024 "Zed AI" launch](https://zed.dev/blog/zed-ai) introduced commands like `/file` (insert file or directory contents), `/tab` (open tab contents), `/terminal` (terminal output), `/diagnostics` (language server errors), `/fetch` (HTTP URL content), and `/symbols` (tree-sitter-extracted structural elements from the current tab). These commands insert content as **folded text** — entire directory trees appear as recursive folds containing the full text of each file, keeping context visible but manageable. Commands are [evaluated once when inserted](https://zed.dev/docs/ai/text-threads) rather than continuously updated, and are extensible via WebAssembly extensions or the MCP protocol.

**The Agent Panel**, [shipped in May 2025](https://zed.dev/blog/fastest-ai-code-editor), shifted to a hybrid model. Users add context explicitly through **@-mentions** (`@file`, `@directory`, `@symbol`, `@thread`, `@rules`, `@diagnostics`) in the message editor, while the agent itself [autonomously searches the codebase](https://zed.dev/docs/ai/agent-panel) using built-in tools — `grep` for regex search, `find_path` for glob-based file discovery, `read_file` for direct content access, and `diagnostics` for language server errors. This operates without pre-indexing: Richard Feldman noted the agent works without needing to *"teach the agent anything about my code base first, or wait for an indexing process to finish."*

At the protocol level, the mention system implements structured context grouping. The `UserMessage::to_request()` method [groups mentions by type into XML tags](https://deepwiki.com/zed-industries/zed/8-ai-agent-system) — `<files>`, `<symbols>`, `<diagnostics>`, and others — defined in `crates/acp_thread/src/mention.rs` as a `MentionUri` enum with types including File, Symbol, Selection, Diagnostics, GitDiff, Fetch, Thread, and Rule. Each type resolves to structured content: full file text, function definitions, compiler errors, or git diffs. This grouping strategy reduces token usage by batching similar mentions, and the latest user message is marked with `cache: true` to [leverage Anthropic's Prompt Caching](https://zed.dev/blog/zed-ai) for fast responses even with thousands of lines of context.

Notably, Zed **experimented with and then removed** an embedding-based semantic index. During 2023, the team built a [`semantic_index` crate](https://zed.dev/blog/this-week-at-zed-12) that used tree-sitter to parse code into symbol objects, sent those spans to the OpenAI embedding API in batches, and cached embeddings to [reduce incremental indexing tokens by ~75%](https://zed.dev/blog/zed-weekly-18). Nathan Sobo [acknowledged he used semantic search heavily for navigation](https://zed.dev/blog/building-a-text-editor-in-times-of-ai) but found result quality insufficient for production. The `semantic_index` crate no longer appears in Zed's current `Cargo.toml` workspace members. The current approach — agent-driven grep and file reading — trades recall for reliability, a significant architectural decision for anyone designing code intelligence systems.

## Tree-sitter provides the structural backbone for nine-plus features

Tree-sitter's role in Zed extends far beyond syntax highlighting into structural code understanding that directly feeds AI context. Max Brunsfeld (tree-sitter's creator and Zed co-founder) [detailed the architecture](https://zed.dev/blog/syntax-aware-editing) in a March 2023 blog post that reveals how deeply the parser is integrated.

Tree-sitter produces **Concrete Syntax Trees (CSTs)** — unlike ASTs, these retain every token's exact location in source text. Zed uses [Generalized LR (GLR) parsing](https://zed.dev/blog/syntax-aware-editing) with context-free grammars and incremental reparsing, meaning only changed regions are re-parsed after edits. The parser's novel error-recovery handles invalid or in-progress code — critical for an editor where code is perpetually in a broken state.

**Tree queries** are the key abstraction. These are compiled into [compact state machines that find all matches in a single depth-first tree walk](https://zed.dev/blog/syntax-aware-editing), maintaining a set of in-progress matches that are advanced or removed as nodes are visited. Every language-specific rule in Zed is expressed as a tree query — no custom procedural code per language. Adding a new language requires only a tree-sitter grammar plus [a set of `.scm` query files](https://zed.dev/docs/extensions/languages):

- **`highlights.scm`** — syntax highlighting with capture names matching theme styles
- **`outline.scm`** — code structure extraction using `@name`, `@item`, and `@context` captures, powering symbol navigation and the `/symbols` slash command that feeds AI context
- **`indents.scm`** — context-aware auto-indentation via `@indent.begin` and `@indent.end` captures
- **`brackets.scm`** — bracket matching and rainbow brackets
- **`injections.scm`** — multi-language document support (CSS in HTML, SQL in Python strings)
- **`runnables.scm`** — detecting executable code blocks (test functions, main functions)

The outline queries are particularly relevant to AI context assembly. They use structured captures — `@name` marks the symbol name, `@item` marks the full definition for nesting detection, and `@context` adds qualifying keywords like `fn`, `pub`, or `async`. A Rust function definition query looks like: `(function_item (visibility_modifier)? @context "fn" @context name: (_) @name) @item`. This same outline infrastructure [powers the `/symbols` command in text threads](https://zed.dev/docs/ai/text-threads), providing tree-sitter-extracted function and class definitions directly to LLM context.

For the earlier semantic index, tree-sitter queries parsed code into symbol objects for embedding generation. The team [reworked the tree-sitter query engine to include options for collapsing nested objects](https://zed.dev/blog/this-week-at-zed-12) — reducing token count while maintaining hierarchical context, which sped up embedding generation and reduced redundant search results. This technique of using tree-sitter to chunk code at semantically meaningful boundaries (function definitions, class bodies) rather than fixed-size windows remains a broadly applicable pattern for code intelligence systems.

Zed's [hybrid native-WASM architecture](https://zed.dev/blog/language-extensions-part-1) for tree-sitter grammars also merits attention. Since tree-sitter parsers are C code, extension grammars are distributed as WebAssembly for safety and portability. But rather than running everything in WASM, Zed extracts the parsing logic (static data tables) from WASM linear memory and runs it natively — **only the lexing phase runs in WASM**, via the Wasmtime engine. This achieves WASM's sandboxing guarantees with near-native parsing performance, bounded by a `MAX_BYTES_TO_QUERY` limit of 1MB for safety.

## MCP integration follows an LSP-inspired protocol-first architecture

Zed's [MCP implementation](https://zed.dev/blog/mcp), developed in collaboration with Anthropic and [announced in November 2024](https://zed.dev/blog/mcp), follows the Language Server Protocol's architectural pattern: a standardized protocol enabling one-to-many integrations. Rather than building custom integrations for each data source, MCP lets a single server implementation (database schema, production logs, analytics) work across any MCP-capable editor.

The implementation lives in the [`crates/context_server`](https://github.com/zed-industries/zed/pull/16103) crate, introduced in PR #16103. It consists of four components: `client.rs` (JSON-RPC communication), `types.rs` (MCP protocol serialization), `protocol.rs` (session lifecycle), and `manager.rs`/`registry.rs` (server pool management). The client communicates over **stdio** using JSON-RPC — the same transport pattern as LSP. Zed implements its own MCP client rather than using an official SDK, though [community discussion suggests migration to the `modelcontextprotocol/rust-sdk`](https://github.com/zed-industries/zed/discussions/29370).

**Currently supported MCP features** are [Tools and Prompts](https://zed.dev/docs/ai/mcp), with handling for the `notifications/tools/list_changed` notification for dynamic tool list reloading. Discovery, Sampling, and Elicitation are not yet supported. Streamable HTTP transport (from the 2025-03-26 MCP spec) is also pending.

MCP servers can be installed in two ways. **Extensions** from the Zed Extension Store provide packaged MCP servers (GitHub, Puppeteer, Brave Search, Figma, Context7). **Custom servers** are configured directly in `settings.json` with either a local command (stdio transport) or a remote URL (HTTP with authentication headers):

```json
{
  "context_servers": {
    "local-server": { "command": "npx", "args": ["-y", "server-package"] },
    "remote-server": { "url": "https://api.example.com", "headers": { "Authorization": "Bearer <token>" } }
  }
}
```

The **per-tool permission system** provides granular security control. Each tool is addressed as `mcp:<server_name>:<tool_name>` and can be set to `confirm` (prompt user), `allow` (auto-approve), or `deny` (block). [Custom agent profiles](https://zed.dev/docs/ai/agent-panel) can selectively enable or disable specific MCP tools alongside built-in tools, creating task-specific tool configurations. A [security advisory (GHSA-cv6g-cmxc-vw8j)](https://github.com/zed-industries/zed/security/advisories/GHSA-cv6g-cmxc-vw8j) documented the risk of malicious MCP configurations in project `.zed/settings.json` files — mitigated with a worktree trust mechanism in v0.218.2+.

For developers building standalone code intelligence MCP servers, several patterns from Zed's integration are directly applicable. First, **stdio transport is the baseline** — every MCP client supports it, and it provides process isolation by default. Second, the **tools-as-capabilities model** works well: Zed's agent profiles demonstrate that granular tool selection per task (read-only for exploration, full access for editing) improves both safety and user confidence. Third, **dynamic tool registration** via `tools/list_changed` notifications enables servers that adapt their capabilities based on project context. Fourth, MCP tools in Zed are [registered via `ContextServerRegistry`](https://deepwiki.com/zed-industries/zed/8-ai-agent-system) and appear alongside built-in tools in the agent's tool inventory, meaning MCP tools compete on equal footing with native capabilities — a strong incentive for server authors to make tools composable and single-purpose.

## CRDT buffers and streaming diffs define the responsiveness architecture

Zed's most architecturally distinctive feature is that its **CRDT (Conflict-free Replicated Data Types) buffer system** — originally designed for real-time human collaboration — naturally accommodates AI agents as participants. Nathan Sobo explained this in a [June 2025 interview](https://freeplay.ai/blog/how-zed-built-an-incredible-agentic-code-editor-a-conversation-with-nathan-sobo): collaborative data structures *"originally designed for humans"* now serve *"handy new human-like entities"* where *"you can follow the agent around... its edits would cleanly interleave with stuff that you're doing."*

The buffer layer uses a **Rope data structure** stored in what the team calls "sum trees" — [copy-on-write B-trees that index everything needed](https://freeplay.ai/blog/how-zed-built-an-incredible-agentic-code-editor-a-conversation-with-nathan-sobo) for concurrent access from multiple threads. This contrasts with Atom's architecture of *"a JavaScript array of strings."* The sum tree structure enables efficient snapshot creation for background processing (tree-sitter parsing, language server communication) while the main thread handles rendering via [GPUI, Zed's custom GPU-accelerated UI framework](https://github.com/zed-industries/zed).

The **streaming diff protocol** for inline transformations represents a key engineering tradeoff. Activated with `ctrl-enter`, the Inline Assistant sends selected code to an LLM and applies changes [token-by-token as they stream back](https://zed.dev/blog/zed-ai). This required a **custom version of the Needleman-Wunsch algorithm** — a dynamic programming algorithm traditionally requiring both complete texts to compute an optimal edit path. Zed's variant [produces diff hunks incrementally](https://zed.dev/blog/building-a-text-editor-in-times-of-ai) as text arrives, finding the lowest-cost edit path between the original code and the partial LLM response. This works because the CRDT buffers can accept concurrent edits from both the streaming LLM response and the user typing simultaneously.

For structured output, Zed uses **XML tag parsing** rather than JSON: `<patch>`, `<edit>`, `<oldText>`, `<newText>` tags structure the model's code modifications. [XML was chosen over JSON](https://www.southbridge.ai/blog/understanding-code-assistance-in-zed) for easier inline text parsing within a text-editor context. During development, AI-generated XML was initially malformed roughly 50% of the time; systematic prompt evaluation improved success rates to over 95%.

## Responsiveness versus context richness involves five deliberate tradeoffs

Zed's engineering blog and team interviews reveal specific, quantified tradeoffs between fast responses and comprehensive codebase understanding.

**Tradeoff 1: Prompt caching over smaller contexts.** Rather than limiting context size for speed, Zed [integrated Anthropic's Prompt Caching beta](https://zed.dev/blog/zed-ai) to enable large context windows — thousands of lines of code — without proportional latency increases. The cache-aware mention system marks the latest user message with `cache: true`, ensuring repeated context (project files, rules) is cached server-side. This lets Zed favor context richness while maintaining responsiveness.

**Tradeoff 2: Agent-driven search over pre-computed embeddings.** The most consequential architectural decision was [removing the semantic index](https://zed.dev/blog/building-a-text-editor-in-times-of-ai). Pre-computed embeddings offered fast retrieval but suffered from stale indexes, privacy concerns (embeddings sent to OpenAI), and result quality that Nathan Sobo described as insufficient. The replacement — agent tools like `grep`, `find_path`, and `read_file` that search on-the-fly — trades retrieval speed for freshness and reliability. The agent decides what to read based on the task, rather than relying on a fixed similarity metric. For developers building code intelligence tools, this suggests that **agentic search may outperform static embeddings** when context relevance depends heavily on task framing.

**Tradeoff 3: Smarter models with deliberate invocation over fast-but-dumb per-keystroke.** Nathan Sobo [explicitly contrasted Zed's philosophy](https://zed.dev/blog/building-a-text-editor-in-times-of-ai) with Copilot's approach: he prefers *"using a smarter model but being more deliberate"* over faster-but-less-capable per-keystroke suggestions. The Agent Panel and Inline Assistant are user-initiated, sending comprehensive context to capable models, rather than firing on every keystroke with minimal context.

**Tradeoff 4: Speculative decoding for edit prediction latency.** The [Zeta edit prediction model](https://zed.dev/blog/edit-prediction) targets **P50 under 200ms and P90 under 500ms**. The key optimization is speculative decoding: since the model predicts edits by rewriting code chunks, generated output often mirrors the input closely. Zeta uses **n-gram search** to identify "jumping-off points" where the output matches the input, enabling parallel token generation that dramatically reduces effective generation time. This "editing by rewriting" approach — where the model rewrites a code snippet around the cursor rather than producing granular insertions — was chosen because [models aren't good at producing granular edits](https://zed.dev/blog/edit-prediction) but excel at rewriting chunks.

**Tradeoff 5: Transparency over magic.** Zed's system prompts are open source (since the editor is GPL-licensed). Nathan Sobo noted on the [Changelog podcast](https://changelog.com/podcast/640): *"We leak them ourselves, because we're open source."* The assistant panel exposes every token sent to the model — no hidden system prompts, no opaque context selection. This transparency costs some convenience (users must understand what context they're providing) but builds trust and debuggability. The [rules system](https://zed.dev/docs/ai/rules) — supporting `.rules`, `.cursorrules`, `CLAUDE.md`, and `AGENTS.md` files — extends this transparency to project-level prompt configuration, with rules auto-included in all Agent Panel interactions.

## The crate architecture reveals a layered, protocol-driven design

Zed's [Rust workspace](https://github.com/zed-industries/zed) organizes AI features across approximately 20 crates in a layered architecture. The `agent` crate contains core business logic: conversation state management in `thread.rs`, tool orchestration, and the `NativeAgent` session coordinator that maintains a shared `ProjectContext` across threads. The `agent_ui` crate handles Agent Panel rendering. The `agent_servers` crate implements ACP protocol communication in `acp.rs`, managing external agent processes via stdio or TCP.

The [Agent Client Protocol (ACP)](https://zed.dev/blog/bring-your-own-agent-to-zed), launched in collaboration with Google, defines a JSON-RPC interface for external agents running as subprocesses. Zed's own native agent was [refactored to use the same code paths](https://zed.dev/blog/bring-your-own-agent-to-zed) as third-party agents, unifying the internal and external agent architectures. Claude Code runs through a [TypeScript adapter](https://zed.dev/blog/claude-code-via-acp) bridging the Claude Code SDK to ACP's JSON-RPC format; Codex required special handling for its [PTY-based terminal model](https://zed.dev/blog/codex-is-live-in-zed). ACP has been adopted by JetBrains, Neovim (via CodeCompanion), and others.

The `language_model` crate defines the `LanguageModelProvider` and `LanguageModel` traits — the abstraction boundary between agent logic and provider-specific API code. Individual provider crates (`anthropic`, `bedrock`, `deepseek`) implement these traits with conversion layers translating between Zed's unified request format and each provider's API. The `context_server` crate implements MCP with its own JSON-RPC client. Thread persistence uses SQLite with zstd compression in the `db.rs` module.

A `spawn_agent` tool enables [subagent creation](https://zed.dev/docs/ai/tools) — each subagent gets its own context window but shares tool access. A `MAX_SUBAGENT_DEPTH` of 1 prevents recursive spawning beyond one level, a pragmatic limit that prevents runaway context explosion.

## What this means for code intelligence tool builders

Zed's architecture offers several concrete lessons. **Tree-sitter queries provide language-agnostic code structure extraction** that can be repurposed from highlighting to chunking to outline generation — a single investment yielding multiple AI-relevant features. The decision to **abandon embeddings in favor of agentic search** suggests that for many use cases, letting the model decide what to read outperforms static retrieval, especially when freshness and task-specificity matter more than recall. **Protocol-first extensibility** via MCP creates an ecosystem moat: MCP servers written for Zed work in any MCP-capable environment, and Zed benefits from servers built for other clients. The **CRDT-as-AI-infrastructure** pattern — originally designed for human collaboration — proved unexpectedly valuable, suggesting that investing in robust concurrent editing primitives pays compound returns when AI agents enter the picture. Finally, Zed's edit prediction work demonstrates that **rewrite-based models with speculative decoding** can achieve interactive latency at the cost of higher bandwidth, a tradeoff increasingly favorable as inference infrastructure improves.
