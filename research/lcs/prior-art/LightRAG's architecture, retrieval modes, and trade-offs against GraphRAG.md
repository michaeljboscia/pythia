# LightRAG's architecture, retrieval modes, and trade-offs against GraphRAG

*Version: March 11, 2026 | Based on LightRAG v1.4.9.x, paper arXiv:2410.05779 (accepted EMNLP 2025)*

**LightRAG eliminates the expensive community-detection pipeline that defines Microsoft's GraphRAG, replacing it with a dual-level keyword-driven vector search over entities and relationships.** This single architectural choice cuts per-query token consumption from ~610,000 tokens to fewer than 100, according to the [original paper's cost analysis](https://arxiv.org/html/2410.05779v1), while enabling incremental document updates that GraphRAG's community-rebuild cycle cannot match. The system now supports six retrieval modes, nine graph storage backends, and a multi-tiered pluggable storage architecture spanning key-value, vector, and graph layers. However, independent evaluation work published in 2025 reveals that LightRAG's headline benchmark gains are [substantially inflated by evaluation biases](https://arxiv.org/abs/2506.06331), and production-scale deployments remain unproven. This report examines the technical architecture in depth, dissects each retrieval mode, compares storage backends, and evaluates the practical gap between LightRAG and GraphRAG.

## How LightRAG builds its knowledge graph from raw documents

LightRAG's indexing pipeline transforms unstructured text into a queryable knowledge graph through three processing stages, each driven by LLM calls on document chunks. Documents are first segmented into chunks of **1,200 tokens** with a 100-token overlap, as described in the [paper's implementation details](https://arxiv.org/html/2410.05779v1).

The first stage, **R(·)**, prompts an LLM to extract entities (nodes) and relationships (edges) from each chunk. The paper gives the canonical example: from "Cardiologists assess symptoms to identify potential heart issues," the system extracts entities "Cardiologists" and "Heart Disease" and the relationship "Cardiologists diagnose Heart Disease." The [GitHub README recommends models with ≥32B parameters](https://github.com/HKUDS/LightRAG) for reliable extraction, with a context window of at least 32K tokens (64K preferred). An `entity_extract_max_gleaning` parameter controls how many extraction loops run per chunk, defaulting to 1.

The second stage, **P(·)**, generates key-value profiles for each entity and relationship. Entity keys use the entity name; relationship keys are enriched by the LLM to include **global themes derived from connected entities**, which is critical for high-level retrieval. Each entity stores `entity_name`, `entity_type`, `description`, `source_id`, and `file_path`. Each relationship stores `src_id`, `tgt_id`, `description`, `keywords`, `weight`, and `source_id`. The weight is an LLM-assigned relationship strength score — [an "educated guess" based on contextual co-occurrence](https://github.com/HKUDS/LightRAG), not a statistically derived metric.

The third stage, **D(·)**, deduplicates identical entities and relationships from different chunks using [map-reduce summarization to merge descriptions](https://arxiv.org/html/2410.05779v1). This reduces graph overhead and prevents redundant nodes. The resulting graph is formally represented as D̂ = (V̂, Ê) = Dedupe ∘ Prof(V, E).

Three parallel vector databases are then populated: `entities_vdb` stores entity embeddings (formatted as `"{entity_name}\n{description}"`), `relationships_vdb` stores relationship embeddings (formatted as `"{src_id}\t{tgt_id}\n{keywords}\n{description}"`), and `chunks_vdb` stores the original text chunk embeddings. This triple-index structure is what enables LightRAG's multi-modal retrieval.

## The six retrieval modes and their technical mechanics

LightRAG exposes six query modes through its [`QueryParam` class](https://github.com/HKUDS/LightRAG): `naive`, `local`, `global`, `hybrid`, `mix`, and `bypass`. The first five represent different retrieval strategies; bypass skips retrieval entirely and sends queries directly to the LLM.

**Naive mode** performs standard vector-similarity retrieval against `chunks_vdb` with no graph involvement. It functions identically to a conventional RAG pipeline — the query is embedded, matched against chunk embeddings, and top-k results are assembled as context. As [Bryan Lee's Neo4j Developer Blog analysis notes](https://medium.com/neo4j/under-the-covers-with-lightrag-retrieval-89a84ef818a3), this mode exists as a baseline and for queries where structured knowledge adds no value.

**Local mode** drives entity-centric retrieval through low-level keywords. The system first prompts the LLM to extract low-level keywords from the query — specific entities, details, and concrete terms. These keywords are embedded and compared via vector similarity against `entities_vdb`. For each matched entity, the system retrieves node properties from the knowledge graph, computes **node degree** (number of connections) as a ranking signal, gathers **one-hop relationships** with their edge metadata (weight, description, keywords), and finally retrieves associated text chunks via `source_id` references. Entities are sorted by degree descending, and relationships by `(rank, weight)` descending. As [the DeepWiki architecture documentation explains](https://deepwiki.com/HKUDS/LightRAG), this graph-centrality-based ranking ensures the most structurally important entities surface first, not merely the most semantically similar.

**Global mode** drives relationship-centric retrieval through high-level keywords. The LLM extracts high-level keywords — overarching concepts and themes — which are embedded and matched against `relationships_vdb`. The system then queries the knowledge graph for actual edge properties between matched node pairs, computes **edge centrality** as the sum of degrees of both endpoint nodes, and sorts by `(rank, weight)` descending. The [paper's keyword extraction prompt](https://arxiv.org/html/2410.05779v1) illustrates this with the query "What are the environmental consequences of deforestation on biodiversity?": high-level keywords include "Environmental consequences" and "Biodiversity loss," while low-level keywords capture "Species extinction" and "Habitat destruction."

**Hybrid mode** combines both local and global paths. It extracts both keyword types, runs entity retrieval and relationship retrieval in parallel, then merges results through a `process_combine_contexts()` function that [deduplicates and interleaves CSV-formatted context blocks](https://deepwiki.com/HKUDS/LightRAG). This mode delivers the balanced performance shown in the paper's ablation studies.

**Mix mode** represents the most comprehensive retrieval strategy and is [recommended as the default when a reranker is enabled](https://github.com/HKUDS/LightRAG). It runs two parallel paths via `asyncio.gather`: `get_kg_context()` (identical to hybrid's dual-level KG retrieval) and `get_vector_context()` (semantic similarity search against `chunks_vdb`). The vector path augments the query with conversation history, retrieves chunks with a reduced top_k (`min(10, query_param.top_k)`), and enriches results with file metadata. Both contexts are then presented to the LLM through a dedicated `mix_rag_response` prompt template that explicitly separates "Knowledge Graph(KG)" and "Document Chunks(DC)" sources, as documented in the [repository's operate.py implementation](https://github.com/HKUDS/LightRAG). Token budgets control context assembly: **6,000 tokens** for entities, **8,000 for relationships**, and **30,000 total**.

The [Neo4j Developer Blog's technical analysis](https://medium.com/neo4j/under-the-covers-with-lightrag-retrieval-89a84ef818a3) provides a useful summary: "Knowledge graph context provides explicit entities, relationships, and supporting facts with clear structure; vector context provides broader semantic matching with full textual context." Mix mode outperforms pure graph or pure vector retrieval when queries span both specific entities and broader themes, when the corpus contains both well-structured relational data and unstructured narrative text, and when factual precision and contextual breadth are both needed. The paper's [ablation study](https://arxiv.org/html/2410.05779v1) confirms this directly: removing either low-level retrieval (-Low variant) or high-level retrieval (-High variant) degrades performance, with the -High variant showing the steepest decline due to excessive focus on specific entities at the expense of thematic understanding.

## Storage backends and their practical performance characteristics

LightRAG's [multi-tiered storage architecture](https://github.com/HKUDS/LightRAG) separates persistence into four distinct categories, each with multiple pluggable implementations. The system maintains **12 storage instances** across these categories: seven key-value stores (text chunks, full documents, full entities, full relations, entity-chunk mappings, relation-chunk mappings, LLM response cache), three vector stores (entities, relationships, chunks), one graph store, and one document-status store.

For **graph storage**, the system now supports nine backends. **NetworkXStorage** is the default, storing the entire graph in memory as a Python NetworkX object and persisting it to a GraphML file. It requires zero external dependencies and is fast for small graphs, but [the documentation explicitly warns](https://github.com/HKUDS/LightRAG) it is single-process only, offers no concurrent write safety, and loads the entire graph into RAM. It is suitable only for prototyping and small-scale research.

**Neo4JStorage** is [explicitly recommended for production](https://github.com/HKUDS/LightRAG) deployments. It uses the `neo4j` Python async driver with Cypher queries for all operations — `MERGE` for upserting nodes/edges, pattern matching for traversals, and degree queries for centrality computation. Workspace isolation is achieved through labels applied to nodes, enabling logical data separation within a single Neo4j instance. The [Neo4j blog's code-level walkthrough](https://medium.com/neo4j/under-the-covers-with-lightrag-retrieval-89a84ef818a3) reveals the actual Cypher queries used: `MATCH (start:base {entity_id: $source_entity_id})-[r]-(end:base {entity_id: $target_entity_id}) RETURN properties(r) as edge_properties` for edge retrieval. Neo4j supports graphs exceeding available RAM through disk-backed storage and offers enterprise features including clustering and role-based access control.

**PGGraphStorage** uses PostgreSQL with the [Apache AGE (A Graph Extension)](https://github.com/HKUDS/LightRAG) plugin, wrapping Cypher queries in SQL executed through `asyncpg`. Each workspace gets its own AGE graph with a workspace prefix. The official documentation is direct about its limitations: "For high-performance graph database requirements, Neo4j is recommended as Apache AGE's performance is not as competitive." A [known bug in the AGE release version](https://github.com/HKUDS/LightRAG) causes empty node/edge properties (`apache/age#1721`), requiring compilation from source to fix. The key advantage is operational simplicity — PGGraphStorage can be combined with PGKVStorage, PGVectorStorage (pgvector), and PGDocStatusStorage for a **unified single-database deployment**, eliminating the need for separate infrastructure components.

**MemgraphStorage** connects via the Bolt protocol (Neo4j-compatible) to [Memgraph](https://github.com/HKUDS/LightRAG), an in-memory graph database written in C++ designed for sub-millisecond latency on hot datasets. It uses openCypher query language and supports read replicas but no built-in sharding. The trade-off is clear: excellent latency when the graph fits in memory, but vertical scaling only — the graph must fit in available RAM.

For **vector storage**, options span NanoVectorDB (default, development-only), PostgreSQL with pgvector, Milvus (with configurable indexes: AUTOINDEX, HNSW, HNSW_SQ, IVF_FLAT), ChromaDB, FAISS, Qdrant, and MongoDB Atlas. For **key-value storage**, options include JSON files (default), PostgreSQL, Redis, and MongoDB. An important constraint noted in the [documentation](https://github.com/HKUDS/LightRAG): storage implementations cannot be changed after documents are added — data migration between backends is not officially supported.

The practical performance hierarchy for graph backends, based on documentation statements and community reports, is: **Neo4j > Memgraph > PostgreSQL AGE > NetworkX** for production workloads. However, a [GitHub issue (#894)](https://github.com/HKUDS/LightRAG/issues/894) reports ingestion throughput of only ~250 records per 10 minutes (~1,500 articles/hour) with Neo4j + OpenAI Tier 5, well below the reporter's target of 80,000 records/hour. Another [issue (#1303)](https://github.com/HKUDS/LightRAG/issues/1303) from a user with 4 million vectors found no confirmed successful million-scale deployment.

## How incremental updates and document deletion prevent orphaned nodes

LightRAG's incremental update mechanism is architecturally straightforward compared to GraphRAG's community-rebuild requirement. When new documents arrive, the system processes them through the [same extraction pipeline](https://arxiv.org/html/2410.05779v1), producing new node and edge sets (V̂', Ê'). These are merged with the existing graph through set union: V̂ ∪ V̂' and Ê ∪ Ê'. Existing nodes receive merged descriptions via map-reduce summarization, combining old and new information. The deduplication function ensures the same entity from multiple documents is represented once. This stands in direct contrast to [GraphRAG, which must rebuild its entire community structure](https://arxiv.org/html/2410.05779v1) — dismantling existing communities, incorporating new entities, and regenerating all community reports.

Document deletion follows a more complex five-step process, [optimized in v1.4.6](https://github.com/HKUDS/LightRAG/releases/tag/v1.4.6) with the addition of entity/relation indexing for faster lookups. The mechanism centers on the `source_id` field that every entity and relationship carries, tracking which document chunks contributed to its creation.

When `delete_by_doc_id` is called, the system first identifies all entities and relationships associated with the document using the `full_entities` and `full_relations` key-value stores, which serve as [indexes mapping document IDs to their extracted entities and relationships](https://github.com/HKUDS/LightRAG). For each identified entity and relationship, the system performs **shared-knowledge detection**: if the entity's `source_id` references only chunks from the deleted document, it is fully removed from the graph store, vector store, and KV store. If the entity also appears in other documents (shared knowledge), it is preserved but its description is **incrementally rebuilt** from the remaining documents' data via map-reduce summarization. After cleanup, corresponding vectors are removed or updated in all three vector databases, related LLM cache entries are cleared, and the document's status record is removed.

This `source_id` cross-referencing is the [primary mechanism for preventing orphaned nodes](https://github.com/HKUDS/LightRAG). Entities with no remaining source documents after deletion are fully purged from the graph. The system also supports targeted operations: `delete_by_entity("name")` removes a specific entity and all its relationships plus embedding vectors, and `delete_by_relation("src", "tgt")` removes a specific edge. An entity-merging capability (`merge_entities`) offers configurable strategies — `concatenate`, `keep_first`, or `join_unique` — for consolidating duplicate entities that escaped automatic deduplication.

## Where LightRAG diverges from GraphRAG in architecture and cost

The [foundational architectural difference](https://medium.com/neo4j/under-the-covers-with-lightrag-extraction-and-retrieval-3ab930172b35) is that GraphRAG builds hierarchical communities of related entities with summary reports and traverses them at query time, while LightRAG retrieves entities and relationships directly through vector similarity on LLM-generated keyword profiles, bypassing community structure entirely. As the Neo4j Developer Blog frames it: "LightRAG builds upon the idea that structured knowledge extracted from raw documents can enhance retrieval quality but without requiring a community summarization layer like Microsoft's version of GraphRAG."

The cost implications are dramatic. The [paper's analysis on the Legal dataset](https://arxiv.org/html/2410.05779v1) shows GraphRAG generating 1,399 communities, with 610 level-2 communities used for retrieval. Each community report averages ~1,000 tokens, producing total retrieval consumption of **610,000 tokens and hundreds of API calls** per query. LightRAG uses **fewer than 100 tokens** for keyword generation and retrieval, requiring a **single API call** — a roughly **6,000× reduction** in per-query token cost.

However, the paper's headline benchmark numbers require significant qualification. A [June 2025 paper from Wuhan University (arXiv:2506.06331)](https://arxiv.org/abs/2506.06331) identified three systematic biases in the evaluation framework used by LightRAG and other GraphRAG papers: **position bias** (changing answer order in evaluation changes win rates by >30%), **length bias** (LLMs favor longer answers), and **trial bias** (LLMs produce different results when evaluating the same answers multiple times). Under their unbiased framework, LightRAG's reported 66.70% win rate versus NaiveRAG on Agriculture dropped to **39.06%**, and its 56.38% win rate versus GraphRAG dropped to **32.33%**. The paper found that "performance gains are much more moderate than reported previously," with tie rates exceeding 20% in most comparisons.

The [GraphRAG-Bench benchmark (arXiv:2506.05690)](https://arxiv.org/html/2506.05690v1) provides additional nuance. On Medical datasets with the Qwen2.5-14B model, LightRAG led in contextual summarization (**69.37% accuracy**) and creative generation (**70.84% F-score**), but other frameworks excelled in different tasks. The benchmark also noted that LightRAG "achieves slight performance improvements" over baseline in retrieval-augmented generation, indicating "retrieved content provides useful but not transformative benefit."

Neo4j's own assessment is measured: ["There are currently no production-level statistics showing that LightRAG delivers meaningful improvements in real-world enterprise environments"](https://medium.com/neo4j/under-the-covers-with-lightrag-retrieval-89a84ef818a3), though they classify LightRAG as a promising technique within the broader GraphRAG family.

## Practical deployment trade-offs and when to choose each system

The decision between LightRAG and GraphRAG reduces to a trade-off between retrieval cost, update flexibility, and query-type strength. LightRAG's per-query cost advantage is unambiguous — the 6,000× token reduction is architectural, not benchmark-dependent. Its incremental update capability avoids GraphRAG's community-rebuild bottleneck, making it the stronger choice for corpora that change frequently. For teams already running PostgreSQL, the unified-stack deployment (PGKVStorage + PGVectorStorage + PGGraphStorage + PGDocStatusStorage) offers meaningful operational simplicity, even if graph query performance trails Neo4j.

GraphRAG retains advantages for certain query profiles. Its hierarchical community summaries are purpose-built for broad thematic questions ("What are the main themes in this corpus?"), where traversing pre-computed summaries can outperform LightRAG's keyword-driven approach. The [paper's own results show GraphRAG slightly outperforming LightRAG on the Mix dataset](https://arxiv.org/html/2410.05779v1) for comprehensiveness and empowerment — notably the most diverse dataset in the evaluation.

For LightRAG specifically, the storage backend choice shapes deployment complexity significantly. NetworkX requires zero infrastructure but caps at small-scale, single-process use. Neo4j adds operational overhead (separate server, license considerations for enterprise features) but is the only [backend explicitly recommended for production graph workloads](https://github.com/HKUDS/LightRAG). PostgreSQL with AGE trades graph performance for stack simplicity. Memgraph offers the lowest latency for graphs that fit in memory but provides no horizontal scaling path.

Production users should be aware of several constraints. The LLM model used for extraction materially affects graph quality — [models below 32B parameters frequently produce empty or noisy graphs](https://github.com/HKUDS/LightRAG/issues/30). Ingestion throughput is bottlenecked by LLM calls and graph writes, with real-world reports suggesting [~1,500 documents per hour](https://github.com/HKUDS/LightRAG/issues/894) as a practical ceiling with Neo4j. Entity normalization (merging different surface forms of the same entity) is not automated in core LightRAG — the [ApeRAG fork](https://github.com/apecloud/ApeRAG) adds this capability along with distributed task queues and Kubernetes deployment. And metadata growth in production (entity/relation `source_id` fields accumulating references) can degrade vector DB performance, addressed in recent releases through [configurable limits (MAX_SOURCE_IDS_PER_ENTITY=300)](https://github.com/HKUDS/LightRAG).

## Conclusion

LightRAG's core contribution is **proving that community detection is not necessary** for graph-enhanced RAG. By replacing community traversal with vector-based keyword matching over entity and relationship profiles, it achieves dramatic cost reductions per query while enabling incremental updates — two properties that GraphRAG's architecture cannot easily provide. The dual-level retrieval paradigm, splitting queries into low-level entity keywords and high-level thematic keywords, provides a principled mechanism for handling both specific and abstract queries through a single graph structure.

The system's **mix mode represents its most mature retrieval strategy**, combining KG-derived structured context with vector-derived semantic context through parallel execution. Its storage architecture is genuinely flexible, with nine graph backends and seven vector backends covering everything from zero-dependency prototyping to enterprise PostgreSQL/Neo4j stacks.

Three caveats temper the narrative. First, the original benchmark gains are **overstated by roughly 2×** according to unbiased evaluation, placing LightRAG's actual advantage over naive RAG in the moderate rather than dominant range. Second, **no confirmed million-scale production deployment exists** in public records. Third, extraction quality is critically dependent on LLM capability, creating a floor on the model size and cost that can practically be used. For teams weighing LightRAG against GraphRAG, the decision hinges on whether per-query cost and update frequency matter more than thematic summarization quality — and whether the operational maturity gap relative to established vector RAG pipelines is acceptable for their use case.
