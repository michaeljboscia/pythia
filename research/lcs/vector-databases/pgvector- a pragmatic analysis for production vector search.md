# pgvector: a pragmatic analysis for production vector search

**pgvector has evolved from a curiosity into a credible vector search engine**, delivering 99% recall at sub-6ms p99 latency on one million 1536-dimensional vectors — performance that would have been unthinkable two years ago. For teams already running PostgreSQL, it eliminates the operational tax of a second data store while preserving ACID guarantees that no purpose-built vector database offers. Yet real limits remain: HNSW index builds consume tens of gigabytes of RAM, write-heavy workloads trigger lock contention, and scaling beyond roughly 10 million vectors on a single node demands careful architectural planning. This report provides the quantitative evidence a senior database engineer needs to make that build-versus-buy decision.

## Index architecture and parameter tuning

pgvector ships two approximate nearest-neighbor index types, each with distinct tradeoffs. **IVFFlat** partitions vectors into inverted lists via k-means clustering, then searches only the closest lists at query time. It builds fast but delivers inferior recall-vs-latency tradeoffs compared to graph-based methods. The [pgvector README](https://github.com/pgvector/pgvector) recommends setting `lists` to `rows / 1000` for tables under one million rows and `sqrt(rows)` above that threshold, with `ivfflat.probes` controlling the recall-speed tradeoff at query time. A critical caveat: IVFFlat requires existing data for its training step, and clusters do not rebalance as data distribution shifts — degraded recall is only fixed by a [full index rebuild](https://alex-jacobs.com/posts/the-case-against-pgvector/).

**HNSW** (Hierarchical Navigable Small World) builds a multi-layer navigable graph and dominates modern pgvector deployments. It can be created on an empty table, requires no training step, and delivers dramatically better recall-vs-latency curves. Three parameters govern its behavior. At build time, `m` (default **16**) controls maximum connections per node and `ef_construction` (default **64**) sets the dynamic candidate list size during graph construction. At query time, `hnsw.ef_search` (default **40**) determines how many candidates the algorithm explores before returning results. Increasing any of these improves recall at the cost of build time, index size, or query latency, respectively.

The most authoritative benchmark for the canonical 1M-vector, 1536-dimension OpenAI embedding workload comes from Jonathan Katz's [year-in-review analysis](https://jkatz05.com/post/postgres/pgvector-performance-150x-speedup/) conducted on AWS r7gd.16xlarge instances (64 vCPU, 512 GiB RAM) running PostgreSQL 16.2. With HNSW parameters of m=16 and ef_construction=256, pgvector 0.7.0 achieves **253 queries per second at 99% recall with a p99 latency of 5.51ms** on a single connection. This represents a roughly **30× QPS improvement** and 30× latency reduction compared to IVFFlat on the same dataset, which managed just 8 QPS with 150ms p99 latency. The general pattern across ef_search values follows a predictable curve: ef_search=40 (default) yields roughly 85–95% recall, ef_search=80–120 pushes to 95–99%, and ef_search=200+ crosses the 99% threshold with diminishing returns beyond 400.

Quantization further shifts the performance frontier. Half-precision indexing via [pgvector's `halfvec` type](https://github.com/pgvector/pgvector) (introduced in v0.7.0) stores each element in 16 bits rather than 32, cutting index size from **7.55 GiB to 3.78 GiB** while actually improving QPS to 282 with a p99 of 4.87ms at 99% recall. Binary quantization using expression indexes on the `bit` type compresses indexes to just **0.46 GiB** — a 16.4× reduction — and still maintains 99% recall when combined with a reranking step over original vectors. Build time drops from 250 seconds to 49 seconds. These numbers, all from the [Katz benchmark](https://jkatz05.com/post/postgres/pgvector-performance-150x-speedup/), demonstrate that pgvector's quantization story is production-ready.

Version 0.8.0 introduced **iterative index scans**, a feature that fundamentally changes pgvector's behavior for filtered queries. Previously, pgvector would find the K nearest neighbors and then apply WHERE clause filters, potentially returning far fewer results than requested. AWS Aurora benchmarks on [10 million 384-dimensional vectors](https://aws.amazon.com/blogs/database/supercharging-vector-search-performance-and-relevance-with-pgvector-0-8-0-on-amazon-aurora-postgresql/) show iterative scans improving filtered recall from as low as **1% to 100%** while simultaneously reducing p99 latency by up to 9.4× (from 123ms to 13ms for basic top-10 queries). This single feature eliminates what was previously pgvector's most painful limitation for real-world applications where metadata filtering is ubiquitous.

## Why PostgreSQL integration changes the calculus

The strongest argument for pgvector is not raw vector search speed — purpose-built databases will often win that race — but the **elimination of architectural complexity** that comes from keeping vectors alongside relational data. This advantage manifests in three concrete ways.

First, **transactional consistency** ensures that a document and its embedding are always in sync. When you update a product description and its embedding in a single PostgreSQL transaction, both commit or both roll back. Pinecone, by contrast, operates under [eventual consistency](https://www.instaclustr.com/education/vector-database/pgvector-vs-pinecone-8-key-differences-and-how-to-choose/) — search results may briefly reflect stale data after mutations. For financial applications, healthcare systems, or any pipeline where vector-relational drift could produce incorrect results, ACID guarantees are not optional. pgvector inherits all of PostgreSQL's isolation levels, row-level security, and crash recovery semantics without any special handling.

Second, **SQL JOINs with vector search** eliminate the dual-query anti-pattern. Consider a product recommendation system that must return only in-stock items within a user's price range, ranked by embedding similarity. With a purpose-built vector database, the application must query the vector store for candidates, extract IDs, query the relational database for inventory and pricing, then merge results in application code — a pattern that introduces latency, consistency gaps, and code complexity. With pgvector, this is a [single SQL query](https://dev.to/polliog/postgresql-as-a-vector-database-when-to-use-pgvector-vs-pinecone-vs-weaviate-4kfi) joining the embeddings table with inventory and pricing tables, filtered by stock status, and ordered by cosine distance. CTEs, window functions, and aggregations all compose naturally with vector operations.

Third, **operational simplicity compounds over time**. pgvector data is included automatically in [pg_dump and pg_restore](https://github.com/pgvector/pgvector). Vector columns replicate through PostgreSQL's WAL-based streaming replication and logical replication (with a fix in v0.6.0). Connection poolers like PgBouncer work without modification. Monitoring via `pg_stat_statements` captures vector query performance alongside relational workloads. Backup, failover, PITR, and audit logging all work identically to any other PostgreSQL data type. As one [practitioner summarized](https://devslane.com/blog/pgvector-vs-purpose-built-vector-db/): "One database means one backup strategy, one monitoring setup, one credential rotation process. That simplicity compounds."

Hybrid search — combining keyword (BM25) and semantic (vector) relevance — is another area where PostgreSQL integration pays off. pgvector works alongside PostgreSQL's built-in `tsvector`/`tsquery` full-text search, and results can be fused using [Reciprocal Rank Fusion](https://jkatz05.com/post/postgres/hybrid-search-postgres-pgvector/) in a single query. Extensions like ParadeDB add [production-grade BM25 scoring](https://www.paradedb.com/blog/hybrid-search-in-postgresql-the-missing-manual) directly within PostgreSQL, enabling hybrid retrieval pipelines that stay entirely within one database engine.

PostgreSQL's native partitioning integrates cleanly with pgvector. The [README documents](https://github.com/pgvector/pgvector) support for declarative partitioning by any column — tenant ID, category, time window — with per-partition HNSW indexes that enable the query planner to prune irrelevant partitions at plan time. This is the primary mechanism for scaling pgvector beyond single-table limits: partition a 50-million-row table into 50 one-million-row partitions, each with its own efficiently sized HNSW index. For cross-instance distribution, [Jonathan Katz has demonstrated](https://jkatz05.com/post/postgres/distributed-pgvector/) combining partitioning with `postgres_fdw` for federated vector queries across multiple PostgreSQL instances.

## Memory, build times, and the scaling wall

pgvector's practical limits become apparent at scale, and understanding them quantitatively is essential for capacity planning.

**HNSW memory consumption** is the binding constraint for most deployments. The index stores the full graph structure in memory for optimal performance, and this graph is roughly **2–3× the size of the base vector data**. For one million 1536-dimensional vectors stored as single-precision floats, the raw vector data occupies approximately 5.7 GiB (4 bytes × 1536 × 1M + overhead). The HNSW index adds another 7.55 GiB at default m=16, bringing the total to over 13 GiB for vectors alone. A [production scaling report](https://medium.com/@dikhyantkrishnadalai/optimizing-vector-search-at-scale-lessons-from-pgvector-supabase-performance-tuning-ce4ada4ba2ed) documenting 50 million 768-dimensional product embeddings estimates total memory requirements at approximately **852 GiB**: 150 GiB for base vectors, 450 GiB for the HNSW index, 200 GiB for shared_buffers, plus connection pool and OS buffer overhead. While pgvector's README notes that indexes [do not strictly need to fit in memory](https://github.com/pgvector/pgvector), performance degrades substantially when the graph spills to disk.

The `maintenance_work_mem` parameter is particularly critical during index builds. pgvector emits a warning when the HNSW graph exceeds this allocation, and build time increases dramatically beyond that threshold. Users report that [40-million-vector HNSW builds](https://github.com/pgvector/pgvector/issues/822) with insufficient `maintenance_work_mem` can stall at 29% progress for eight or more hours, with estimated total build times exceeding 19 hours. The [pgvector documentation](https://github.com/pgvector/pgvector) recommends setting `maintenance_work_mem` high enough to contain the full graph — for large datasets, this can mean 8 GiB or more — and warns Docker users to ensure `--shm-size` matches or exceeds this value.

**Index build times** have improved dramatically across versions but remain a significant operational consideration. The [Katz benchmarks](https://jkatz05.com/post/postgres/pgvector-performance-150x-speedup/) show HNSW build time for 1M/1536d vectors dropping from **7,479 seconds in v0.5.0 to 250 seconds in v0.7.0** — a 30× improvement attributable primarily to parallel index builds introduced in v0.6.0. Binary quantization further reduces this to **49 seconds**. However, parallel worker scaling has a ceiling: [testing on 104-core machines](https://github.com/pgvector/pgvector/issues/500) showed no improvement beyond approximately 10 parallel workers. At 50 million vectors, even with pgvectorscale's StreamingDiskANN algorithm, [index construction takes roughly 11 hours](https://www.tigerdata.com/blog/pgvector-vs-qdrant) compared to Qdrant's 3.3 hours for the same dataset.

**Concurrent read/write performance** presents another challenge. Inserting rows into a table with an existing HNSW index is [significantly slower](https://github.com/pgvector/pgvector/issues/588) than bulk-loading data and building the index afterward, because each insert must update the graph structure while holding locks. Under heavy write load, this creates contention that slows both reads and writes. The recommended mitigation is `CREATE INDEX CONCURRENTLY`, which avoids blocking writes during index construction but still consumes substantial resources. VACUUM operations on HNSW indexes are [notably slow](https://github.com/pgvector/pgvector), and the official recommendation is to REINDEX first, then VACUUM — an operational quirk that can surprise teams accustomed to standard PostgreSQL maintenance patterns.

The scaling trajectory across dataset sizes follows a predictable pattern. Below **one million vectors**, pgvector performs well with minimal tuning. Between **1–5 million**, HNSW index builds start consuming 10+ GiB of RAM and require careful `maintenance_work_mem` configuration. At **5–10 million**, one production team [reported latency creeping from 50ms to 800ms](https://medium.com/@dikhyantkrishnadalai/optimizing-vector-search-at-scale-lessons-from-pgvector-supabase-performance-tuning-ce4ada4ba2ed) without tuning. At **10 million**, pgvector 0.8.0 on Aurora achieves [reasonable 70ms p99 latency](https://aws.amazon.com/blogs/database/supercharging-vector-search-performance-and-relevance-with-pgvector-0-8-0-on-amazon-aurora-postgresql/) for filtered queries. Beyond **40–50 million**, pure pgvector on a single node becomes increasingly impractical without partitioning or supplementary extensions like pgvectorscale.

## Head-to-head against dedicated vector databases

Timescale's benchmarks — acknowledged as vendor-produced but methodologically documented — provide the most comprehensive cross-database comparisons. On **50 million Cohere 768-dimensional embeddings**, PostgreSQL with pgvector and [pgvectorscale achieved 471 QPS at 99% recall](https://www.tigerdata.com/blog/pgvector-vs-qdrant) versus Qdrant's 41 QPS — an 11.4× throughput advantage attributed to PostgreSQL's mature process-based concurrency model. However, Qdrant delivered better per-query latency: **p95 of 36.73ms versus 60.42ms** for pgvectorscale. Against [Pinecone's storage-optimized tier](https://www.tigerdata.com/blog/pgvector-vs-pinecone) at 99% recall, pgvectorscale showed 28× lower p95 latency and 16× higher throughput at 75% lower cost when self-hosted.

These numbers require context. The pgvectorscale results use Timescale's StreamingDiskANN algorithm with Statistical Binary Quantization, not vanilla pgvector HNSW — a meaningful distinction since pgvectorscale is not available on AWS RDS or most managed PostgreSQL services. The Timescale team [acknowledged difficulty optimizing Qdrant's parameters](https://www.tigerdata.com/blog/pgvector-vs-qdrant) and invited community feedback. Independent validation through ann-benchmarks has been [requested but not yet completed](https://github.com/erikbern/ann-benchmarks/issues/535).

Earlier benchmarks tell a different story about vanilla pgvector. A [June 2023 comparison](https://nirantk.com/writing/pgvector-vs-qdrant/) found pgvector's IVFFlat implementation lagging Qdrant's HNSW by 15× in QPS on 1M/1536d vectors — though this predated pgvector's own HNSW support (added in v0.5.0, August 2023) and is now largely historical.

The competitive landscape among purpose-built databases adds further nuance. **Milvus** offers GPU acceleration and distributed architecture capable of handling billions of vectors with 11 index types. **Weaviate** provides built-in vectorization modules and GraphQL APIs for multimodal search. **Qdrant**, written in Rust, excels at [single-query latency and complex metadata filtering](https://qdrant.tech/benchmarks/). Each solves problems that pgvector does not attempt to address: native horizontal sharding, auto-scaling, and purpose-built query optimizers for vector-only workloads.

## Distance functions and type system

pgvector supports **six distance metrics** through dedicated operators. Euclidean distance (`<->`) and cosine distance (`<=>`) handle the most common embedding comparison tasks. Negative inner product (`<#>`) serves normalized vectors where inner product equals cosine similarity. L1/Manhattan distance (`<+>`), Hamming distance (`<~>`), and Jaccard distance (`<%>`) were [added in v0.7.0](https://github.com/pgvector/pgvector). Hamming and Jaccard operate exclusively on `bit` vectors, making them natural fits for binary quantization workflows.

The type system comprises four vector types. Standard `vector` stores single-precision floats with up to **16,000 storage dimensions and 2,000 indexable dimensions**. `halfvec` halves storage requirements at `2 * dimensions + 8` bytes with up to **4,000 indexable dimensions**. The `bit` type enables binary vectors indexable up to **64,000 dimensions**. `sparsevec`, storing only non-zero elements at `8 * nnz + 16` bytes, supports HNSW indexing with up to **1,000 non-zero elements** but [does not support IVFFlat](https://github.com/pgvector/pgvector). Expression indexes enable mixing types: a full-precision `vector` column can have a `halfvec` or `bit` expression index for reduced memory consumption, with optional reranking against original vectors for higher recall.

## Managed service landscape and deployment patterns

Every major managed PostgreSQL provider now supports pgvector. **Supabase** [pre-installs it](https://supabase.com/docs/guides/ai/vector-columns) with a purpose-built AI toolkit including Python client libraries and Edge Function integrations for embedding generation. **Neon** offers [serverless PostgreSQL with pgvector](https://neon.com/docs/extensions/pgvector), featuring scale-to-zero for bursty workloads and instant database branching for testing index parameter changes. **AWS Aurora** supports pgvector 0.8.0 with [documented benchmarks](https://aws.amazon.com/blogs/database/supercharging-vector-search-performance-and-relevance-with-pgvector-0-8-0-on-amazon-aurora-postgresql/) on Graviton4 instances. Azure Database for PostgreSQL includes pgvector with [detailed optimization guides](https://learn.microsoft.com/en-us/azure/postgresql/flexible-server/how-to-optimize-performance-pgvector) covering Citus-based sharding strategies.

Production deployment patterns that consistently appear across practitioner accounts include several key practices. Build indexes after bulk loading data, not during inserts. Use `CREATE INDEX CONCURRENTLY` in production to avoid write blocking. Set `maintenance_work_mem` high enough to contain the full HNSW graph. Partition large tables by tenant, category, or time window, with per-partition HNSW indexes. Reduce dimensionality where possible — 768-dimensional models often suffice where 1536 was used — and use `halfvec` for a further 50% storage reduction. Monitor with `pg_stat_statements` and alert on index size growth and p99 latency drift. Use `pg_prewarm` to warm HNSW index pages after failovers or deployments.

## When pgvector is — and is not — the right choice

The decision framework distills to a handful of discriminating factors. pgvector is the clear choice when you already run PostgreSQL, need ACID consistency between vectors and relational data, require SQL JOINs across vector and non-vector tables, manage fewer than **10 million vectors** (or up to 50 million with pgvectorscale and partitioning), and want to minimize operational surface area. Organizations report [40–60% lower total cost of ownership](https://airbyte.com/data-engineering-resources/postgresql-as-a-vector-database) compared to dedicated vector databases for qualifying workloads. [Discourse uses pgvector](https://news.ycombinator.com/item?id=45798479) across thousands of databases serving billions of page views. Supabase customers [store over 1.6 million embeddings](https://supabase.com/modules/vector) with reported satisfaction.

A purpose-built vector database becomes the stronger option when vector search is the primary workload rather than a feature alongside relational data, when datasets exceed 50 million vectors and require distributed multi-node search, when auto-scaling for unpredictable traffic is essential, or when sub-millisecond latencies at extreme query rates are non-negotiable. At these scales, Milvus's GPU acceleration, Qdrant's latency profile, or Pinecone's fully managed infrastructure solve problems that PostgreSQL's architecture was not designed to address.

The gap between these camps is narrowing. pgvector's version-over-version trajectory — from 8 QPS with IVFFlat in v0.4.1 to 253 QPS with HNSW in v0.7.0 to iterative scans achieving 100% filtered recall in v0.8.0, all on the same [1M/1536d benchmark](https://jkatz05.com/post/postgres/pgvector-performance-150x-speedup/) — suggests that the PostgreSQL ecosystem's investment in vector search is accelerating. For the majority of AI applications in the early-to-mid scale range, the database you already have may well be the database you need.

## Conclusion

pgvector has crossed the threshold from "good enough for prototypes" to "production-viable for most workloads." Its HNSW implementation delivers **99% recall at 253 QPS and 5.5ms p99** on the standard million-vector OpenAI benchmark, with quantization options that halve index size while improving throughput. Version 0.8.0's iterative scans solved the filtered-query recall problem that was arguably pgvector's most damaging limitation. The integration advantages — ACID guarantees, SQL composability, operational unity with existing PostgreSQL infrastructure — represent genuine architectural value that no purpose-built vector database replicates.

The honest constraints are equally important to internalize. HNSW indexes consume 2–3× the base vector storage in memory. Build times at 40M+ vectors stretch into hours and require careful resource provisioning. Write-heavy workloads with existing HNSW indexes face lock contention. Beyond 10 million vectors on a single node, pgvector demands partitioning, pgvectorscale, or both — and at 100 million vectors, purpose-built solutions offer a fundamentally different scaling architecture. The engineer who chooses pgvector should do so with eyes open on these constraints, a partitioning strategy in hand, and a clear threshold at which migration to a dedicated vector database becomes warranted.
