# Vector index algorithms: HNSW, DiskANN, and IVF-PQ compared

**Three algorithms dominate modern vector search — and choosing between them reduces to a single question: where does your data live?** HNSW delivers the lowest latency when your dataset fits in RAM. DiskANN extends graph-based search to billion-scale collections on SSDs with surprisingly small memory overhead. IVF-PQ compresses vectors aggressively enough to keep datasets in memory that would otherwise require disk, at the cost of recall. Each algorithm occupies a distinct point in the design space defined by memory budget, acceptable latency, and dataset scale. This report examines their internal mechanics, parameter tradeoffs, and the workload conditions that favor each one.

## HNSW builds a probabilistic skip list over proximity graphs

Hierarchical Navigable Small World graphs, introduced by [Malkov and Yashunin in 2018](https://arxiv.org/abs/1603.09320), construct a multi-layer structure where each layer is a proximity graph over a nested subset of the dataset. **Layer 0 contains every vector** with dense short-range connections, while higher layers contain exponentially fewer elements with sparser, longer-range links. The structure mirrors a [probabilistic skip list](https://arxiv.org/pdf/1603.09320) — but instead of sorted linked lists, each layer is a navigable small-world graph.

The maximum layer for each inserted element is drawn from an exponentially decaying distribution: `l = ⌊−ln(unif(0,1)) · mL⌋`, where the optimal normalization factor `mL = 1/ln(M)` produces a [logarithmic expected number of layers](https://arxiv.org/pdf/1603.09320). The parameter M controls maximum outgoing connections per node — at layer 0, each node maintains up to `2M` connections, while higher layers cap at `M`. This asymmetry gives the ground layer enough connectivity to sustain high recall during the fine-grained search phase.

Insertion follows a two-phase procedure described in [Algorithm 1 of the paper](https://arxiv.org/pdf/1603.09320). Phase one performs greedy search with `ef=1` from the top layer down to the new element's assigned layer, identifying the best entry point. Phase two, operating from the assigned layer down to layer 0, runs a broader search using `ef_construction` candidates to find neighbors, then connects them bidirectionally. If any neighbor's degree exceeds the maximum, HNSW applies a pruning heuristic.

The heuristic neighbor selection ([Algorithm 4](https://arxiv.org/pdf/1603.09320)) is what separates HNSW from naive k-NN graph construction. Rather than simply keeping the M closest points, it evaluates whether each candidate provides *directional diversity* — a candidate is retained only if it is closer to the base element than to any already-selected neighbor. This prevents clustering of connections and preserves global navigability, particularly in datasets with non-uniform density.

Search ([Algorithm 5](https://arxiv.org/pdf/1603.09320)) mirrors the insertion path: greedy descent with `ef=1` through upper layers, then a beam search with configurable `ef_search` at layer 0. The `SEARCH-LAYER` subroutine maintains two priority queues — one for candidates to evaluate, one for the current best results — expanding the nearest unevaluated candidate at each step until all candidates are exhausted or the nearest candidate is farther than the worst result. Search complexity is [O(log N)](https://arxiv.org/pdf/1603.09320), a meaningful improvement over the polylogarithmic O(log²N) complexity of flat NSW graphs.

### The three parameters that control everything

**M** (max connections per node) directly determines both recall quality and memory consumption. The [hnswlib documentation](https://github.com/nmslib/hnswlib/blob/master/ALGO_PARAMS.md) recommends M=12–48 for most workloads, with M=6 for low-dimensional data and M=48–64 for high-dimensional embeddings like face descriptors. Memory overhead from the graph alone scales as roughly **M × 8–10 bytes per element**, independent of vector dimensionality. [Qdrant defaults to M=16](https://qdrant.tech/documentation/concepts/indexing/), and their [tuning guide](https://qdrant.tech/course/essentials/day-2/what-is-hnsw/) suggests `m=8, ef_construct=100` for memory-constrained setups, `m=16, ef_construct=200` as balanced, and `m=32, ef_construct=400` for maximum quality.

**ef_construction** sets the beam width during index building. Higher values produce better-connected graphs — and therefore higher recall at query time — at the cost of longer build times. It [does not affect memory usage](https://github.com/nmslib/hnswlib/blob/master/ALGO_PARAMS.md). Typical ranges span 100–500. A useful diagnostic: if recall for M-nearest-neighbor search with `ef = ef_construction` falls below 0.9, there is [room to increase ef_construction](https://github.com/nmslib/hnswlib/blob/master/ALGO_PARAMS.md).

**ef_search** controls the speed-recall tradeoff at query time and is the only parameter adjustable after index construction. It must be [≥ K](https://github.com/nmslib/hnswlib/blob/master/ALGO_PARAMS.md) (the number of requested neighbors). On SIFT1M, [hnswlib benchmarks](https://issues.apache.org/jira/browse/LUCENE-9937) show ef_search=50 achieving 0.950 recall@10 at 28,022 QPS, while ef_search=100 reaches 0.985 recall at 16,109 QPS — a clean illustration of the diminishing-returns curve.

### Memory is HNSW's binding constraint

Total memory per vector is `4 × d` bytes for the float32 vector plus the graph overhead. The [OpenSearch team](https://aws.amazon.com/blogs/big-data/choose-the-k-nn-algorithm-for-your-billion-scale-use-case-with-opensearch/) models total memory as approximately `1.1 × (4 × d + 8 × M) × N` bytes. For 1 million 128-dimensional vectors with M=32, [FAISS benchmarks report](https://github.com/facebookresearch/faiss/wiki/Indexing-1M-vectors) 512 MB for vectors plus 796 MB for the HNSW graph — **roughly 1.3 GB total**. For 1 million 768-dimensional embedding vectors (typical of modern transformers), the vector storage alone consumes ~3 GB. Scaling to 1 billion 768-dimensional vectors would require on the order of 500 GB of RAM, which makes HNSW impractical at that scale without quantization.

[Qdrant addresses this](https://qdrant.tech/documentation/guides/optimize/) through several strategies: on-disk HNSW via memory-mapped files, scalar and product quantization that compress vectors before storing them in the graph, and a filterable HNSW variant that augments the graph with [payload-based edges](https://qdrant.tech/documentation/concepts/indexing/) so that filtered vector search doesn't degrade into disconnected subgraph traversals.

## DiskANN's Vamana graph trades hierarchy for disk-friendliness

Where HNSW uses a layered hierarchy to provide long-range shortcuts, DiskANN's Vamana algorithm ([Subramanya et al., NeurIPS 2019](https://suhasjs.github.io/files/diskann_neurips19.pdf)) achieves the same goal with a **single flat directed graph** and a tunable distance-scaling parameter α. The elimination of layers is not merely aesthetic — it is architecturally critical for SSD-based search, where predictable I/O access patterns determine throughput.

Vamana construction begins by initializing a [random R-regular directed graph](https://suhasjs.github.io/files/diskann_neurips19.pdf) and computing the dataset's medoid as a fixed entry point. The algorithm then makes **two passes** over the data. In the first pass (α=1), for each point p, it runs `GreedySearch` from the medoid to find visited nodes, then applies `RobustPrune` to select p's out-neighbors — keeping only neighbors that are not "shadowed" by a closer already-selected neighbor. The second pass uses α > 1 (typically 1.2), which relaxes the shadowing criterion: a candidate is pruned only if `α · d(p*, p') ≤ d(p, p')`, where p* is the shadowing neighbor. This [introduces long-range edges](https://suhasjs.github.io/files/diskann_neurips19.pdf) that reduce graph diameter to approximately log_α(n) hops — achieving what HNSW accomplishes through its upper layers, but within a single graph level.

The critical algorithmic difference is in how candidate sets are constructed. HNSW only considers the final ef-nearest candidates from its layer search when selecting neighbors. Vamana uses the [entire set of nodes visited](https://suhasjs.github.io/files/diskann_neurips19.pdf) during greedy search, which naturally includes both nearby and distant points encountered along the search path. This gives `RobustPrune` a richer pool from which to select diverse long-range connections. The paper demonstrates that Vamana achieves [2–3× fewer hops than HNSW and NSG](https://suhasjs.github.io/files/diskann_neurips19.pdf) for search convergence on large datasets.

### The hybrid RAM-SSD architecture

DiskANN's design rests on a simple but effective partitioning of data between memory and storage. In RAM, it keeps only [PQ-compressed representations](https://suhasjs.github.io/files/diskann_neurips19.pdf) of each vector (typically 32 bytes per 128-dimensional vector) plus cached neighborhoods of the first 3–4 BFS hops from the medoid. On SSD, it stores full-precision vectors co-located with their neighbor lists, laid out so that each node's data occupies a [contiguous disk sector](https://suhasjs.github.io/files/diskann_neurips19.pdf). Since reading a 4KB sector costs the same as reading 512 bytes on modern SSDs, this co-location means each disk access retrieves both the full vector and its neighbors — enabling exact distance re-ranking at zero extra I/O cost.

The search procedure, `BeamSearch`, uses a [beam width W](https://suhasjs.github.io/files/diskann_neurips19.pdf) (typically 2–8) to issue multiple SSD reads in parallel at each step. The PQ-compressed vectors in RAM guide the search direction through approximate distance calculations, while the full vectors piggy-backing on neighborhood reads provide exact distances for re-ranking. The paper reports that with W=2–8, each search thread spends [40–50% of time in I/O](https://suhasjs.github.io/files/diskann_neurips19.pdf) and keeps SSD load at 30–40%.

### Billion-scale results on commodity hardware

On SIFT1B (1 billion 128-dimensional vectors), DiskANN achieves [>5,000 QPS with <3ms mean latency and 95%+ 1-recall@1](https://www.microsoft.com/en-us/research/publication/diskann-fast-accurate-billion-point-nearest-neighbor-search-on-a-single-node/) running on a 16-core workstation with 64GB DDR4 and two Samsung 960 EVO SSDs in RAID-0. For comparison, the best compression-based baseline (FAISS IVFOADC+G+P) [plateaus at 62.74% 1-recall@1](https://suhasjs.github.io/files/diskann_neurips19.pdf) with comparable memory budget. DiskANN can index and serve [5–10× more points per node](https://www.microsoft.com/en-us/research/publication/diskann-fast-accurate-billion-point-nearest-neighbor-search-on-a-single-node/) than in-memory graph methods in the high-recall regime.

The memory savings are dramatic. Where HNSW would require roughly **500+ GB of RAM** for a billion 768-dimensional vectors, DiskANN needs approximately [32 GB for PQ-compressed vectors](https://suhasjs.github.io/files/diskann_neurips19.pdf) plus SSD storage for the full index — a **15–50× reduction in RAM** depending on dimensionality. Microsoft has deployed DiskANN in production in [Azure Cosmos DB](https://cazton.com/blogs/technical/hnsw-vs-diskann) and [SQL Server 2025](https://www.mytechmantra.com/sql-server/sql-server-2025-diskann-vector-indexing-guide/). Milvus also [supports DiskANN as an index type](https://milvus.io/docs/diskann.md).

The primary limitations are **construction cost and mutability**. Building the one-shot SIFT1B index takes [~2 days on a machine with 1,792 GB RAM](https://suhasjs.github.io/files/diskann_neurips19.pdf). The merge-based approach reduces RAM to 64GB but extends build time to [~5 days](https://suhasjs.github.io/files/diskann_neurips19.pdf). The original DiskANN index is immutable once built; [FreshDiskANN](https://arxiv.org/abs/2105.09613) adds incremental update support with roughly 30% latency overhead. This makes DiskANN best suited for static or batch-updated workloads, while HNSW's incremental insertion handles dynamic datasets more naturally.

## IVF-PQ compresses vectors into bytes at the cost of recall

Inverted File with Product Quantization combines two independent techniques. IVF partitions the vector space into Voronoi cells via k-means clustering, creating `nlist` inverted lists that allow search to [examine only the nprobe nearest partitions](https://github.com/facebookresearch/faiss/wiki/Faiss-indexes) rather than the full dataset. Product Quantization, introduced by [Jégou, Douze, and Schmid (2011)](https://inria.hal.science/inria-00514462v2/document), decomposes each d-dimensional vector into M subvectors of dimension d/M, independently quantizing each subspace against a codebook of 2^nbits centroids.

When combined as [IVFADC](https://inria.hal.science/inria-00514462v2/document), the system first assigns each vector to its nearest IVF centroid, then encodes the **residual** (the difference between the vector and its centroid) using PQ rather than encoding the original vector. Residuals have lower norms than the original vectors, which [improves quantization accuracy](https://developer.nvidia.com/blog/accelerating-vector-search-nvidia-cuvs-ivf-pq-deep-dive-part-1/). At search time, the query is compared against centroids to select the `nprobe` nearest lists, then PQ distance tables are used for fast approximate distance computation within each list — requiring only [M table lookups and a sum per candidate](https://github.com/facebookresearch/faiss/wiki/Faiss-indexes) rather than a full d-dimensional distance computation.

### Memory savings are the primary advantage

The compression arithmetic is straightforward. With M=8 subquantizers and nbits=8, each vector is encoded in `ceil(8 × 8 / 8) = 8 bytes`, plus 8 bytes for the vector ID in each inverted list — **16 bytes total per vector**. A 128-dimensional float32 vector occupies 512 bytes in a flat index, making this a [32× compression ratio](https://github.com/facebookresearch/faiss/wiki/Faiss-indexes). On SIFT1M, FAISS's IndexIVFPQ with M=8 uses [~9.2 MB](https://www.pinecone.io/learn/series/faiss/product-quantization/) compared to 512 MB for the flat index — a **96% reduction**.

The [FAISS guidelines](https://github.com/facebookresearch/faiss/wiki/Guidelines-to-choose-an-index) provide concrete memory formulas across index types. HNSW with M=32 on 128-dimensional data requires approximately **768 bytes per vector** (512 for the vector, 256 for graph links). IVF-PQ with M=8 requires **16 bytes per vector**. For a billion-vector dataset, that translates to ~715 GB for HNSW versus ~15 GB for IVF-PQ — more than enough to determine infrastructure choices.

### The recall penalty is real but manageable

Pure IVF-PQ recall on SIFT1M with M=8 and default nprobe=1 reaches only [~34% recall@10](https://www.pinecone.io/learn/series/faiss/product-quantization/). Increasing nprobe improves recall at the cost of scanning more inverted lists, but even with nprobe=nlist (exhaustive scan of all lists), recall saturates at about [50% for that compression level](https://www.pinecone.io/learn/series/faiss/product-quantization/) because PQ's lossy compression introduces irreducible approximation error. The [FAISS documentation is explicit](https://www.pinecone.io/learn/series/faiss/product-quantization/): very high recall is out of reach for PQ indexes alone.

Several techniques close the gap. **OPQ** (Optimized Product Quantization) applies a [learned rotation](https://github.com/facebookresearch/faiss/wiki/Guidelines-to-choose-an-index) before quantization that better distributes variance across subspaces, yielding meaningful recall improvements — particularly on data with uneven dimensional distributions. **Re-ranking** (or refining) recomputes exact distances for the top candidates using stored raw vectors, recovering much of the recall lost to quantization. The FAISS [4-bit PQ FastScan with re-ranking](https://github.com/facebookresearch/faiss/wiki/Fast-accumulation-of-PQ-and-AQ-codes-(FastScan)) achieves **280,000 QPS at 0.9 1-recall@1** on SIFT1M — **2× faster than HNSW while using 2.7× less memory**.

### FAISS optimizations make IVF-PQ production-ready

FAISS implements several layers of optimization that transform IVF-PQ from an academic technique into a practical system. The [FastScan variant](https://github.com/facebookresearch/faiss/wiki/Fast-accumulation-of-PQ-and-AQ-codes-(FastScan)) uses 4-bit quantization (nbits=4) with SIMD instructions (AVX2/ARM Neon) to pack lookup tables into CPU registers rather than cache, processing 32 vectors simultaneously per SIMD operation. Lookup table entries are quantized to 8-bit unsigned integers and accumulated in 16-bit unsigned integers, enabling throughputs exceeding [1 million QPS](https://github.com/facebookresearch/faiss/wiki/Fast-accumulation-of-PQ-and-AQ-codes-(FastScan)) without re-ranking.

IVF-PQ is [fully supported on GPU](https://github.com/facebookresearch/faiss/wiki/Guidelines-to-choose-an-index) — unlike HNSW, which has no GPU implementation in FAISS. The NVIDIA cuVS library provides a [highly optimized GPU IVF-PQ](https://developer.nvidia.com/blog/accelerating-vector-search-nvidia-cuvs-ivf-pq-deep-dive-part-1/) integrated into FAISS 1.8+ and Milvus 2.3+. For billion-scale datasets, FAISS also supports [on-disk inverted lists](https://github.com/facebookresearch/faiss/wiki/Indexes-that-do-not-fit-in-RAM), enabling IVF-PQ indexes that exceed RAM without requiring DiskANN's specialized graph structure.

Recommended [FAISS factory strings](https://github.com/facebookresearch/faiss/wiki/Guidelines-to-choose-an-index) for common scales: `"IVF4096,PQ32"` for 1M vectors, `"IVF65536_HNSW32,PQ32"` for 1M–10M (using HNSW as the coarse quantizer for faster centroid assignment), and `"OPQ32_128,IVF262144_HNSW32,PQ32"` for 10M–100M with OPQ preprocessing.

## How these algorithms compare across workloads

The three algorithms occupy distinct regions in a three-dimensional tradeoff space defined by memory efficiency, query latency, and recall quality. The following analysis draws on [FAISS benchmarks](https://github.com/facebookresearch/faiss/wiki/Indexing-1M-vectors), [Milvus documentation](https://milvus.io/docs/index-explained.md), and the original [DiskANN paper](https://suhasjs.github.io/files/diskann_neurips19.pdf).

At the million-vector scale, HNSW dominates. On SIFT1M, [FAISS benchmarks show](https://github.com/facebookresearch/faiss/wiki/Indexing-1M-vectors) HNSW reaching 0.9492 1-recall@1 in 0.020ms per query, while IVFFlat requires 0.141ms for 0.9470 recall — HNSW is **7× faster at equivalent recall**. The memory cost is real (1.3 GB vs. ~16 MB for IVF-PQ on SIFT1M) but manageable on modern hardware.

At 100 million vectors, the calculus shifts. Practical benchmarks suggest that [HNSW with M=48 and ef_search=96 delivers p95 latency of 12ms at 0.94 recall@50, consuming ~55 GB of RAM](https://medium.com/@2nick2patel2/vector-db-showdown-hnsw-diskann-recall-8b908960f7b4). DiskANN with 8-byte PQ codes and beam width 8 achieves [p95 of 15ms at 0.95 recall@50 using 16 GB RAM plus NVMe storage](https://medium.com/@2nick2patel2/vector-db-showdown-hnsw-diskann-recall-8b908960f7b4) — comparable recall with a fraction of the RAM, at a modest latency premium.

At billion scale, DiskANN stands alone among graph methods. Where HNSW would require [hundreds of gigabytes of RAM](https://github.com/facebookresearch/faiss/wiki/Indexing-1M-vectors), DiskANN serves SIFT1B with [>5,000 QPS at 95%+ recall using 64 GB RAM](https://www.microsoft.com/en-us/research/publication/diskann-fast-accurate-billion-point-nearest-neighbor-search-on-a-single-node/). IVF-PQ can also scale to billions with on-disk inverted lists, but [FAISS's IVFOADC baseline plateaus at ~63% recall](https://suhasjs.github.io/files/diskann_neurips19.pdf) at the same memory budget — highlighting the fundamental recall advantage of graph-based search over cell-probe methods at extreme scale.

The [Milvus decision matrix](https://milvus.io/docs/index-explained.md) codifies these tradeoffs clearly: use HNSW when raw data fits in memory, DiskANN when data lives on SSD but a quarter of it fits in RAM as compressed vectors, and IVF-PQ when both RAM and SSD are constrained but some recall degradation is acceptable. For workloads with high filter ratios (>85%), [IVF variants maintain stable performance](https://milvus.io/blog/understanding-ivf-vector-index-how-It-works-and-when-to-choose-it-over-hnsw.md) because filtering happens at the centroid level, while HNSW graphs can fragment under restrictive filters — though [Qdrant's filterable HNSW](https://qdrant.tech/documentation/concepts/indexing/) with payload-based edges partially addresses this.

### Hybrid architectures blur the boundaries

Production systems increasingly combine these techniques. **HNSW-PQ** stores PQ-compressed vectors in an HNSW graph, reducing memory for 1M 128-dimensional vectors from [640 MB to ~136 MB](https://milvus.io/docs/index-explained.md) while retaining graph-based search quality. FAISS supports [using HNSW as the coarse quantizer](https://github.com/facebookresearch/faiss/wiki/Indexing-1M-vectors) for IVF, combining HNSW's fast centroid lookup with IVF-PQ's compression — the factory string `"IVF65536_HNSW32,PQ32"` creates this hybrid. LanceDB takes a different approach, [building sub-HNSW indices within each IVF partition](https://medium.com/@amineka9/the-future-of-vector-search-exploring-lancedb-for-billion-scale-vector-search-0664801bc915) rather than constructing a single HNSW graph over the full dataset. DiskANN itself is inherently hybrid — Vamana graph plus PQ in RAM plus full vectors on SSD.

## Conclusion

The choice between these three algorithms reduces to infrastructure constraints rather than algorithmic superiority. **HNSW remains the default for datasets under ~100 million vectors** when RAM is available — its O(log N) search, incremental insertability, and consistently Pareto-optimal recall-versus-throughput performance across [ann-benchmarks](http://ann-benchmarks.com/) make it the lowest-risk choice. Start with M=16 and ef_construction=200; tune ef_search at query time.

**DiskANN is the correct choice when your dataset exceeds available RAM** and you need >90% recall. Its Vamana graph achieves the shortest search paths of any ANN graph algorithm by using a tunable α parameter to control long-range connectivity, and its SSD-optimized layout converts the cost of disk access into a throughput problem rather than a latency wall. The immutability constraint is real but addressed by FreshDiskANN for workloads requiring updates.

**IVF-PQ earns its place through compression** — when you need to fit a billion vectors into 15 GB, no graph method can compete. The recall penalty relative to HNSW is significant without re-ranking, but FAISS's FastScan with re-ranking closes the gap while maintaining a 2.7× memory advantage. GPU support makes IVF-PQ uniquely suited to GPU-accelerated inference pipelines.

The most sophisticated production deployments no longer pick one algorithm exclusively. They compose them — HNSW as a quantizer, PQ as the storage layer, re-ranking for precision recovery — engineering the recall/latency/memory Pareto frontier to match their specific workload.
