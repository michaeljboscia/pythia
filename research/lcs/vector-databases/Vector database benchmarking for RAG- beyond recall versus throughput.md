# Vector database benchmarking for RAG: beyond recall versus throughput

**Standard ANN benchmarks systematically miss the workload characteristics that define production RAG systems.** The widely cited [ann-benchmarks](https://github.com/erikbern/ann-benchmarks) framework evaluates pure vector similarity on static, in-memory datasets using a single CPU thread — a scenario that bears little resemblance to a RAG pipeline performing filtered metadata queries, ingesting new chunks, and serving concurrent users. Building a reliable local-first RAG system on Qdrant, LanceDB, or pgvector demands a benchmarking protocol that measures at least six additional dimensions: filtered search across varying selectivities, write throughput during indexing, query stability under concurrent ingestion, tail latency percentiles, memory consumption at steady state, and recovery time after crashes. This report examines ann-benchmarks' methodology and its blind spots, catalogs the benchmarking dimensions that RAG workloads actually require, and proposes a concrete protocol for a 500K-chunk local comparison of the three databases.

## How ann-benchmarks structures its evaluations and where they break down

The [ANN-Benchmarks framework](https://arxiv.org/abs/1807.05614), introduced by Aumüller, Bernhardsson, and Faithfull, has become the default reference for comparing approximate nearest-neighbor algorithms. It ships with **15 pre-computed HDF5 datasets** ranging from Fashion-MNIST (60K vectors, 784 dimensions) to DEEP1B (roughly 10M vectors, 96 dimensions), with ground-truth top-100 neighbors computed via brute force. Distance metrics span Euclidean, angular/cosine, Jaccard, and Hamming. Each algorithm is wrapped in a Python interface and executed inside an isolated Docker container on standardized hardware — currently an [AWS r6i.16xlarge with hyperthreading disabled](https://github.com/erikbern/ann-benchmarks), enforcing single-CPU execution via Linux cpusets.

The parameter-sweep mechanism uses YAML configuration files that define build-time arguments (controlling index structure) and query-time arguments (controlling search effort) as Cartesian products. A single index build is reused across all query-parameter combinations, and the framework plots the [Pareto frontier](https://arxiv.org/pdf/1807.05614) of recall-versus-QPS across all configurations for each algorithm. Build time and index size are recorded but receive far less attention than the headline recall-QPS curves.

For RAG workloads, this methodology has at least seven critical gaps. First, **no filtered or hybrid search**: every query is pure vector similarity with zero metadata predicates. A [2025 ETH Zurich study](http://htor.inf.ethz.ch/publications/img/2025_iff_fanns_benchmark.pdf) on filtered ANN explicitly notes the absence of any comprehensive filtered-ANN benchmark, and a separate [2025 paper extending ann-benchmarks](https://arxiv.org/html/2602.11443v1) had to add filtered search support from scratch. Second, **no concurrency testing** — the single-thread constraint means the framework cannot reveal how algorithms behave under the parallel query load of a multi-user RAG application. Third, datasets are **static and fully indexed** before any query runs, so there is no measurement of insert-while-querying behavior. Fourth, dataset dimensionality tops out at 960 for dense vectors (GIST), while modern embedding models from OpenAI and Cohere produce [768 to 3,072 dimensions](https://milvus.io/blog/benchmarks-lie-vector-dbs-deserve-a-real-test.md). Fifth, all datasets fit in RAM; there is no disk-based or memory-constrained testing. Sixth, only average QPS is reported — no P95 or P99 tail latency, which is the metric that actually governs SLA compliance. Seventh, as [Weaviate's benchmark documentation](https://docs.weaviate.io/weaviate/benchmarks/ann) observes, ann-benchmarks returns only matched IDs, whereas a real database must retrieve full objects from storage, adding I/O overhead that the framework never measures.

## The six benchmarking dimensions RAG workloads actually need

Production RAG systems stress vector databases in ways that pure ANN recall curves cannot predict. The following dimensions, drawn from the methodologies of [Qdrant's benchmark suite](https://qdrant.tech/benchmarks/), [VDBBench 1.0 by Zilliz](https://zilliz.com/blog/vdbbench-1-0-benchmarking-with-your-real-world-production-workloads), and [Weaviate's server-level benchmarks](https://weaviate.io/developers/weaviate/benchmarks/ann), capture the missing surface area.

**Filtered search across selectivity levels.** RAG queries routinely include metadata predicates — date ranges, document types, tenant isolation, access controls. Filter selectivity (the fraction of vectors that pass the predicate) has an outsized impact on performance. [VDBBench tests selectivities from 50% down to 99.9%](https://milvus.io/blog/benchmarks-lie-vector-dbs-deserve-a-real-test.md) and reports that query speeds can fluctuate by orders of magnitude as selectivity changes. Qdrant publishes a [dedicated filtered-search benchmark](https://qdrant.tech/benchmarks/filtered-search-intro/) using datasets enriched with keyword and geospatial payloads, testing their "filterable HNSW" approach that avoids the accuracy collapse of naive post-filtering when most vectors are excluded. pgvector, starting with [version 0.8.0's iterative index scans](https://aws.amazon.com/blogs/database/supercharging-vector-search-performance-and-relevance-with-pgvector-0-8-0-on-amazon-aurora-postgresql/), achieves up to **5.7× improvement** in filtered query performance by automatically retrieving additional HNSW candidates when WHERE clauses are selective.

**Write performance and index build cost.** The ann-benchmarks paper itself acknowledges that [graph-based methods like HNSW can take hours to build](https://arxiv.org/pdf/1807.05614), making them poor choices for frequently changing datasets. In practice, pgvector's HNSW build on a [Cohere 10M-row dataset improved 30× in version 0.7.0](https://aws.amazon.com/blogs/database/load-vector-embeddings-up-to-67x-faster-with-pgvector-and-amazon-aurora/) with parallel workers, while [LanceDB builds an IVF-PQ index over 1M 960-dimensional vectors in roughly one minute on a MacBook M2 Max](https://medium.com/etoai/benchmarking-lancedb-92b01032874a). These are order-of-magnitude differences that ann-benchmarks' build-time metric captures in principle but that the community systematically ignores in favor of QPS headlines.

**Concurrent read/write (streaming) behavior.** VDBBench is currently the [only major benchmark tool](https://zilliz.com/blog/vdbbench-1-0-benchmarking-with-your-real-world-production-workloads) that systematically measures search performance during active ingestion. Its streaming test inserts at a controlled rate (e.g., 500 rows/second across five parallel producers) and triggers search measurements after every 10% of data ingested. The results are revealing: on Cohere 10M, Elasticsearch's QPS [drops 84% under write load](https://zilliz.com/vdbbench-leaderboard?dataset=vectorSearch) (376 → 62 QPS), while Qdrant drops only 22% (447 → 348 QPS). For a RAG system processing continuous document ingestion, this difference dominates user experience.

**Tail latency under concurrency.** Both [Qdrant](https://qdrant.tech/benchmarks/) and [VDBBench](https://milvus.io/blog/benchmarks-lie-vector-dbs-deserve-a-real-test.md) measure P95 and P99 latency under multi-threaded load — Qdrant tests with 100 parallel readers, VDBBench gradually increases concurrency to find sustainable maximum throughput. Average latency, as Zilliz notes, "masks the outliers that frustrate real users." A [50-million-vector comparison](https://www.tigerdata.com/blog/pgvector-vs-qdrant) of pgvector and Qdrant found that Qdrant exhibits **smaller variance between percentiles**, meaning more consistent tail latency even though pgvector achieved higher raw throughput (471 vs. 41 QPS at 99% recall in that particular configuration).

**Memory consumption under load.** Qdrant's [memory consumption study](https://qdrant.tech/articles/memory-consumption/) demonstrates that serving 1M 128-dimensional vectors can require as little as **135 MB with full mmap** versus 1.2 GB fully in RAM, at the cost of reduced throughput. pgvector's HNSW index overhead runs [roughly 2–3× the base vector size](https://medium.com/@dikhyantkrishnadalai/optimizing-vector-search-at-scale-lessons-from-pgvector-supabase-performance-tuning-ce4ada4ba2ed), making shared_buffers sizing critical — performance degrades sharply when the index spills from memory. LanceDB's disk-native Lance columnar format provides inherently [low memory requirements](https://docs.lancedb.com/faq/faq-oss) by keeping the index on SSD, trading some latency for dramatically reduced RAM pressure.

**Crash recovery.** Qdrant uses [write-ahead logging](https://qdrant.tech/articles/qdrant-1.2.x/) and, since version 1.2, enters a recovery mode that prevents crash loops from OOM conditions during heavy ingestion. pgvector inherits PostgreSQL's WAL and MVCC durability guarantees. LanceDB's append-based Lance format provides transactional semantics. None of the major public benchmarks measure time-to-recover-and-serve after a hard crash — an oversight that matters enormously for local-first deployments where the process may be killed by the OS under memory pressure.

## A practical protocol for comparing Qdrant, LanceDB, and pgvector at 500K chunks

The following protocol is designed for a senior engineer evaluating these three databases for a local-first RAG system on consumer hardware (16–32 GB RAM, NVMe SSD). It synthesizes methodological best practices from [Qdrant's open-source benchmark framework](https://github.com/qdrant/vector-db-benchmark), [VDBBench's streaming test design](https://github.com/zilliztech/VectorDBBench), and [pgvector tuning guides from Jonathan Katz](https://jkatz05.com/post/postgres/pgvector-hnsw-performance/) and [AWS](https://aws.amazon.com/blogs/database/accelerate-hnsw-indexing-and-searching-with-pgvector-on-amazon-aurora-postgresql-compatible-edition-and-amazon-rds-for-postgresql/).

**Dataset construction.** Embed your actual corpus (or a representative sample) using your production embedding model. If you need a proxy, the [dbpedia-openai-1M dataset](https://qdrant.tech/blog/qdrant-benchmarks-2024/) (1,536 dimensions, OpenAI embeddings) is the most RAG-representative public benchmark dataset; subsample 500K vectors. Attach synthetic metadata columns with controlled selectivity: a categorical field with 10 values (10% selectivity per value), a timestamp field spanning 365 days, and a boolean access-control field (50% selectivity). Precompute brute-force ground truth for top-20 neighbors both unfiltered and under each filter condition.

**Database configuration.** Run all three on identical hardware, one at a time, with caches cleared between runs. For pgvector (0.8.0+), set `shared_buffers` to 25% of available RAM, `maintenance_work_mem` to 2 GB, and create HNSW indexes with `m=16, ef_construction=128`. Create B-tree indexes on all filter columns. For Qdrant, run via Docker with a memory limit matching available RAM minus 2 GB headroom; use default HNSW parameters (`m=16, ef_construct=128`); create payload indexes on filter fields. For LanceDB, use embedded Python mode with IVF-PQ indexing (`num_partitions=256, num_sub_vectors` tuned to dimensionality), and test with varying `nprobes` and `refine_factor` values as [recommended by LanceDB's benchmarking guide](https://medium.com/etoai/benchmarking-lancedb-92b01032874a).

**Test phases and metrics.** The protocol has five phases, each collecting recall@10, recall@20, P50/P95/P99 latency, and peak RSS memory:

- **Phase 1 — Index build.** Bulk-load all 500K vectors. Record wall-clock build time and peak memory. For pgvector, measure both `CREATE INDEX` time and total COPY + index time. LanceDB's `create_index` call includes both data write and index construction. Qdrant begins serving immediately during background indexing — record time until the collection leaves "yellow" optimization state.

- **Phase 2 — Static search.** Run 10,000 queries single-threaded to establish baseline latency, then run with 8 and 32 concurrent threads to measure throughput scaling. Following [Qdrant's methodology](https://qdrant.tech/benchmarks/), compare performance only at matched recall levels — sweep query parameters (ef_search for pgvector/Qdrant, nprobes for LanceDB) to hit 0.90, 0.95, and 0.99 recall targets, and report latency and QPS at each target.

- **Phase 3 — Filtered search.** Repeat Phase 2 with metadata predicates at four selectivity levels: 50%, 10%, 1%, and compound filters combining two predicates. This phase will expose the most dramatic performance differences. pgvector's [iterative scan mechanism](https://aws.amazon.com/blogs/database/supercharging-vector-search-performance-and-relevance-with-pgvector-0-8-0-on-amazon-aurora-postgresql/) should handle high-selectivity filters gracefully; Qdrant's [filterable HNSW](https://qdrant.tech/benchmarks/filtered-search-intro/) integrates filtering into graph traversal; LanceDB applies filters during partition pruning in IVF. Track recall separately for each selectivity level, since accuracy often degrades under aggressive filtering.

- **Phase 4 — Streaming ingestion.** Adopt [VDBBench's streaming protocol](https://zilliz.com/blog/vdbbench-1-0-benchmarking-with-your-real-world-production-workloads): after initial load of 400K vectors, begin inserting the remaining 100K at a controlled rate (100 rows/second from a background thread) while running continuous search queries. Record search latency and recall every 10K inserts. This phase tests the interaction between write amplification and read performance — a critical factor for RAG systems that ingest new documents throughout the day. pgvector's HNSW [insert performance degrades with index size](https://github.com/pgvector/pgvector/issues/588) (each insert updates the graph), while LanceDB's [append-based format](https://docs.lancedb.com/faq/faq-oss) and Qdrant's segment-based architecture handle streaming writes differently.

- **Phase 5 — Recovery.** Kill each database process with SIGKILL during active ingestion, then measure cold-start time to first successful query. For pgvector, this means PostgreSQL WAL replay. For Qdrant, this triggers [WAL recovery and potentially recovery mode](https://qdrant.tech/articles/qdrant-1.2.x/). For LanceDB, the Lance format's transactional writes should provide immediate consistency on restart. Record both time-to-first-query and whether any data loss occurred.

**Avoiding common pitfalls.** As [Qdrant's benchmark FAQ](https://qdrant.tech/benchmarks/benchmark-faq/) emphasizes, never compare databases at different recall levels — a 5% recall gap can easily explain a 2× throughput difference. Pre-warm all caches with throwaway queries before measurement runs. Run each configuration a minimum of three times and report medians. Use `pg_prewarm` for pgvector to ensure the HNSW index is [in shared memory before testing](https://www.clarvo.ai/blog/optimizing-filtered-vector-queries-from-tens-of-seconds-to-single-digit-milliseconds-in-postgresql). Monitor with OS-level tools (`/proc/[pid]/status` or Docker stats) rather than self-reported metrics. And perhaps most importantly, as one [small-scale comparison found](https://medium.com/@TheWake/qdrant-vs-pgvector-theyre-the-same-speed-5ac6b7361d9d), at 500K chunks the embedding computation itself often dominates end-to-end latency by 20:1 — so optimize the embedding pipeline before obsessing over vector search microseconds.

## Conclusion

The gap between what ann-benchmarks measures and what RAG systems demand is not incremental — it is structural. A benchmark that ignores filtered search, concurrent ingestion, tail latency, and memory constraints under load will systematically mislead database selection for production RAG. The most valuable insight from this analysis is that **no single existing benchmark covers the full RAG surface area**; VDBBench comes closest with its streaming and filtered-search tests, but still omits crash recovery and memory profiling. For a local-first 500K-chunk deployment, the five-phase protocol described above — static search, filtered search at multiple selectivities, streaming ingestion, and crash recovery — provides the minimum viable evaluation surface. The architectural differences between the three candidates (pgvector's PostgreSQL-native filtering and MVCC, Qdrant's filterable HNSW and segment optimization, LanceDB's disk-native columnar format) mean that each will dominate different phases. The engineer's job is to weight those phases according to the actual workload profile, not according to whichever recall-versus-QPS chart looks best on a vendor's blog.
