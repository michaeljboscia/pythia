# PYTHIA MASTER DESIGN SPECIFICATION

## 1. System Overview
Pythia is a self-updating, deterministic digital twin of a codebase equipped with a persistent AI reasoning daemon. Operating entirely locally as an MCP server, it uses a dual-engine indexer (Tree-sitter + LSP) to map the codebase into a hybrid vector-graph database (SQLite), and provides an LLM with the context required to reason across thousands of files without hallucination or context window exhaustion.

## 2. The Four-Layer Architecture
The architecture strictly decouples deterministic data (Code) from probabilistic reasoning (LLM), and abstracts the infrastructure to allow for $0 local execution or premium cloud scaling.

```text
+-------------------------------------------------------------+
|                     UI PLANE (OBSIDIAN)                     |
|  Reads markdown MADRs generated by Pythia. Passive view.    |
+-------------------------------------------------------------+
                              ^
                              | (Writes MADR files)
                              v
+-------------------------------------------------------------+
|                COGNITION PLANE (PYTHIA DAEMON)              |
|  Maintains conversational state. Writes Episodic Checkpoints|
|  Provider: Gemini CLI ($0 fallback) OR Gemini SDK (Premium) |
+-------------------------------------------------------------+
                              ^
                              | (MCP Protocol JSON-RPC)
                              v
+-------------------------------------------------------------+
|                  INTERFACE PLANE (MCP SERVER)               |
|  Exposes lcs_investigate, oracle_commit_decision, etc.      |
|  Acts as Adaptive Router & Re-ranker (ONNX Cross-Encoder)   |
+-------------------------------------------------------------+
                              ^
                              | (SQL / Vector Search)
                              v
+-------------------------------------------------------------+
|                  DATA PLANE (PYTHIA DB)                  |
|  Storage: SQLite-vec + FTS5 + Recursive CTEs (Single File)  |
|  Indexer: Tree-sitter (Fast Path) + TS LSP (Slow Path)      |
+-------------------------------------------------------------+
```

## 3. The ReasoningProvider Interface
The system requires an abstraction over the AI daemon to support both the zero-cost CLI hack and the high-performance SDK. The MCP server instantiates the appropriate provider based on the presence of `GEMINI_API_KEY` in `~/.pythia/config.json`.

```typescript
// The fundamental interface (D-80)
export interface ReasoningProvider {
  /**
   * Boot the daemon.
   * CLI: Creates ~/.gemini/tmp/daemon-<name>
   * SDK: Initializes ai.chats.create() in memory
   */
  spawn(sessionName: string, systemInstruction: string, contextChunks: string[]): Promise<string>;

  /**
   * Send a query and return the response.
   * CLI: Spawns new process, parses full history (O(N) latency).
   * SDK: Uses KV-cached chat session (~3s latency).
   */
  ask(sessionId: string, prompt: string, additionalContext?: string[]): Promise<string>;

  /**
   * Terminate the daemon.
   * CLI: SIGKILLs the tracked PID and deletes tmp dir.
   * SDK: Releases memory.
   */
  dismiss(sessionId: string): Promise<void>;
}
```

## 4. The Dual-Engine Indexer Pipeline
Code intelligence requires both syntactic speed and semantic precision (D-76).

**The Fast Path (Vector Indexing)**
*   **Trigger:** `sync_workspace` runs at boot and on manual `pythia_force_index` calls, using the unified MTime/Hash File Scanner defined in Section 11.5.
*   **Pipeline:** Raw file → Tree-sitter AST → Chunked by Function/Class → Embedded via `nomic-embed-text-v1.5` (ONNX, truncated to 256d) → Upserted to `sqlite-vec`.
*   **Timing:** < 100ms per file. Updates are available to semantic search almost instantly.

**The Slow Path (Semantic Graph Building)**
*   **Trigger:** Runs asynchronously after the Fast Path completes a batch.
*   **Pipeline:** TypeScript Compiler API (`tsserver`) walks the AST → Resolves identifiers (`LanguageService.getDefinitionAtPosition()`) → Generates exact `CALLS` and `IMPORTS` edges → Upserts to `graph_edges` table.
*   **Timing:** ~500ms - 2s per file. Runs in a background Worker Thread (D-83) to prevent blocking the MCP event loop.

## 5. The Six MCP Tools (v1)
The tool surface is consolidated to prevent LLM confusion and optimized for plain-text output (D-81). `lcs_global_search` is deferred to Sprint 2 (§11.7) and is not registered in v1.

1.  **`lcs_investigate`**
    *   *Description:* Primary retrieval tool. Use this to find code, documentation, or architectural context.
    *   *Input:* `{ query: string; intent: "semantic" | "structural" }`
    *   *Output:* Plain-text formatted chunks with file metadata.

2.  **`pythia_force_index`**
    *   *Description:* Manually trigger the indexing pipeline if context feels stale.
    *   *Input:* `{ path?: string }` // Optional subdirectory
    *   *Output:* Success/failure string.

3.  **`spawn_oracle`**
    *   *Description:* Start a stateful reasoning session.
    *   *Input:* `{ session_name: string; initial_context_query: string }`
    *   *Output:* JSON string: `{"session_id":"<uuid>","status":"active","created":true|false,"generation_id":N,"decommission_secret":"<32-char-hex>"}` — `decommission_secret` is only present when `created: true`.

4.  **`ask_oracle`**
    *   *Description:* Send a message to an active oracle session.
    *   *Input:* `{ session_id: string; message: string; additional_context_query?: string }`
    *   *Output:* The oracle's response string.

5.  **`oracle_commit_decision`**
    *   *Description:* Permanently record an architectural decision. Use this immediately after resolving a complex design problem.
    *   *Input:* `{ title: string; problem: string; drivers: string[]; options: string[]; decision: string; impacts_files: string[] }`
    *   *Output:* `madr_id` (e.g., MADR-012).

6.  **`oracle_decommission`**
    *   *Description:* Securely wipe a session's temporary state (does not delete MADRs).
    *   *Input:* `{ session_id: string; verification_phrase: string }`
    *   *Output:* Success string.

## 6. The Pythia Database Schema
The unified SQLite database (`.pythia/lcs.db`) holds code, vectors, and memory (D-85).

```sql
-- 1. The Raw Code Chunks
CREATE TABLE lcs_chunks (
    id TEXT PRIMARY KEY,           -- CNI e.g., 'src/auth.ts::function::login'
    file_path TEXT NOT NULL,
    chunk_type TEXT NOT NULL,      -- 'function', 'class', 'method', 'interface', 'type', 'enum', 'namespace', 'doc', 'module'
    content TEXT NOT NULL,         -- The raw source code
    start_line INTEGER,            -- AST node start line (Tree-sitter, 0-based)
    end_line INTEGER,              -- AST node end line (Tree-sitter, 0-based)
    is_deleted BOOLEAN DEFAULT 0,  -- Soft delete tombstone
    deleted_at TEXT NULL,          -- ISO8601 timestamp when soft-deleted (GC uses this)
    content_hash TEXT NOT NULL     -- algo:digest (e.g., 'blake3:abc123...') for CDC
);

-- 2. The Vector Index (sqlite-vec extension)
CREATE VIRTUAL TABLE vec_lcs_chunks USING vec0(
    id TEXT PRIMARY KEY,
    embedding float[256]           -- 256d Matryoshka truncated vector
);

-- 3a. Keyword FTS Index — exact symbol matching (unicode61 + code tokenchars)
CREATE VIRTUAL TABLE fts_lcs_chunks_kw USING fts5(
    id UNINDEXED,
    content,
    tokenize="unicode61 tokenchars '._:/#<>?!-'"
);

-- 3b. Substring FTS Index — trigram fallback for quoted/path/CNI queries
CREATE VIRTUAL TABLE fts_lcs_chunks_sub USING fts5(
    id UNINDEXED,
    content,
    tokenize="trigram"
);

-- 4a. Per-File Scan Cache — mtime/hash CDC state (survives server restart)
CREATE TABLE file_scan_cache (
    file_path TEXT PRIMARY KEY,
    mtime_ns INTEGER NOT NULL,     -- file mtime in nanoseconds
    size_bytes INTEGER NOT NULL,
    content_hash TEXT NOT NULL,    -- algo:digest of last-indexed content
    last_scanned_at TEXT NOT NULL  -- ISO8601
);

-- 5. The Episodic Memory (Pythia Checkpoints)
CREATE TABLE pythia_memories (
    seq INTEGER PRIMARY KEY AUTOINCREMENT,
    id TEXT NOT NULL UNIQUE,       -- e.g., 'MADR-012', derived as printf('MADR-%03d', seq)
    generation_id INTEGER NOT NULL,
    timestamp TEXT NOT NULL,
    status TEXT NOT NULL,          -- 'accepted', 'superseded'
    title TEXT NOT NULL,
    context_and_problem TEXT NOT NULL,
    decision_drivers TEXT NOT NULL, -- JSON array
    considered_options TEXT NOT NULL, -- JSON array
    decision_outcome TEXT NOT NULL,
    supersedes_madr TEXT,
    FOREIGN KEY(supersedes_madr) REFERENCES pythia_memories(id)
);

-- 6. The Polymorphic Graph Edges
CREATE TABLE graph_edges (
    source_id TEXT NOT NULL,
    target_id TEXT NOT NULL,
    edge_type TEXT NOT NULL,       -- 'CALLS', 'IMPORTS', 'IMPLEMENTS'
    PRIMARY KEY (source_id, target_id, edge_type)
);
CREATE INDEX idx_graph_edges_target ON graph_edges(target_id, edge_type);

-- 7. Embedding Model Version Pinning (singleton)
CREATE TABLE embedding_meta (
    id INTEGER PRIMARY KEY CHECK (id = 1),
    provider TEXT NOT NULL,
    model_name TEXT NOT NULL,
    model_revision TEXT NOT NULL,
    dimensions INTEGER NOT NULL,
    normalization TEXT NOT NULL,
    indexed_at TEXT NOT NULL
);

-- NOTE: lcs_communities and vec_communities are deferred to Sprint 2 (§11.7).
-- They will be added via forward-only migration when lcs_global_search is implemented.
```

## 8. The Obsidian-as-Glass Memory Model
Obsidian acts purely as the UI layer (D-82). Pythia does not read Obsidian's proprietary structures; it writes to them.
*   **The Structure:** The MCP server writes to a `Pythia/` directory inside the resolved vault root (e.g., `<repo>/Pythia/`). The `.obsidian/` directory is used only as a vault detection marker — never as a content destination.
*   **The Write:** When `oracle_commit_decision` is called, the MCP server inserts the JSON into SQLite, and simultaneously renders a formatted Markdown file (`MADR-012-Supabase-Middleware.md`) into the Obsidian vault.
*   **The Content:** The Markdown file contains YAML frontmatter mapping to the SQLite schema, allowing Obsidian's Dataview plugin to render automated dashboards of architectural decisions.

## 9. The Boot Sequence
1.  **Cold Start:** User runs `pythia init`. System creates `.pythia/lcs.db`, runs all pending migrations, validates config, and kicks off the first full cold-start index via the Worker Thread. `pythia init` does NOT start the MCP server.
2.  **Warm Start:** User runs `pythia start`. The MCP server process launches, runs pending migrations, then runs `sync_workspace` — scanning mtime against `file_scan_cache` and BLAKE3 hashes against `content_hash`. Only changed files are re-indexed. Tools are available immediately; indexing completes in background.
3.  **Generation Reconstitution:** Claude Code calls `spawn_oracle`. The MCP server executes:
    `SELECT * FROM pythia_memories WHERE status = 'accepted' ORDER BY timestamp ASC;`
    The results are formatted into a JSON string and injected directly into the `systemInstruction` of the ReasoningProvider.

## 10. The First-Sprint Build Order
Build strictly from the bottom up. Do not build the daemon until the indexer works.

1.  **Sprint 1: The Core Data Plane (SQLite + ONNX)**
    *   Set up `sqlite-vec` and `transformers.js`.
    *   Implement basic file chunking (regex first, Tree-sitter later).
    *   *Proof:* Write a script that embeds a file and retrieves it via cosine similarity.
2.  **Sprint 2: Tree-sitter & The Fast Path**
    *   Integrate `tree-sitter-typescript`.
    *   Extract functions/classes and upsert to SQLite.
    *   *Proof:* `lcs_investigate` MCP tool returns perfect AST-bounded chunks.
3.  **Sprint 3: The Graph Engine & Slow Path**
    *   Integrate TypeScript Compiler API.
    *   Extract `CALLS` and `IMPORTS` edges.
    *   *Proof:* SQL CTE successfully traverses from Function A to Function C.
4.  **Sprint 4: The Reasoning Provider & Pythia Tools**
    *   Implement `ReasoningProvider` interface (CLI first).
    *   Implement `oracle_commit_decision` JSON extraction and SQLite insertion.
    *   *Proof:* End-to-end integration test. Claude asks a question, Pythia retrieves the code, makes a decision, and writes a MADR to Obsidian and SQLite.

## 11. Resolved Engineering Issues (Spec Patches)
The following patches resolve critical omissions in the initial design and establish the strict contracts required for Sprint 1.

### 11.1 Chunk Identity Scheme (CNI)
**Resolution for BLOCKING 1 & Sprint 1 Prep:**
All nodes in Pythia (Code, Modules, MADRs) share a Canonical Node Identity (CNI) scheme to prevent drift.
*   **Format:** `uri::type::name[#discriminator]`
*   **File/Module:** `src/auth.ts::module::default`
*   **Function:** `src/auth.ts::function::login`
*   **Method:** `src/auth.ts::class::AuthManager::method::login`
*   **Overloads/Duplicates:** Append a 0-based index or line number hash (e.g., `src/auth.ts::function::helper#L45`).
*   **Anonymous/Default Exports:** Use `#default` or `#anonymous_L{line}`.
*   This CNI is strictly used as the `id` in `lcs_chunks`, `vec_lcs_chunks`, `fts_lcs_chunks`, and `graph_edges`.

### 11.2 Graph Schema & Edge Taxonomy
**Resolution for BLOCKING 2:**
The `graph_edges` table must differentiate between file-level and symbol-level endpoints. 
*   **Nodes:** Files are implicitly represented in `lcs_chunks` where `chunk_type = 'module'`. 
*   **Taxonomy:**
    *   `IMPORTS`: Always `module` -> `module`.
    *   `CALLS`: Always `function/method` -> `function/method`.
    *   `CONTAINS`: Always `module` -> `function/class` (hierarchy).
    *   `IMPLEMENTS`: Always `MADR` -> `module/function/class`.
*   **Constraint:** `graph_edges` enforces that `source_id` and `target_id` must exist in `lcs_chunks` or `pythia_memories` via application-level validation (or SQLite triggers).

### 11.3 Session & Transcript Persistence
**Resolution for BLOCKING 3:**
The database requires dedicated tables to persist the actual daemon sessions and their temporal logs, replacing the fragile filesystem JSONL approach.
```sql
-- The Active Daemon Sessions
CREATE TABLE pythia_sessions (
    id TEXT PRIMARY KEY,           -- opaque UUID, used for all foreign keys
    name TEXT NOT NULL,            -- human-readable session name (e.g., 'auth-refactor')
    status TEXT NOT NULL,          -- 'active', 'idle', 'dead', 'decommissioned'
    decommission_hash TEXT,        -- Argon2id hash of the decommission secret
    decommission_salt TEXT,        -- Salt for Argon2id (required for verification)
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
-- Uniqueness enforced only among live sessions; reuse after decommission is allowed
CREATE UNIQUE INDEX idx_pythia_sessions_active_name
    ON pythia_sessions(name)
    WHERE status IN ('active', 'idle');

-- The Temporal Transcript (Event Sourcing Log)
CREATE TABLE pythia_transcripts (
    session_id TEXT NOT NULL,
    turn_index INTEGER NOT NULL,
    role TEXT NOT NULL,            -- 'user', 'model', 'system', 'tool'
    content TEXT NOT NULL,         -- JSON payload of the message
    timestamp TEXT NOT NULL,
    PRIMARY KEY (session_id, turn_index),
    FOREIGN KEY(session_id) REFERENCES pythia_sessions(id) ON DELETE CASCADE
);
```

### 11.4 `oracle_commit_decision` Input Alignment
**Resolution for BLOCKING 4:**
The `impacts_files` input parameter is mapped to the graph via the `IMPLEMENTS` edge.
*   **ID Allocation:** The MCP server auto-generates MADR IDs using the `seq INTEGER PRIMARY KEY AUTOINCREMENT` column in `pythia_memories` (see §13.8). The public `id` is derived as `printf('MADR-%03d', last_insert_rowid())` within the same `BEGIN IMMEDIATE` transaction. `COUNT(*) + 1` is explicitly superseded and must not be used.
*   **Impact Mapping:** When `impacts_files` (e.g., `["src/auth.ts"]`) is provided, the MCP server resolves it to the module CNI (`src/auth.ts::module::default`) and inserts a row into `graph_edges`: `source_id: 'MADR-012', target_id: 'src/auth.ts::module::default', edge_type: 'IMPLEMENTS'`.

### 11.5 The Change Data Capture (CDC) Model
**Resolution for BLOCKING 5 & Sprint 1 Prep:**
We abandon Git hooks and time-based polling in favor of a unified MTime/Hash File Scanner.
*   **The Model:** A `sync_workspace` routine runs at boot and manually via the `pythia_force_index` tool.
*   **The Logic:**
    1. Scan directory, respecting `.gitignore`.
    2. Check file `mtime` against a lightweight in-memory or SQLite cache.
    3. If `mtime` changed, read file and compute BLAKE3 hash.
    4. Compare hash to `lcs_chunks.content_hash`. If different, trigger Tree-sitter pipeline.
*   This handles untracked files, uncommitted edits, and avoids Git state entirely.

### 11.6 True Progressive Enhancement Abstraction
**Resolution for BLOCKING 6:**
The architecture must treat SQLite as the ultimate source of truth, treating Vector and Graph stores as replaceable derived indexes.
*   `lcs_chunks`, `pythia_memories`, and `pythia_sessions` are **always** in SQLite.
*   The `VectorStore` interface handles routing:
    *   If Default: Upserts to the local SQLite `vec_lcs_chunks` virtual table.
    *   If Premium: Pushes the vector payload to the external Qdrant cluster.
*   The `GraphStore` interface handles routing:
    *   If Default: Upserts to the local SQLite `graph_edges` table.
    *   If Premium: Mirrors the edges into the external FalkorDBLite instance.
*   This ensures swapping configs only rebuilds the derived index, never losing the raw code or episodic memory.

### 11.7 Community Summaries Pipeline (Deferred)
**Resolution for BLOCKING 7:**
`lcs_global_search` and the `lcs_communities` tables are explicitly **removed from Sprint 1**. 
*   Global thematic search via Leiden community detection is an advanced feature that requires its own design sub-spec for background LLM batch processing, cost control, and invalidation.
*   For Sprint 1, all RAG queries will use local hybrid search (`lcs_investigate`). The global search pipeline will be designed in Sprint 2 after the core AST graph is stabilized.

### 11.8 The Sync Contract (Data Integrity)
**Resolution for Technical Gap:**
To prevent the vector index, FTS index, and relational table from drifting, all indexing writes must occur within a strict SQLite transaction.
```typescript
// Pseudocode for the atomic sync contract
const now = new Date().toISOString();
db.exec('BEGIN TRANSACTION;');
try {
  // 1. Soft-delete old chunks for this file; stamp deleted_at for GC TTL
  db.run('UPDATE lcs_chunks SET is_deleted = 1, deleted_at = ? WHERE file_path = ?', [now, file]);

  // 2. Delete stale derived index rows by stale chunk id (delete-then-insert, NOT INSERT OR REPLACE)
  for (const staleId of staleChunkIds) {
    db.run('DELETE FROM vec_lcs_chunks WHERE id = ?', [staleId]);
    db.run('DELETE FROM fts_lcs_chunks_kw WHERE id = ?', [staleId]);
    db.run('DELETE FROM fts_lcs_chunks_sub WHERE id = ?', [staleId]);
    // Cascade graph edges to prevent phantom traversal
    db.run('DELETE FROM graph_edges WHERE source_id = ? OR target_id = ?', [staleId, staleId]);
  }

  // 3. Insert new chunks and derived index rows
  for (const chunk of newChunks) {
    db.run('INSERT INTO lcs_chunks (id, file_path, chunk_type, content, start_line, end_line, is_deleted, deleted_at, content_hash) VALUES (?, ?, ?, ?, ?, ?, 0, NULL, ?)',
      [chunk.id, chunk.filePath, chunk.type, chunk.content, chunk.startLine, chunk.endLine, chunk.contentHash]);
    db.run('INSERT INTO vec_lcs_chunks (id, embedding) VALUES (?, ?)', [chunk.id, chunk.embedding]);
    db.run('INSERT INTO fts_lcs_chunks_kw (id, content) VALUES (?, ?)', [chunk.id, chunk.content]);
    db.run('INSERT INTO fts_lcs_chunks_sub (id, content) VALUES (?, ?)', [chunk.id, chunk.content]);
  }

  db.exec('COMMIT;');
} catch (e) {
  db.exec('ROLLBACK;');
  throw e;
}
```

## 12. Decision Resolutions — Cycle 1
These are binding decisions resulting from the first architecture interrogation, resolving specific edge cases and requirements.

### 12.1 Extensibility of the ReasoningProvider
**Reference: Resolution 1 (Q6)**
The `ReasoningProvider` interface must not hard-wire Gemini assumptions. It acts as an abstraction over any LLM or daemon process capable of fulfilling the contract.
*   **v1 Implementations:** `SdkReasoningProvider` (Gemini SDK) and `CliReasoningProvider` (Gemini CLI hack).
*   **Future Stubs:** `LocalReasoningProvider` (Ollama/LM Studio) is reserved but not implemented in v1.

### 12.2 MCP Registration is Explicit
**Reference: Resolution 2 (Q13)**
`pythia init` must not silently mutate the host's global configuration files (e.g., `~/.claude.json`).
*   Registration requires explicit user action via `pythia mcp install claude-code`.
*   This command must be idempotent and must preview the configuration changes to standard output before applying them.

### 12.3 Context Window Cap for `spawn()`
**Reference: Resolution 3 (Q15)**
The `spawn()` method is restricted to initializing the preamble and structural memory.
*   **Hard Cap:** The serialized `contextChunks` string passed to `spawn()` must not exceed 180,000 characters.
*   **Overflow:** If the context exceeds this limit, the provider must truncate lower-priority chunks and rely on dynamic retrieval during the `ask()` cycle.

### 12.4 Inactivity Reaper (TTL)
**Reference: Resolution 4 (Q16)**
Local MCP sessions are ephemeral and scoped to active coding sessions.
*   **TTL:** The MCP server's inactivity reaper will automatically execute `dismiss()` on any session that has not received an `ask()` call within **30 minutes**.

### 12.5 Hard Fail on Invalid Credentials
**Reference: Resolution 5 (Q17)**
Silently downgrading from a targeted API (SDK) to a generalized endpoint (CLI) without user consent violates privacy expectations.
*   If the `SdkReasoningProvider` receives a `401 Unauthorized` or similar auth failure, it must immediately throw an `AUTH_INVALID` error back to the MCP client.
*   The spawn attempt is marked as failed. No silent fallback occurs.

### 12.6 Single Active Session Constraint
**Reference: Resolution 6 (Q18)**
To prevent uncoordinated state mutation and context splintering, Pythia supports only one active Oracle session per repository in v1.
*   If `spawn_oracle` is called with the name of an already-active session, it returns the existing `session_id`.
*   If `spawn_oracle` is called with a new name while another session is active, it returns a `SESSION_ALREADY_ACTIVE` error. It does not automatically dismiss the live state.

### 12.7 Sequential MADR ID Generation
**Reference: Resolution 7 (Q28)**
MADR (Architecture Decision Record) IDs must be human-readable and sequential.
*   **Mechanism:** Pythia uses the native SQLite `INTEGER PRIMARY KEY AUTOINCREMENT` on a shadow column in `pythia_memories` (or relies on the implicit `rowid`).
*   **Formatting:** The public `id` is derived within the same insertion transaction using string formatting (e.g., `MADR-%03d`). External ID generators (like ULID) are rejected.

### 12.8 API Retry Logic Encapsulation
**Reference: Resolution 8 (Q38)**
The MCP caller should not handle transient provider errors.
*   **Responsibility:** The `ReasoningProvider` implementation wraps its network calls with an exponential backoff retry loop (1s, 5s, 15s).
*   **Failure:** Only after all retries are exhausted does the provider return a `PROVIDER_UNAVAILABLE` error to Claude Code.

### 12.9 Durable Schema Migration
**Reference: Resolution 9 (Q40)**
Wiping the local database on a version upgrade is unacceptable, as it destroys the architectural memory.
*   **Durable Tables:** `lcs_chunks`, `pythia_memories`, `pythia_sessions`, and `pythia_transcripts` must be migrated in-place using strictly forward-only SQL migration scripts.
*   **Ephemeral Indexes:** Derived tables (`vec_lcs_chunks`, `fts_lcs_chunks`, `graph_edges`) may be safely dropped and rebuilt from the durable tables during a schema migration if required.

## 13. Decision Resolutions — Cycle 2
These are binding decisions resulting from the second architecture interrogation cycle, synthesizing answers from dual-daemon review (Gemini oracle + Codex). Where daemons diverged, the technically superior answer is adopted with explicit justification.

### 13.1 Soft-Delete GC Policy (Q1)
**Decision:** `is_deleted=1` rows are retained for **30 days**, then hard-purged by a GC job that runs at every boot and after any sync batch where tombstones exceed 10,000 rows or 20% of total chunk count.
**Adopted from:** Codex. Referential integrity for MADR history is preserved at the `graph_edges` (IMPLEMENTS) and `pythia_transcripts` layer, not by keeping stale chunk rows forever. GC must also delete matching rows from `vec_lcs_chunks`, `fts_lcs_chunks`, and `graph_edges`, then run `PRAGMA incremental_vacuum`. The `gc.deleted_chunk_retention_days` config field controls the TTL.

### 13.2 Dual FTS5 Index Strategy (Q2)
**Decision:** Two FTS virtual tables are required: `fts_lcs_chunks_kw` (tokenize=`unicode61` with code-friendly tokenchars) for exact symbol matching, and `fts_lcs_chunks_sub` (tokenize=`trigram`) for substring fallback. `lcs_investigate` queries `fts_lcs_chunks_kw` first; `fts_lcs_chunks_sub` is only used for quoted/substring-oriented queries.
**Adopted from:** Codex. A single tokenizer cannot serve both exact-keyword and substring search without precision loss. The dual-index adds modest storage cost but eliminates false positives for exact symbol lookups.

### 13.3 `ask()` Context Injection Ceiling (Q3)
**Decision:** `additionalContext` passed to `ask()` is capped at **12 chunks** and **48,000 serialized characters** total. The MCP server trims excess chunks by rank before the provider call. If total prompt + transcript + context still exceeds the session budget, `ask_oracle` returns `CONTEXT_BUDGET_EXCEEDED`.
**Adopted from:** Codex. 12,000 chars (Gemini's proposal) is too restrictive for multi-chunk retrieval-augmented queries. The MCP server owns the budget; callers cannot exceed it.

### 13.4 Worker Thread Crash Supervisor (Q4)
**Decision:** The MCP server supervises the Slow Path Worker Thread. On crash: mark indexing as `degraded`, restart with exponential backoff. After **3 crashes within 10 minutes**, trip a circuit breaker — auto-restarts stop, the MCP server continues serving Fast Path results only, and `lcs_investigate` indicates structural results may be stale. Only a manual `pythia_force_index` or process restart clears the degraded state.
**Adopted from:** Codex. A bare restart loop (Gemini) can thrash on OOM or native module failures. The circuit breaker prevents resource exhaustion while keeping the MCP server operational.

### 13.5 Re-Export CNI Resolution (Q5)
**Decision:** Re-exports do NOT get duplicate symbol CNIs. The defining symbol holds the canonical CNI. The re-export site is represented as a `RE_EXPORTS` edge from the barrel module's CNI to the canonical symbol's CNI.
**Edge taxonomy addition:** `RE_EXPORTS`: always `module` → `function/class`.
**Adopted from:** Codex (with Gemini agreeing on canonicalization; Codex adds the explicit `RE_EXPORTS` edge which preserves re-export topology for graph traversal).

### 13.6 Graph Traversal Depth Limit (Q6)
**Decision:** Recursive CTE traversals for `intent: "structural"` are hard-capped at **depth 6**. Cycle detection is implemented by carrying a visited-path string column and rejecting any hop already present in that path. Materialized form: `(node_id, depth, path TEXT)`, halt when `depth >= 6` OR candidate is in `path`.
**Adopted from:** Codex. Depth 3 (Gemini) is insufficient to cross realistic 3-tier module boundaries. Depth 6 with explicit path-string cycle detection is deterministic and bounded.

### 13.7 `pythia_transcripts` Content Schema (Q7)
**Decision:** The `content` column stores role-specific JSON objects:
- `user`: `{ "text": string }`
- `model`: `{ "text": string, "provider": string, "model": string, "finish_reason": string, "usage"?: { "input_tokens": number, "output_tokens": number } }`
- `system`: `{ "kind": string, "text": string, "metadata"?: object }` — `kind` is typed: `"spawn_preamble"`, `"reaper_notice"`, `"context_trim_notice"`
- `tool`: `{ "tool_name": string, "tool_call_id": string, "input": object, "output": object|null, "status": "success"|"error", "error_code"?: string, "duration_ms": number }`
**Adopted from:** Codex. Role-specific typed schemas enable session replay, auditing, and partial failure diagnosis. Gemini's flat structure is insufficient.

### 13.8 Atomic MADR ID Generation (Q8)
**Decision:** `pythia_memories` adds an `INTEGER PRIMARY KEY AUTOINCREMENT` column named `seq`. The public `id` is derived as `printf('MADR-%03d', seq)` inside the same INSERT transaction using `last_insert_rowid()`. If SQLite generated columns are unavailable, fallback is `INSERT ...; UPDATE pythia_memories SET id = printf('MADR-%03d', last_insert_rowid()) WHERE rowid = last_insert_rowid()` inside `BEGIN IMMEDIATE`.
**Adopted from:** Codex. Separates sequence allocation (SQLite) from ID formatting (application layer) in a single atomic transaction. Superior to trigger-based UPDATE (Gemini) due to portability.

### 13.9 `oracle_decommission` Verification Contract (Q9)
**Decision:** `spawn_oracle` generates a **128-bit random secret**, returns it once in the spawn response as `decommission_secret`, stores only its Argon2id hash in `pythia_sessions`. `oracle_decommission.verification_phrase` must be exactly `DECOMMISSION <session_id> <secret>`. Hash mismatch is a hard failure.
**Adopted from:** Codex. Gemini's 6-char string provides only ~36 bits of entropy — insufficient for a destructive, irreversible operation. 128-bit entropy + Argon2id prevents brute force.

### 13.10 Obsidian Vault Path Resolution (Q10)
**Decision:** The workspace root is the **repository root containing `.pythia/`**. `~/.pythia/config.json` may specify an `obsidian_vault_path` override. If no directory with `.obsidian/` exists at the resolved path, Obsidian mirroring is silently disabled — `oracle_commit_decision` still writes to SQLite and returns `OBSIDIAN_DISABLED` in its response metadata.
**Adopted from:** Codex. CWD (Gemini) is nondeterministic in multi-repo MCP setups. The repo root containing `.pythia/` is stable across MCP restarts.

### 13.11 Embedding Model Version Pinning (Q11)
**Decision:** A new `embedding_meta` table stores `{ provider, model_name, model_revision, dimensions, normalization }` as the active fingerprint. On startup, if runtime fingerprint differs from stored fingerprint, the vector index is marked stale: the system serves **FTS-only results** until a full re-embed completes, then atomically swaps in the new index.
**Schema addition:**
```sql
CREATE TABLE embedding_meta (
    id INTEGER PRIMARY KEY CHECK (id = 1), -- singleton row
    provider TEXT NOT NULL,
    model_name TEXT NOT NULL,
    model_revision TEXT NOT NULL,
    dimensions INTEGER NOT NULL,
    normalization TEXT NOT NULL,
    indexed_at TEXT NOT NULL
);
```
**Adopted from:** Codex. FTS-only fallback during rebuild (Codex) is more user-friendly than a hard block (Gemini).

### 13.12 `pythia_force_index` Concurrency (Q12)
**Decision:** A process-local **mutex** guards all sync work. `pythia_force_index` and background sync share this mutex. Overlapping requests are **coalesced**: if a force-index arrives for a path that is already mid-sync, it is merged into the live job rather than queued.
**Adopted from:** Codex. Coalescing is semantically correct — merging overlapping path-scoped jobs avoids double-work and prevents the queue from growing unboundedly.

### 13.13 Cold-Start Progress Signal (Q13)
**Decision:** While cold-start indexing is in progress, every `lcs_investigate` response prepends a **machine-readable metadata header** with: `index_state`, `indexed_files`, `total_files`, `percent_complete`, `last_sync_at`. Once the index reaches 100%, the header collapses to `index_state: "ready"`. No separate `pythia_status` tool is added in v1.
**Adopted from:** Codex. Machine-readable fields (Codex) are parseable by code and tooling. Prose warnings (Gemini) serve only LLM consumption.

### 13.14 Premium Stack Vector Migration (Q14)
**Decision:** The $0 stack (`vec_lcs_chunks` 256d SQLite) and Premium stack (Qdrant 1024d) are **separate backends** — there is no in-place migration. Switching stacks triggers a full re-embed from `lcs_chunks` into the newly selected backend. The old backend is marked inactive, not immediately dropped. The `pythia migrate` CLI command orchestrates this transition.
**Adopted from:** Codex. Maintaining separate backends is architecturally correct per §11.6. Forcing SQLite to hold 1024d vectors (Gemini) contradicts the Premium stack's mandate of Qdrant for vector storage.

### 13.15 `pythia_sessions` Schema Patch (Q15)
**Decision:** `pythia_sessions` must add a `name TEXT NOT NULL` column and a **partial unique index** on `(name)` for rows where `status IN ('active', 'idle')`. `spawn_oracle` uses `session_name` for idempotent lookup; all foreign keys use the opaque `id`.
```sql
ALTER TABLE pythia_sessions ADD COLUMN name TEXT NOT NULL DEFAULT '';
CREATE UNIQUE INDEX idx_pythia_sessions_active_name 
    ON pythia_sessions(name) 
    WHERE status IN ('active', 'idle');
```
**Both daemons agreed** on this decision. Codex's partial unique index is adopted as it correctly permits the same name to be reused after a session is decommissioned.

### 13.16 Graph Edge Validation Enforcement (Q16)
**Decision:** Graph edge validation is enforced by **SQLite `BEFORE INSERT` triggers** on `graph_edges`. An invalid insert (source or target not found in `lcs_chunks` or `pythia_memories`) aborts the entire sync transaction with `INVALID_GRAPH_ENDPOINT`.
**Adopted from:** Codex. Integrity logic belongs in the database. Performance concern (Gemini) is overstated since edge volume is low relative to chunk volume. A single bad edge rolling back the entire batch is the correct behavior.

### 13.17 LSP Server Lifecycle (Q17)
**Decision:** The Slow Path Worker Thread owns **one long-lived `tsserver` process per workspace**. It is reused across batches, isolated from the MCP event loop. On crash: restart with backoff, re-queue affected files. The MCP server does not own `tsserver` directly.
**Both daemons agreed** on the long-lived model. Codex's explicit note on MCP event loop isolation is adopted as a binding constraint.

### 13.18 Config Schema (Q18)
**Decision:** `~/.pythia/config.json` is validated at startup via Zod against this schema. Missing required fields are a hard startup error.
```typescript
// Minimum required shape
{
  workspace_path: string,
  obsidian_vault_path?: string,
  reasoning: { mode: "cli" | "sdk", gemini_api_key?: string },
  embeddings: { mode: "local" | "voyage", model: string, revision: string },
  vector_store: { mode: "sqlite" | "qdrant", qdrant_url?: string },
  graph_store: { mode: "sqlite" | "falkor" },
  limits: { spawn_chars_max: number, ask_context_chars_max: number },
  indexing: { scan_on_start: boolean, max_worker_restarts: number },
  gc: { deleted_chunk_retention_days: number }
}
```
**Adopted from:** Codex (concrete schema shape). Zod validation approach agreed by both daemons.

### 13.19 BLAKE3 Dependency (Q19)
**Decision:** File hashing uses **`hash-wasm`** (WASM, no native compilation). If WASM initialization fails, fallback is Node.js `crypto` SHA-256. The `content_hash` column stores `algo:digest` format (e.g., `blake3:abc123...` or `sha256:def456...`) so mixed-algorithm records never compare as equal.
**Adopted from:** Codex. WASM is faster than pure JS (`@noble/hashes`, Gemini's choice) and more portable than native bindings. The `algo:digest` format is a critical correctness invariant for mixed-algorithm detection.

### 13.20 `lcs_investigate` Ranking Algorithm (Q20)
**Decision:** Hybrid search uses **weighted Reciprocal Rank Fusion** over top-30 vector hits and top-30 keyword-FTS hits. Fusion formula: `score = wv/(60 + rank_vec) + wf/(60 + rank_fts)`. Weights are intent-dependent: `intent="semantic"` uses `wv=0.7, wf=0.3`; `intent="structural"` uses `wv=0.3, wf=0.7`. The top-12 fused candidates are then passed through the ONNX cross-encoder for final re-ranking. The cross-encoder is a second-stage re-ranker owned by the MCP server.
**Adopted from:** Codex. Intent-based weight inversion is a significant precision improvement over fixed weights (Gemini). Top-12 into cross-encoder (Codex) vs. top-50 (Gemini) is more practical given cross-encoder latency.

## 14. Decision Resolutions — Cycle 3
These are binding decisions from the third architecture interrogation cycle. Where daemons diverged, the technically superior answer is adopted.

### 14.1 `embedding_meta` Bootstrap Contract (Q1)
**Decision:** `pythia init` creates the `embedding_meta` table (via the forward-only migration set) but does NOT insert a fingerprint row. The singleton row is inserted only at the end of the first successful full vector build, inside the same commit transaction. Before that row exists, `SELECT id=1` returning no row means "vector index uninitialized."
**Adopted from:** Codex. Writing a pre-embed fingerprint makes `indexed_at` inaccurate and weakens startup drift detection.

### 14.2 Dual FTS Sync Contract (Q2)
**Decision:** The §11.8 atomic transaction pseudocode is updated. Within `BEGIN TRANSACTION`, after soft-deleting old chunks, the indexer must:
1. `DELETE FROM fts_lcs_chunks_kw WHERE id = ?` (for each stale chunk in this file)
2. `DELETE FROM fts_lcs_chunks_sub WHERE id = ?` (for each stale chunk)
3. `INSERT INTO fts_lcs_chunks_kw (id, content) VALUES (?, ?)` (for each new chunk)
4. `INSERT INTO fts_lcs_chunks_sub (id, content) VALUES (?, ?)` (for each new chunk)
Plain `INSERT` (not `INSERT OR REPLACE`) is correct because FTS5 tables with unindexed `id` columns do not correctly support `OR REPLACE` semantics.
**Adopted from:** Codex.

### 14.3 Circuit Breaker Recovery Path (Q3)
**Decision:** Circuit breaker state is process-local (in-memory) only. Restarting the MCP server resets the crash counter and clears `degraded`. Within the same process, calling `pythia_force_index` also triggers a supervised recovery batch: if that batch completes with a healthy heartbeat, the circuit breaker resets.
**Adopted from:** Codex (both daemons agreed on in-memory; Codex adds in-process recovery via `pythia_force_index`).

### 14.4 Argon2id Parameters for Decommission Secret (Q4)
**Decision:** The decommission secret is hashed with Argon2id via `hash-wasm` (WASM, no native bindings). Parameters: `memory_cost=65536 KiB`, `time_cost=3`, `parallelism=1`, `hash_length=32`, `salt_length=16`. `hash-wasm` is already the chosen WASM runtime for BLAKE3, avoiding a second WASM dependency.
**Adopted from:** Codex. Stronger memory cost (65536 vs. 19456) is appropriate for a destructive, irreversible operation.

### 14.5 Transcript Replay Policy (Q5)
**Decision:** `pythia_transcripts` are NOT replayed during session reconstitution. Only accepted `pythia_memories` (MADRs) are injected into the `systemInstruction`. Transcripts remain an offline audit log and evaluation corpus only.
**Both daemons agreed.**

### 14.6 `lcs_global_search` v1 Status (Q6)
**Decision:** `lcs_global_search` is completely absent from the v1 MCP manifest. It is not registered as a tool. It returns no stub. It does not appear in tool discovery.
**Both daemons agreed.**

### 14.7 Cross-Encoder Model Identity (Q7)
**Decision:** The v1 cross-encoder is `cross-encoder/ms-marco-MiniLM-L-6-v2`, exported to ONNX, downloaded lazily from HuggingFace Hub on first rerank call, cached locally thereafter. Latency budget: target ≤150ms per top-12 candidate window. Hard fallback: if reranking exceeds 250ms, return fused RRF order without cross-encoder re-ranking.
**Adopted from:** Codex. `ms-marco-MiniLM-L-6-v2` is the industry-standard reranker for this use case. The explicit latency budget and hard fallback are operationally critical for a local MCP tool.

### 14.8 `pythia_force_index` Path Semantics (Q8)
**Decision:** Exact semantics:
- `path` omitted: full-workspace CDC scan (mtime/hash check). Only changed files are re-indexed.
- `path` = directory: recurse that subtree. Apply mtime/hash check per file.
- `path` = specific file: **force re-embed unconditionally**, bypassing mtime/hash check for that file. This is the override for repairing stale derived indexes after crashes.
**Adopted from:** Codex. A tool named `force_index` must be able to force re-indexing of a specific file, or the name is dishonest.

### 14.9 MADR Slug Rules (Q9)
**Decision:** Filename format: `MADR-%03d-<slug>.md`. Slugification: lowercase ASCII, Unicode normalized, all non-alphanumeric runs collapsed to `-`, trimmed, truncated to 64 characters. Empty slug → `untitled`. Collision is impossible because the numeric MADR prefix (from `seq`) is the uniqueness key.
**Adopted from:** Codex (adds 64-char truncation and `untitled` fallback for edge cases).

### 14.10 Graph Trigger Performance Threshold (Q10)
**Decision:** The `BEFORE INSERT` trigger on `graph_edges` performs two indexed `EXISTS` lookups per edge (against `lcs_chunks(id)` and `pythia_memories(id)` primary key indexes). This design is acceptable for v1. Profiling threshold: if a single Slow Path batch exceeds ~50,000 edge inserts or trigger overhead exceeds 250ms, the trigger design must be revisited.
**Adopted from:** Codex. TypeScript pre-validation filters most bad edges before reaching SQLite, so the trigger is a correctness backstop, not the hot path.

### 14.11 MCP Attach Warm Start Behavior (Q11)
**Decision:** When the MCP server initializes against an already-initialized repo, it automatically launches Warm Start indexing in the background **if and only if** `indexing.scan_on_start=true` in config (default: `true`). Tools register immediately. `lcs_investigate` serves existing indexed content. The indexing-progress header communicates background scan status.
**Adopted from:** Codex. Making warm start configurable avoids forced scans in environments where it is unwanted.

### 14.12 `oracle_commit_decision` Transaction Boundary (Q12)
**Decision:** The atomic transaction boundary is SQLite only: `pythia_memories` INSERT and `graph_edges` IMPLEMENTS edges commit together in one transaction. The Obsidian vault write is an out-of-transaction side effect. If it fails: log the failure, surface `OBSIDIAN_DISABLED` in the response metadata, and persist a retry job. The MADR is NOT rolled back.
**Adopted from:** Both daemons agreed. Codex adds the persisted retry job (vs. Gemini's logged failure only).

### 14.13 `lcs_investigate` Output Format (Q13)
**Decision:** Output is plain text. Each chunk is rendered as:
```
--- CHUNK {rank} score={score}
PATH: {file_path}
CNI: {id}
TYPE: {chunk_type}
LINES: {start_line}-{end_line}
```{language}
{content}
```
```
The indexing-progress header (§13.13) is prepended if `index_state != "ready"`.
**Adopted from:** Codex. Block layout is machine-parseable AND LLM-readable. XML-like tags (Gemini) are less standard.

### 14.14 FTS Keyword `tokenchars` (Q14)
**Decision:** The exact tokenizer string for `fts_lcs_chunks_kw` is:
```sql
tokenize="unicode61 tokenchars '._:/#<>?!-'"
```
This preserves: dotted accessors (`AuthManager.login`), CNI fragments (`::` needs `:`), file paths (`/`), generics (`<>`), optional chaining (`?`), bang operators (`!`), and hyphenated names (`kebab-case`).
**Adopted from:** Codex. More complete symbol set than Gemini's `'._<>'`.

### 14.15 Concurrent `ask_oracle` Serialization (Q15)
**Decision:** Concurrent `ask_oracle` calls against the same `session_id` are **queued** behind a per-session FIFO mutex. Only one provider `ask()` call executes at a time per session. Callers block until their turn; they are not rejected.
**Adopted from:** Codex. Queuing preserves transcript ordering without failing the client. Gemini's `SESSION_BUSY` rejection creates avoidable client-side retry loops.

### 14.16 `impacts_files` Type Constraint (Q16)
**Decision:** In v1, `impacts_files` accepts only repository-relative file paths. The MCP server resolves each path to a module CNI (`file::module::default`). Function/class CNIs are rejected as invalid input. If symbol-level targeting is needed in v2, it should use a separate `impacts_nodes` field.
**Adopted from:** Codex. The field is named `impacts_files`. §11.4 explicitly aligns it to file-to-module resolution.

### 14.17 `generation_id` Semantics (Q17)
**Decision:** `generation_id` is the reconstitution epoch of the repository's oracle lineage. It starts at `1` for the first spawned oracle and increments only when a new oracle generation is created from persisted MADRs (after decommission, reconstitution, or provider-state loss). It is NOT a schema version or a per-MADR sequence. It allows auditors to reason about how much inherited context each decision was made with.
**Both daemons agreed.**

### 14.18 `.gitignore` Handling (Q18)
**Decision:** The scanner honors full Git ignore semantics, including nested `.gitignore` files in subdirectories and `.git/info/exclude`, using the `ignore` npm package. A repo-root `.pythiaignore` is additionally supported for index-only exclusions. Binary files not covered by any ignore file are detected via content sniffing (null byte detection) and silently skipped — they do not produce a sync error.
**Adopted from:** Codex. "Respects `.gitignore`" means matching Git's actual behavior. Binary sniffing is more robust than an extension filter.

### 14.19 `additional_context_query` Execution Timing (Q19)
**Decision:** `additional_context_query` is resolved **synchronously** by the MCP server **before** the `ReasoningProvider.ask()` call, using the internal retrieval pipeline (not the public `lcs_investigate` MCP tool surface). If the query returns no results, the provider call proceeds normally with an empty `additionalContext` array — no error is raised.
**Both daemons agreed.**

### 14.20 Error Code Taxonomy (Q20)
**Decision:** A canonical error registry lives at `src/errors.ts`. True failures travel over MCP as JSON-RPC error objects (`{ code: number, message: string, data: { error_code: string, detail?: string } }`). Numeric code ranges:
- `-32010` to `-32019`: auth/config errors (`AUTH_INVALID`, `CONFIG_INVALID`)
- `-32020` to `-32039`: session/lifecycle errors (`SESSION_ALREADY_ACTIVE`, `SESSION_BUSY`, `SESSION_NOT_FOUND`)
- `-32040` to `-32059`: provider/context errors (`PROVIDER_UNAVAILABLE`, `CONTEXT_BUDGET_EXCEEDED`)
- `-32060` to `-32089`: indexing/storage/integrity errors (`INVALID_GRAPH_ENDPOINT`, `INDEX_BATCH_FAILED`, `FULL_REINDEX_REQUIRED`)
Non-fatal metadata (e.g., `OBSIDIAN_DISABLED`) is returned in successful response bodies, not as errors.
**Adopted from:** Codex. Structured error objects with typed `error_code` in `error.data` are correctly machine-parseable. Gemini's string-in-message approach cannot be branched on by code.


## 14-S. Decision Resolutions — Cycle 3 (Additional Questions)
These decisions resolve a second set of 20 Cycle 3 questions covering areas not addressed by §14. They were answered by both Gemini oracle (`pythia-cycle-3-answers-gemini.md`) and Codex (`pythia-cycle-3-answers-codex.md`). Codex won all 14 divergences.

### 16.1 `initial_context_query` mechanics
**Decision:** The MCP server calls the **internal retrieval pipeline** (not the public `lcs_investigate` MCP tool surface) when processing `spawn_oracle.initial_context_query`. Results are injected into `contextChunks` of `ReasoningProvider.spawn()`, NOT into `systemInstruction` (which is reserved for MADR preamble). Budget: up to **60,000 characters** for bootstrap retrieval context after MADR packing.
**Adopted from:** Codex. systemInstruction is reserved for MADR history. Mixing retrieval results into it loses the architectural preamble.

### 16.2 MADR reconstitution bootstrap budget
**Decision:** Pack accepted MADRs into `systemInstruction` **newest-first** until a **120,000-character MADR cap** is reached. The remaining ~60,000 characters are reserved for `initial_context_query` context and fixed preamble text. Total must fit within the 180,000-character `spawn()` cap (§12.3). Oldest MADRs are dropped if the set overflows — most-recent decisions are most relevant to current work.
**Adopted from:** Codex. Explicit budget split with newest-first ordering is more deterministic than Gemini's vague "drop oldest" policy.

### 16.3 SQLite connection pragma set
**Decision:** Every SQLite connection opened by any thread (Thread 0 and Thread 1) must execute this pragma set immediately after open, before any read or write:
```sql
PRAGMA journal_mode=WAL;
PRAGMA synchronous=NORMAL;
PRAGMA busy_timeout=5000;
PRAGMA foreign_keys=ON;
```
`busy_timeout` prevents SQLITE_BUSY errors under concurrent Thread 0 reads + Thread 1 writes. `foreign_keys=ON` enforces FK constraints the spec relies on.
**Adopted from:** Codex. Gemini's set omitted `busy_timeout` and `foreign_keys`.

### 16.4 `pythia init` vs `pythia start` distinction
**Decision:** `pythia init`: creates `.pythia/` directory, creates/opens `lcs.db`, runs all pending migrations, validates config via Zod, kicks off the first **cold-start full index**. `pythia start`: launches the MCP server process, runs pending migrations if the schema is out of date, then performs warm-start background scanning if `indexing.scan_on_start=true`. Both commands are migration-safe.
**Adopted from:** Codex. Gemini's "init just writes config" contradicts Section 9 which explicitly says init creates `lcs.db` and runs the Worker Thread.

### 16.5 FTS query routing heuristic
**Decision:** No new public parameter is added to `lcs_investigate` in v1. The MCP server routes to `fts_lcs_chunks_sub` (trigram) when: (a) the query is enclosed in double quotes, OR (b) keyword FTS returns zero hits AND the query contains punctuation-heavy tokens or CNI-slice syntax (contains `::`, `/`, `.`). Otherwise routes to `fts_lcs_chunks_kw`. Fallback is transparent to the caller.
**Adopted from:** Codex. Quote-only routing (Gemini) is too narrow — CNI fragments and file path substrings legitimately need trigram without explicit quoting.

### 16.6 Migration runner lifecycle
**Decision:** Forward-only SQL migration files live in `src/migrations/` (bundled in the npm package). Migrations run explicitly at three lifecycle points: `pythia init` (first boot), `pythia start` (MCP server launch), and MCP server connection open (upgrade safety net). Migrations always run **before any tool is registered**.
**Adopted from:** Codex. Connection-open-only (Gemini) is too implicit and can mask version skew.

### 16.7 `generation_id` is a lineage epoch, not a session counter
**Decision:** `generation_id` increments only when a new oracle generation is instantiated from persisted MADR memory after the prior live state is gone or intentionally terminated. An idempotent `spawn_oracle` that returns an existing active session does **NOT** increment `generation_id`. Every MADR written during a generation carries that generation's epoch number.
**Adopted from:** Codex. Gemini's "increment every spawn" would make generation_id meaningless as a lineage marker.

### 16.8 `lcs_investigate` structural query input format
**Decision:** For `intent: "structural"`, the `query` field must be an **exact CNI** or a **repository-relative file path**. File paths are canonicalized server-side to the module CNI (`<path>::module::default`). Natural language is not a valid starting node for graph traversal. Intended workflow: semantic search first → get CNI → structural traversal second.
**Adopted from:** Codex. Graph traversal requires a deterministic starting node. Codex adds the file-path canonicalization path.

### 16.9 `oracle_commit_decision` transaction order
**Decision:** Within one `BEGIN IMMEDIATE` transaction: (1) MADR row inserted first (securing the AUTOINCREMENT `seq` and derived `id`), (2) then `IMPLEMENTS` graph edges inserted. If any IMPLEMENTS target CNI is invalid, the `BEFORE INSERT` trigger on `graph_edges` aborts the entire transaction including the MADR insertion.
**Adopted from:** Both daemons agreed. Transaction atomicity is absolute — no partial MADR with missing edges.

### 16.10 Error code taxonomy and transport
**Decision:** **Fatal failures** (`AUTH_INVALID`, `SESSION_ALREADY_ACTIVE`, `PROVIDER_UNAVAILABLE`, `INVALID_GRAPH_ENDPOINT`, `FULL_REINDEX_REQUIRED`): returned as MCP JSON-RPC error objects with `error.data.error_code`. **Non-fatal conditions** (`OBSIDIAN_DISABLED`, `INDEX_ALREADY_RUNNING`, `CROSS_ENCODER_UNAVAILABLE`): included in the success response metadata body, not as errors. Canonical error code registry lives in `src/errors.ts`.
**Adopted from:** Codex. Proper typed error objects are machine-branchable. Gemini's `[ERROR_CODE] ...` string-in-success-body cannot be reliably parsed by code.


## 15. Decision Resolutions — Cycle 4
These are binding decisions resulting from the fourth architecture interrogation cycle. Where daemons diverged, the technically superior answer is adopted.

### 15.1 Retry Job Persistence for Obsidian Writes (Q1)
**Decision:** Failed Obsidian writes are stored in a simple JSON file at `<repo>/.pythia/obsidian-retry-queue.json`; a background loop attempts a maximum of 5 retries with exponential backoff (1m, 5m, 15m, 30m, 1h), after which the job is dropped.
**Adopted from:** Gemini. Bloating SQLite for an optional UI projection side-effect is over-engineering.

### 15.2 Error Registry Collision (Q2)
**Decision:** Non-fatal metadata like `OBSIDIAN_DISABLED` is prepended as a bracketed text line at the very top of the plain-text tool output (e.g., `[METADATA: OBSIDIAN_DISABLED]\n\nResults...`).
**Adopted from:** Gemini. Preserves the D-81 plain-text mandate.

### 15.3 `src/errors.ts` Code Assignment (Q3)
**Decision:** The numeric `-320xx` sub-codes are strictly internal/advisory; the public MCP contract relies entirely on the string constants (e.g., `AUTH_INVALID`) embedded in the error payload.
**Adopted from:** Gemini.

### 15.4 Cross-Encoder Lazy Download Failure (Q4)
**Decision:** If the cross-encoder fails to download, `lcs_investigate` returns a prepended warning (`[METADATA: RERANKER_UNAVAILABLE]`) and falls back to returning the un-reranked RRF output. The download is retried per-session on the next boot, not per-call.
**Adopted from:** Gemini.

### 15.5 `generation_id` Increment Trigger (Q5)
**Decision:** `generation_id` is incremented by a new `oracle_reconstitute` MCP tool (added back to the v1 manifest) which explicitly signals the end of the current epoch and forces the LLM to commit any final MADRs before resetting the session state.
**Adopted from:** Gemini. Explicit generation boundaries are required.

### 15-R.1 `generation_id` Increment — Final Resolution (supersedes §15.5)

### 17.15 Canonical `graph_edges` Trigger SQL (Q15)
**Decision:** The authoritative `BEFORE INSERT` trigger SQL:
```sql
CREATE TRIGGER trg_graph_edges_validate_before_insert
BEFORE INSERT ON graph_edges
FOR EACH ROW
BEGIN
    SELECT CASE
        WHEN
            NOT EXISTS (SELECT 1 FROM lcs_chunks WHERE id = NEW.source_id)
            AND
            NOT EXISTS (SELECT 1 FROM pythia_memories WHERE id = NEW.source_id)
        THEN RAISE(ABORT, 'INVALID_GRAPH_ENDPOINT')
    END;

    SELECT CASE
        WHEN
            NOT EXISTS (SELECT 1 FROM lcs_chunks WHERE id = NEW.target_id)
            AND
            NOT EXISTS (SELECT 1 FROM pythia_memories WHERE id = NEW.target_id)
        THEN RAISE(ABORT, 'INVALID_GRAPH_ENDPOINT')
    END;
END;
```
**Adopted from:** Codex. Two separate `SELECT CASE` blocks is cleaner SQL. `trg_` prefix is conventional. Four indexed existence checks per insert.

**Adopted from:** Codex. A configured vault path implies user intent; transient loss belongs in the retry path.

### 17.18 `pythia init` Does Not Create an Oracle Session (Q18)
**Decision:** `pythia init` does not create an oracle session. Session creation is always an explicit `spawn_oracle` call from Claude Code. §16.4 scopes `pythia init` to repo/DB/index bootstrapping only.
**Both daemons agreed.**

### 17.19 Sequential Dual-FTS Routing Is Acceptable in v1 (Q19)
**Decision:** `lcs_investigate` executes the keyword FTS query (`fts_lcs_chunks_kw`) first; only if it returns zero results does it execute the trigram fallback (`fts_lcs_chunks_sub`). The ~2ms sequential overhead is acceptable in v1. No RegExp router or pre-routing heuristic is added.
**Both daemons agreed.** §17.5 (this spec, keyword-first routing) is the canonical ordering.

### 17.20 Distribution: npm Global Package `@pythia/lcs` (Q20)
**Decision:** Pythia v1 is distributed as a global npm package. Normative install path: `npm install -g @pythia/lcs` → `pythia init` → `pythia start` → `pythia mcp install claude-code`. Homebrew and standalone binary packaging are non-normative future options.
**Merged from both daemons.** Gemini provided the package name `@pythia/lcs`; Codex provided the normative CLI command sequence.

## 18. Decision Resolutions — Cycle 6
These are the binding decisions from the sixth architecture interrogation cycle, resolving schema inconsistencies, stale text, and gaps that survived prior cycles. Codex won 6 divergences, Gemini won 1, 13 agreed.

### 18.1 §6 Dual FTS Schema Patch (Q1)
**Decision:** The single `fts_lcs_chunks` table (trigram-only) is deleted from §6 and replaced with the canonical dual-table definitions: `fts_lcs_chunks_kw` (unicode61 + code tokenchars) and `fts_lcs_chunks_sub` (trigram). This is now reflected directly in §6.
**Both daemons agreed.**

### 18.2 §11.4 MADR ID Generation Patch (Q2)
**Decision:** §13.8 is canonical. §11.4 is updated to reference `seq INTEGER PRIMARY KEY AUTOINCREMENT` + `last_insert_rowid()`. `COUNT(*) + 1` is explicitly forbidden.
**Both daemons agreed.**

### 18.3 `pythia init` Does Not Start MCP Server (Q3)
**Decision:** `pythia init` bootstraps the DB, runs migrations, and kicks off the cold-start index. It does NOT start the MCP server process. The "MCP Server is available immediately" language in §9 is removed. Tools become available only after `pythia start`.
**Both daemons agreed.**

### 18.4 Nullable Decommission Columns Are Correct (Q4)
**Decision:** `decommission_hash` and `decommission_salt` remain nullable. A successful live session always has them populated (set at spawn). They are cleared to NULL by `oracle_decommission` to allow name reuse. No live `active` or `idle` session should ever have NULL in these columns.
**Both daemons agreed.**

### 18.5 `ON DELETE CASCADE` Is a Safety Net — Keep It (Q5)
**Decision:** `ON DELETE CASCADE` on `pythia_transcripts` is retained. No normative code path hard-deletes a session row, so the cascade never fires in normal operation. It remains a safety net for manual admin cleanup and test teardown.
**Adopted from:** Codex. Removing a harmless safety net has no upside.

### 18.6 `dead` Session Status Definition (Q6)
**Decision:** A session becomes `dead` when its live provider state is lost unexpectedly and cannot be safely resumed — e.g., MCP server crash, process kill, or unrecoverable provider failure (`AUTH_INVALID`). `dead` = accidental termination (no teardown, no MADR flush). `decommissioned` = deliberate user-initiated secure wipe with transcript deletion.
**Both daemons agreed.**

### 18.7 Cross-Algorithm Hash Mismatch Is Acceptable (Q7)
**Decision:** When WASM fails and the fallback SHA-256 hash differs from the stored `blake3:...` hash, the CDC system correctly treats this as a changed file and re-indexes. This false-positive re-index is an acceptable trade-off. No cross-algorithm comparison logic is added.
**Both daemons agreed.**

### 18.8 Vector Index Degradation Warning (Q8)
**Decision:** When serving FTS-only results due to a stale or rebuilding vector index, `lcs_investigate` prepends `[METADATA: VECTOR_INDEX_STALE]` to the response. Not silent.
**Both daemons agreed.**

### 18.9 `graph_edges` Validation: Trigger + App Precheck (Q9)
**Decision:** Both layers are used. The SQLite `BEFORE INSERT` trigger on `graph_edges` is the authoritative enforcement: it checks that both `source_id` and `target_id` exist in either `lcs_chunks` or `pythia_memories`, raising `ABORT 'INVALID_GRAPH_ENDPOINT'` if not. Application-level TypeScript pre-validation is an optimization precheck that filters most bad edges before they reach SQLite.
**Adopted from:** Codex. The trigger is the authority; the app check is performance optimization only.

### 18.10 `pythia_force_index` Coalescing with `priority=manual` (Q10)
**Decision:** If a file is already being processed by a background sync, a `force_index` call for that exact file coalesces into the active job with `priority=manual` — meaning that file must be unconditionally re-embedded (bypassing hash check) within the current or immediately following batch. Returns `[STATUS: INDEX_MERGED]`.
**Adopted from:** Gemini. The `priority=manual` flag is the right mechanism to preserve `force_index` unconditional semantics inside a coalesced job.

### 18.11 Worker `DIE` Trigger and Shutdown Behavior (Q11)
**Decision:** `DIE` is sent only on MCP server graceful shutdown (SIGTERM handler). The inactivity reaper does NOT send `DIE` — it only affects oracle sessions, not the shared Worker Thread. On receiving `DIE`, the Worker finishes its current file and commits the active SQLite transaction, then exits cleanly.
**Both daemons agreed.**

### 18.12 `spawn_oracle` JSON Envelope (Q12)
**Decision:** `spawn_oracle` returns a JSON string: `{"session_id":"<uuid>","status":"active","created":true|false,"generation_id":N,"decommission_secret":"<32-char-hex>"}`. `decommission_secret` is present only when `created: true` (new generation spawned). When attaching to an existing session, the secret is omitted.
**Adopted from:** Codex. `created` bool and `generation_id` are essential for the caller to distinguish attach vs. new generation. Secret only on creation matches §13.9's "returned exactly once" requirement.

### 18.13 Decommission Verification Phrase Encoding (Q13)
**Decision:** The 128-bit secret is encoded as lowercase hex (32 characters). The full verification phrase is `DECOMMISSION <session_id> <32-char-hex>`. With a 36-character UUID, the phrase is 82 characters total.
**Both daemons agreed.**

### 18.14 `nomic-embed-text-v1.5` Supports 256d Matryoshka Truncation (Q14)
**Decision:** `nomic-embed-text-v1.5` supports Matryoshka Representation Learning and can be safely truncated to 256d. No model change is needed. The ONNX pipeline slices the first 256 floats of the native output.
**Both daemons agreed.**

### 18.15 CNI for Additional TypeScript Constructs (Q15)
**Decision:** TypeScript `interface`, `type` alias, `enum`, and `namespace` declarations are indexed as chunks. CNI format: `<path>::interface::<name>`, `<path>::type::<name>`, `<path>::enum::<name>`, `<path>::namespace::<name>`. These are added to the canonical `chunk_type` enum alongside `function`, `class`, `method`, `module`, `doc`.
**Both daemons agreed.**

### 18.16 Per-Repo MCP Server Isolation (Q16)
**Decision:** Each Claude Code window/repo gets its own MCP server process. The binary is shared; the live server instance is not. Isolation is enforced by process boundaries, process-local workspace root, and each repo's own `.pythia/lcs.db`. No shared daemon across repositories.
**Both daemons agreed.**

### 18.17 mtime Cache Is SQLite-Backed in a Dedicated Table (Q17)
**Decision:** The mtime cache is stored in SQLite in a dedicated `file_scan_cache` table (now in §6). Schema: `file_path TEXT PRIMARY KEY, mtime_ns INTEGER NOT NULL, size_bytes INTEGER NOT NULL, content_hash TEXT NOT NULL, last_scanned_at TEXT NOT NULL`. Adding `mtime` to `lcs_chunks` is wrong — `lcs_chunks` has one row per chunk, not per file.
**Adopted from:** Codex. Dedicated table is cleaner and avoids per-chunk mtime duplication.

### 18.18 MCP Server Owns Context Budget Per Turn (Q18)
**Decision:** The MCP server tracks and enforces remaining context budget per turn. It sums the serialized size of preamble + transcript + proposed `additionalContext` chunks, trims by rank if necessary, and returns `CONTEXT_BUDGET_EXCEEDED` if the total still exceeds the cap after trimming. The `ReasoningProvider` receives already-trimmed input.
**Adopted from:** Codex. §13.3 explicitly assigns trimming responsibility to the MCP server.

### 18.19 Migration "Connection Open" = Per Client Attach (Q19)
**Decision:** "MCP server connection open" means per MCP client attach — not just once at process start. If a client disconnects and reconnects to a still-running server, the migration check runs again and no-ops if schema is current. This is the upgrade safety net that catches the edge case of a hot binary upgrade.
**Adopted from:** Codex. §16.6 lists `pythia start` and "connection open" as separate lifecycle hooks, confirming they are distinct events.

### 18.20 `lcs_investigate` in Sprint 2 Is a Minimal Stub (Q20)
**Decision:** In Sprint 2, `lcs_investigate` implements basic vector search only — no FTS, no RRF, no cross-encoder. The Sprint 2 proof is that Tree-sitter AST chunking produces correct, AST-bounded chunks that are retrievable. The full hybrid pipeline (§13.2, §13.20, §14.7) is implemented in a later sprint after the core AST/vector flow is validated.
**Both daemons agreed.**
