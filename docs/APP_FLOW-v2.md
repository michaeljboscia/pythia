## System Architecture Overview

```
+-------------------------------------------------------------+
|                     UI PLANE (OBSIDIAN)                     |
|  Reads markdown MADRs generated by Pythia. Passive/read-only|
+-------------------------------------------------------------+
                              ^
                              | (Writes MADR files)
                              v
+-------------------------------------------------------------+
|                COGNITION PLANE (PYTHIA DAEMON)              |
|  Maintains conversational state. Writes Episodic Checkpoints|
|  Provider: Gemini CLI ($0 fallback) OR Gemini SDK (Premium) |
+-------------------------------------------------------------+
                              ^
                              | (MCP Protocol JSON-RPC)
                              v
+-------------------------------------------------------------+
|                  INTERFACE PLANE (MCP SERVER)               |
|  Exposes lcs_investigate, oracle_commit_decision, etc.      |
|  Acts as Adaptive Router & Re-ranker (ONNX Cross-Encoder)   |
+-------------------------------------------------------------+
                              ^
                              | (SQL / Vector Search)
                              v
+-------------------------------------------------------------+
|                  DATA PLANE (PYTHIA DB)                     |
|  Storage: SQLite-vec + FTS5 + Recursive CTEs (Single File)  |
|  Indexer: Tree-sitter (Fast Path) + TS LSP (Slow Path)      |
+-------------------------------------------------------------+
```

## Lifecycle 1 — First-Time Setup

### Step 1: Install
```bash
npm install -g @pythia/lcs
```

### Step 2: Configure
Create `~/.pythia/config.json` (or Pythia scaffolds a default).
Config is Zod-validated at every startup. Missing required fields = hard startup error.

### Step 3: Initialize (`pythia init`)
**What happens:**
1. Creates `.pythia/` directory in workspace root
2. Creates `.pythia/lcs.db`
3. Runs ALL pending forward-only migrations from `src/migrations/`
4. Validates `~/.pythia/config.json` via Zod
5. Kicks off first cold-start **full index** via Worker Thread (foreground)
6. **Does NOT start the MCP server**

**What does NOT happen:**
- MCP server is NOT launched
- Oracle session is NOT created
- Tools are NOT available yet

**Idempotency:** If `.pythia/lcs.db` already exists, runs migrations only and exits silently. To wipe: `rm -rf .pythia` (manual only).

### Step 4: Register MCP Tool
```bash
pythia mcp install claude-code
```
- Idempotent
- Previews configuration changes to stdout before applying
- Modifies `~/.claude.json` to add Pythia as an MCP server

### Step 5: Start (`pythia start`)
**What happens:**
1. MCP server process launches
2. Runs pending migrations (no-op if current)
3. If `indexing.scan_on_start=true` (default): warm-start background scan begins
   - mtime check against `file_scan_cache`
   - BLAKE3 hash check for changed files
   - Only changed files re-indexed
4. MCP tools register immediately (before indexing completes)
5. `lcs_investigate` serves existing indexed content immediately

## Lifecycle 2 — Per-Session (Claude Code Opens a Repo)

### On MCP Client Attach
1. Migration check runs (no-op if current — upgrade safety net)
2. If `indexing.scan_on_start=true`: warm-start background scan begins
3. All 6 MCP tools become available immediately
4. `lcs_investigate` serves existing content; indexing-progress header indicates status

### Indexing Progress Header (while indexing)
Prepended to every `lcs_investigate` response while `index_state != "ready"`:
```
[METADATA: index_state=indexing indexed_files=142 total_files=890 percent_complete=16 last_sync_at=2026-03-11T01:30:00Z]
```
Collapses to `[METADATA: index_state=ready]` once complete.

## Lifecycle 3 — Code Investigation Flow

### User asks Claude about code
```
Claude → lcs_investigate({ query: "auth middleware", intent: "semantic" })
         └→ MCP server
              ├→ Vector search (top-30 from vec_lcs_chunks)
              ├→ FTS search (top-30 from fts_lcs_chunks_kw, fallback to fts_lcs_chunks_sub)
              ├→ RRF fusion (weighted by intent: semantic wv=0.7/wf=0.3)
              ├→ Top-12 → Cross-encoder re-rank (Xenova/ms-marco-MiniLM-L-6-v2)
              └→ Plain-text block output (§14.13 format)
```

### Structural traversal
```
Claude → lcs_investigate({ query: "src/auth.ts::function::login", intent: "structural" })
         └→ MCP server
              ├→ Resolve query to CNI (if file path, canonicalize to module CNI)
              ├→ Bidirectional BFS CTE traversal (depth ≤6, cycle detection, 50-node cap)
              └→ §14.13 format blocks with [DEPTH:N via EDGE_TYPE] prefix per block
```

### Error: Index empty
```
[METADATA: INDEX_EMPTY]
[NO_RESULTS] No indexed chunks available yet.
```

### Error: No match
```
[METADATA: NO_MATCH]
[NO_RESULTS] No matching chunks found.
```

## Lifecycle 4 — Oracle Session Flow

### Spawn Oracle
```
Claude → spawn_oracle({ session_name: "auth-refactor", initial_context_query: "authentication middleware" })
         └→ MCP server
              ├→ Check for existing active/idle session with same name
              │    ├→ Found active/idle: return existing session_id (no secret)
              │    └→ Not found: proceed to create
              ├→ Generate 128-bit random decommission secret
              ├→ Store Argon2id hash + salt in pythia_sessions
              ├→ Load accepted MADRs newest-first up to 120,000 char cap
              ├→ Resolve initial_context_query via internal retrieval (up to 60,000 chars)
              ├→ ReasoningProvider.spawn(sessionName, systemInstruction=MADRs, contextChunks=retrieval)
              │    Total ≤ 180,000 chars
              └→ Return: {"session_id":"<uuid>","status":"active","created":true,"generation_id":1,
                          "decommission_secret":"<32-char-hex>"}
```

**Error: Another session already active**
```json
{ "code": -32020, "message": "SESSION_ALREADY_ACTIVE", "data": { "error_code": "SESSION_ALREADY_ACTIVE" } }
```

### Ask Oracle
```
Claude → ask_oracle({ session_id: "...", message: "...", additional_context_query: "..." })
         └→ MCP server
              ├→ Validate session_id exists, status=active or idle
              │    └→ If idle: auto-respawn from MADRs (not transcript replay)
              ├→ Resolve additional_context_query via internal retrieval (≤12 chunks, 48,000 chars)
              ├→ Check context budget (preamble + transcript + context ≤ session cap)
              │    └→ If over: trim by rank. If still over: return CONTEXT_BUDGET_EXCEEDED
              ├→ Write user turn to pythia_transcripts (write-ahead)
              ├→ Acquire per-session FIFO mutex (queue depth ≤5, 6th → SESSION_BUSY)
              ├→ ReasoningProvider.ask(sessionId, message, additionalContext)
              │    └→ Retry: 1s, 5s, 15s backoff → PROVIDER_UNAVAILABLE
              ├→ Write model turn to pythia_transcripts
              └→ Return provider response string
```

### Idle Session Auto-Resume
```
ask_oracle on idle session
  └→ provider state gone (CLI process killed / SDK memory freed by reaper)
       └→ Re-spawn provider
            ├→ Load accepted MADRs newest-first
            └→ ReasoningProvider.spawn() — fresh provider with MADR context
```

Note: pythia_transcripts NOT replayed. Provider starts fresh from MADR preamble only.

### Commit Decision (MADR)
```
Claude → oracle_commit_decision({ title, problem, drivers, options, decision, impacts_files, supersedes_madr? })
         └→ MCP server
              ├→ BEGIN IMMEDIATE transaction
              │    ├→ INSERT INTO pythia_memories (seq, id=printf('MADR-%03d',seq), ...)
              │    ├→ If supersedes_madr: UPDATE pythia_memories SET status='superseded' WHERE id=?
              │    └→ INSERT INTO graph_edges (source=MADR-id, target=file::module::default, edge_type=IMPLEMENTS)
              ├→ COMMIT
              ├→ Obsidian write (out-of-transaction side effect)
              │    ├→ Vault accessible: write MADR-012-title-slug.md to <repo>/Pythia/
              │    ├→ Vault unconfigured: OBSIDIAN_DISABLED in response metadata
              │    └→ Vault configured but inaccessible: OBSIDIAN_UNAVAILABLE + add to retry queue
              └→ Return: "MADR-012" (+ non-fatal metadata if any)
```

### Decommission Oracle
```
Claude → oracle_decommission({ session_id: "...", verification_phrase: "DECOMMISSION <uuid> <32-hex>" })
         └→ MCP server
              ├→ Verify phrase against stored Argon2id hash (hard fail on mismatch)
              ├→ Hard-delete ALL pythia_transcripts WHERE session_id=?
              ├→ UPDATE pythia_sessions SET status='decommissioned',
              │         decommission_hash=NULL, decommission_salt=NULL
              └→ Return success string
```

Session name becomes available for reuse (partial unique index only covers active/idle rows).
MADRs in pythia_memories are NOT deleted — they persist into next generation.

## Lifecycle 5 — Background Indexing Flow

### File Change Detected (CDC)
```
sync_workspace()
  └→ Walk directory tree (fast-glob + ignore package)
       ├→ Read file_scan_cache (mtime_ns per file)
       ├→ Compare file mtime → changed?
       │    └→ Yes: read file, compute BLAKE3 hash
       │         └→ Different from content_hash? → queue for indexing
       └→ Binary file? (null byte in first 4096 bytes) → skip silently
```

### Fast Path (per changed file, in Worker Thread)
```
BEGIN TRANSACTION
  1. UPDATE lcs_chunks SET is_deleted=1, deleted_at=? WHERE file_path=?  (soft-delete old chunks)
  2. DELETE stale rows from vec_lcs_chunks, fts_lcs_chunks_kw, fts_lcs_chunks_sub, graph_edges
  3. Tree-sitter parse → extract chunks (function, class, method, interface, type, enum, namespace, module)
     - For .md/.mdx: chunk by heading → doc chunks with CNI <path>::doc::<slug>#L<line>
  4. ONNX embed each chunk (nomic-embed-text-v1.5, first 256 floats)
  5. INSERT INTO lcs_chunks, vec_lcs_chunks, fts_lcs_chunks_kw, fts_lcs_chunks_sub
  6. INSERT CONTAINS edges (module→class, class→method hierarchy)
  7. UPDATE file_scan_cache SET mtime_ns=?, content_hash=?  (immediately before COMMIT)
COMMIT
```

### Slow Path (runs after Fast Path batch, in Worker Thread)
```
tsserver.getDefinitionAtPosition() for each symbol in changed files
  └→ Resolve cross-file CALLS and IMPORTS
       └→ INSERT INTO graph_edges (validated by trg_graph_edges_validate_before_insert)
```

### Worker Thread Crash
```
MCP server detects missed HEARTBEAT
  └→ Crash count < 3 in 10min: restart with exponential backoff
       └→ Crash count ≥ 3 in 10min: circuit breaker trips
            ├→ Auto-restart stops
            ├→ lcs_investigate: prepend [METADATA: SLOW_PATH_DEGRADED]
            └→ Recovery: manual pythia_force_index OR process restart
```

## Lifecycle 6 — GC Flow

### GC Trigger Conditions
- Every MCP server boot
- After any sync batch where tombstones > 10,000 rows OR > 20% of total chunk count

### GC Execution
```
BEGIN TRANSACTION
  SELECT id FROM lcs_chunks WHERE is_deleted=1 AND deleted_at < (NOW - retention_days)
  → For each stale chunk:
       DELETE FROM lcs_chunks WHERE id=?
       DELETE FROM vec_lcs_chunks WHERE id=?
       DELETE FROM fts_lcs_chunks_kw WHERE id=?
       DELETE FROM fts_lcs_chunks_sub WHERE id=?
COMMIT
PRAGMA incremental_vacuum
```

## Lifecycle 7 — Obsidian Write + Retry Flow

### MADR File Format
```markdown
---
madr_id: MADR-012
title: Authentication Middleware Strategy
status: accepted
timestamp: 2026-03-11T02:00:00Z
generation_id: 1
decision_outcome: Use JWT with short TTL + refresh token rotation
---

# MADR-012 — Authentication Middleware Strategy

## Context and Problem
[context_and_problem]

## Decision Drivers
- [driver 1]
- [driver 2]

## Considered Options
- [option 1]
- [option 2]

## Decision Outcome
[decision_outcome]
```

### Retry Queue Lifecycle
```
Vault write fails
  └→ Serialize job to obsidian-retry-queue.json (atomic: write .tmp → fsync → rename)
       └→ Background setInterval retries:
            Attempt 1: wait 1m
            Attempt 2: wait 5m
            Attempt 3: wait 15m
            Attempt 4: wait 30m
            Attempt 5: wait 1h
            After 5 failures: drop job silently

MCP server restarts:
  └→ Load obsidian-retry-queue.json on boot
       ├→ Malformed JSON: rename to .corrupt, start fresh empty queue
       └→ Valid: replay pending jobs
```

## Inactivity Reaper Flow

```
Timer fires every minute (or configurable interval)
  └→ For each session WHERE status='active' OR status='idle':
       └→ IF (NOW - last_ask_at) > session_idle_ttl_minutes:
            ├→ ReasoningProvider.dismiss(sessionId)  (free KV cache / kill CLI process)
            └→ UPDATE pythia_sessions SET status='idle'
```

**Important:** Reaper NEVER sends `DIE` to Worker Thread. `DIE` is for MCP server graceful shutdown only.

## Error Flows

### AUTH_INVALID (SDK provider)
```
SdkReasoningProvider receives 401 Unauthorized
  └→ Throw AUTH_INVALID immediately
       └→ No silent fallback to CLI provider
            └→ Spawn attempt marked failed
```

### CONTEXT_BUDGET_EXCEEDED
```
ask_oracle: preamble + transcript + proposed context > session cap
  └→ MCP server trims additionalContext by rank
       └→ Still over cap → return CONTEXT_BUDGET_EXCEEDED JSON-RPC error
```

### INVALID_PATH (force_index)
```
pythia_force_index with path that:
  - doesn't exist on disk
  - is absolute
  - resolves outside workspace root (../ traversal)
  └→ Return INVALID_PATH JSON-RPC error immediately (no soft-delete, no retry)
```

### force_index on in-flight file
```
pythia_force_index({ path: "src/auth.ts" })
  └→ File already mid-sync in active batch
       └→ Merge into active job with priority=manual
            └→ Return: "[STATUS: INDEX_MERGED] Background indexing is already running; your request was added to the queue."
```
